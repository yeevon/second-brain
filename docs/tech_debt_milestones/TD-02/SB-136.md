# SB-136 — SQLite runtime startup/shutdown watchdogs

**Branch:** `feature/sqlite-runtime-watchdogs`
**Source:** TD-P1-001

---

## Goal

SQLite runtime startup, queue operations, job completion, and shutdown can currently wait forever. Unattended EC2 operation requires bounded waits with visible failures — a hung worker should be logged and the main thread must still exit cleanly.

---

## Acceptance criteria

- [ ] Add configurable timeout env vars:
  - `SQLITE_STARTUP_TIMEOUT_S`
  - `SQLITE_QUEUE_WAIT_TIMEOUT_S`
  - `SQLITE_JOB_COMPLETION_TIMEOUT_S`
  - `SQLITE_SHUTDOWN_DRAIN_TIMEOUT_S`
- [ ] Timeout log events include:
  - Step name
  - Timeout value (configured)
  - Elapsed time
- [ ] Timeout log events do not include raw exception bodies.
- [ ] An accepted capture fails visibly on timeout — no silent drop.
- [ ] Queued work is not silently discarded on timeout.
- [ ] Hung shutdown worker is logged and main thread exits regardless.
- [ ] Existing SQLite runtime tests continue to pass.

**Done when:** every SQLite runtime phase has a finite upper bound and a visible failure path on timeout.
