# SB-140 — Legacy schema adoption validation

**Branch:** `feature/schema-adoption-validation`
**Source:** TD-P1-005

---

## Goal

When the migration system is applied to a pre-existing SQLite database, it should validate that the existing schema matches the expected baseline before marking it as migrated. Currently a schema mismatch could be silently adopted, leading to runtime failures or data corruption.

---

## Acceptance criteria

- [ ] Before adopting migration `001`, validate that the existing schema has:
  - Expected columns and types
  - Expected indexes
  - Expected foreign keys
  - No unexpected triggers or views that could bypass constraints
- [ ] Startup fails clearly with a descriptive error on any mismatch.
- [ ] No silent repair or schema patching.
- [ ] Tests cover:
  - Correct schema passes validation
  - Missing column fails with clear error
  - Wrong column type fails with clear error
  - Missing index fails with clear error
  - Missing or wrong foreign key fails with clear error
  - Unexpected trigger fails with clear error
  - Unexpected view fails with clear error

**Done when:** applying migrations to a pre-existing database that does not match the expected baseline produces a clear startup failure rather than silent adoption.
