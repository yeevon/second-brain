# SB-141 — Immutable raw vault substrate

**Branch:** `feature/raw-vault-substrate`
**Source:** TD-P1.5-001 / GitHub #20

---

## Goal

The vault currently stores sanitized/classified notes, but not the exact original input that produced them.

That is an auditability gap. Technical captures, logs, terminal output, code snippets, admin notes, and debug messages can lose important meaning when transformed by the classifier, n8n, or writer-service.

SB-141 adds an immutable raw vault substrate:

```text
00_raw/YYYY/MM/<capture_id>.md
```

Every filed capture gets a deterministic raw Markdown artifact that preserves the original input and links to the sanitized note through frontmatter hash metadata.

---

## Non-goal: strict raw-before-classification

The ideal long-term sequence is:

```text
capture received
raw artifact persisted
classification/transformation happens
sanitized note written
```

TD-03 does **not** fully implement that sequence because the current writer-service API receives a post-classification request. `FileNoteRequest` already contains `classification`.

TD-03 instead implements:

```text
capture-service / n8n performs classification
writer-service receives classification + raw_text
writer-service writes raw file first
writer-service writes sanitized note second
sanitized note links to raw file
```

The TD-03 guarantee is:

```text
raw file is written before sanitized vault note generation
```

The TD-03 guarantee is **not**:

```text
raw file is written before LLM classification
```

Strict raw-before-classification is deferred to a future pipeline ticket that introduces either:

* a raw-intake writer-service endpoint before n8n/classification, or
* capture-service-side durable raw persistence before classification dispatch.

This distinction matters. TD-03 fixes vault auditability now without a larger pipeline rewrite.

---

## Sensitive-capture policy

**Option A is the chosen policy:** raw vault files may contain sensitive content.

The vault is treated as trusted private storage. Sensitive captures get the same lossless raw artifact as non-sensitive captures.

TD-03 does not create redacted raw stubs, encrypted raw files, or special sensitive-only storage.

This is a security posture decision. The vault is already the authoritative private note store. Extending that trust to raw files is consistent with the current system model. If that assumption changes later, raw-file generation for sensitive captures must be revisited in a separate security ticket.

---

## Data-flow and API ownership

writer-service remains the sole writer to the vault in capture-only / Docker mode.

TD-03 uses the existing writer-service filing path and extends it.

Current shape:

```text
capture-service / n8n
  -> writer-service /internal/notes/file
      -> sanitized note written to vault
```

TD-03 shape:

```text
capture-service / n8n
  -> writer-service /internal/notes/file
      -> raw file written to vault
      -> sanitized note written to vault
      -> audit event written
      -> optional Git commit/push
```

`FileNoteRequest` gains:

```python
raw_text: str
```

`raw_text` is the exact original Discord message text as captured by capture-service. It must not be summarized, truncated, rewrapped, normalized, or redacted before being sent to writer-service.

The writer-service handler must write or verify the raw file before rendering/writing the sanitized note.

The raw write must be flushed to disk before sanitized note generation proceeds. Git commit is not required before sanitized note generation, but when Git sync is enabled, the raw file, sanitized note, and audit log should be committed together.

---

## Request/response contract changes

### FileNoteRequest additions

Add:

```python
raw_text: str
```

Optional attachment metadata may be added if it is not already available in the current request path:

```python
attachments: list[AttachmentMetadata] = []
```

Suggested attachment metadata shape:

```python
class AttachmentMetadata(BaseModel):
    filename: str
    content_type: str | None = None
```

TD-03 does not store binary attachment bytes.

### FileNoteResponse additions

Add:

```python
raw_capture_path: str
raw_sha256: str
```

The response allows capture-service to persist the raw linkage if desired and gives downstream status/audit tooling a direct reference to the raw artifact.

## Raw file path

Raw files are stored under:

```text
00_raw/YYYY/MM/<capture_id>.md
```

Rules:

* `YYYY/MM` comes from `created_at` interpreted as UTC.
* Filename is derived only from `capture_id`.
* Classification output must not affect raw path.
* Title, area, project, tags, and model output must not affect raw path.
* Example:

```text
00_raw/2026/06/SB-20260621-0003.md
```

This path is deterministic, which makes idempotency simple and safe.

---

## Raw file format

Raw files are Markdown files with minimal frontmatter.

```markdown
---
capture_id: "SB-YYYYMMDD-NNNN"
created_at: "<ISO-8601 UTC>"
source_message_id: "<discord snowflake>"
raw_sha256: "<hex digest>"
schema_version: 1
---

<raw body>
```

The raw body is built from:

```text
raw_text
optional attachment metadata section
```

No code-fence wrapping is added. No trimming is performed. No line-ending normalization is performed.

---

## Raw body rules

For text-only captures:

```text
raw_body = raw_text
```

For text + attachments:

```text
raw_body = raw_text + "\n\n## Attachments\n\n" + attachment_metadata_lines
```

For attachment-only captures:

```text
raw_body = "## Attachments\n\n" + attachment_metadata_lines
```

Attachment metadata lines use this format:

```markdown
- filename: "photo.png", content_type: "image/png"
```

If content type is unknown:

```markdown
- filename: "archive.zip", content_type: null
```

Binary attachment bytes are not stored in TD-03.

---

## Hash definition

`raw_sha256` is defined as:

```python
sha256(raw_body.encode("utf-8")).hexdigest()
```

Important rules:

* Hash is computed over raw body only.
* Frontmatter is excluded.
* No line-ending normalization happens before hashing.
* No trimming happens before hashing.
* No Markdown wrapping happens before hashing.
* The body read back from the raw file must recompute to the same hash.

The sanitized note frontmatter must contain the same `raw_sha256`.

---

## Sanitized note frontmatter additions

Every sanitized note written by writer-service gains:

```yaml
raw_capture_path: "00_raw/2026/06/SB-20260621-0003.md"
raw_sha256: "<hex digest>"
derived_from_capture_id: "SB-20260621-0003"
```

Rules:

* `derived_from_capture_id` must match the note `capture_id`.
* `raw_capture_path` must point to the deterministic raw path.
* `raw_sha256` must match the raw file frontmatter and recomputed raw body hash.
* Existing sanitized note idempotency must continue to work.

---

## Write ordering

Within writer-service, the required order is:

```text
1. Validate request.
2. Compute deterministic raw path.
3. Build raw body.
4. Compute raw_sha256.
5. Write or verify raw file.
6. Render sanitized note with raw linkage frontmatter.
7. Write sanitized note.
8. Append audit event.
9. If Git sync is enabled, git add raw + sanitized + audit log, commit, push.
```

Raw file write/verify must happen before sanitized note rendering/writing.

TD-03 does not require raw file persistence before classification because classification has already happened before the current `FileNoteRequest` reaches writer-service.

---

## Atomicity and durability

Raw file writes should use the same vault write lock already used by writer-service.

Raw file creation should be atomic enough to avoid corrupt partial files:

```text
write temp file
flush/fsync temp file
rename into final path
```

At minimum, writer-service must not leave a partially written final raw file on expected recoverable failures.

When Git sync is enabled:

* raw file
* sanitized note
* audit log

should be part of the same Git commit.

If Git commit or push fails after files are written, existing writer-service rollback behavior should restore the vault to the pre-write state where practical.

---

## Idempotency and failure semantics

### New raw file

If `00_raw/YYYY/MM/<capture_id>.md` does not exist:

* create it
* include frontmatter
* include raw body
* compute and store `raw_sha256`
* proceed to sanitized note generation

### Existing raw file with matching hash

If the raw file already exists:

* read existing raw file
* parse `raw_sha256`
* recompute hash from existing body
* compute hash from incoming raw body
* if all hashes match, treat as idempotent
* do not rewrite the raw file
* proceed to sanitized note generation

### Existing raw file with mismatched hash

If the raw file already exists but hash does not match the incoming raw body:

* fail hard
* do not overwrite
* do not write sanitized note
* log safe metadata only:

  * `capture_id`
  * expected hash
  * incoming hash
  * error type
* return a writer-service error

This indicates a data integrity violation.

### Raw write failure

If the raw file cannot be written:

* writer-service returns an error
* sanitized note is not written
* audit event is not appended as a successful file event
* capture remains retryable through existing delivery/reaper behavior

### Sanitized note failure after raw write success

If raw write succeeds but sanitized note write fails:

* retry must reuse the existing raw file
* retry must not rewrite raw file
* retry proceeds only if incoming hash matches existing raw hash

---

## Audit log behavior

The existing audit event should include raw linkage fields:

```json
{
  "capture_id": "SB-20260621-0003",
  "event": "FILED",
  "path": "20_projects/example/2026-06-21--SB-20260621-0003--title.md",
  "raw_capture_path": "00_raw/2026/06/SB-20260621-0003.md",
  "raw_sha256": "<hex digest>",
  "timestamp": "<ISO-8601>"
}
```

For inbox notes, event remains `INBOX`.

For idempotent delivery, audit behavior should remain consistent with the current writer-service idempotency policy. Do not create duplicate raw files.

## Acceptance criteria

* [ ] `FileNoteRequest` includes `raw_text: str`.
* [ ] Every successfully filed capture writes or verifies a raw Markdown file under `00_raw/YYYY/MM/`.
* [ ] Raw file path is `00_raw/YYYY/MM/<capture_id>.md`.
* [ ] Raw path is derived from `created_at` UTC and `capture_id`, not classification output.
* [ ] Raw file is written or verified before sanitized note generation.
* [ ] Raw content preserves the original input without trimming, wrapping, or line-ending normalization.
* [ ] Raw body includes attachment metadata when attachments are present.
* [ ] Attachment-only captures produce a raw file with an `## Attachments` section and no binary bytes.
* [ ] Sensitive captures receive the same lossless raw artifact as non-sensitive captures.
* [ ] Raw file frontmatter includes `raw_sha256`.
* [ ] `raw_sha256` equals `SHA-256(raw_body.encode("utf-8"))`.
* [ ] Sanitized note frontmatter includes `raw_capture_path`, `raw_sha256`, and `derived_from_capture_id`.
* [ ] Sanitized note `raw_sha256` matches raw file `raw_sha256`.
* [ ] Existing raw file with matching hash is treated as idempotent.
* [ ] Existing raw file with mismatched hash fails hard and is not overwritten.
* [ ] Raw write failure prevents sanitized note generation.
* [ ] Sanitized note failure after raw write success can be retried without rewriting raw.
* [ ] When Git sync is enabled, raw file, sanitized note, and audit log are committed together.
* [ ] Strict raw-before-classification is documented as deferred, not falsely claimed.

## Tests

Required tests:

* Raw file exists after successful writer-service filing.
* Raw file path is `00_raw/YYYY/MM/<capture_id>.md`.
* Raw file is written before sanitized note generation.
* Raw file body exactly matches original `raw_text` for text-only capture.
* Raw file preserves whitespace, code fences, terminal output, and line breaks.
* `raw_sha256` in raw frontmatter matches recomputed SHA-256 of raw body.
* Sanitized note frontmatter includes `raw_capture_path`, `raw_sha256`, and `derived_from_capture_id`.
* Sanitized note `raw_sha256` matches raw file `raw_sha256`.
* Re-delivery of same capture ID with same raw text does not create a second raw file.
* Re-delivery of same capture ID with different raw text fails hard and does not overwrite.
* Raw write failure prevents sanitized note generation.
* Raw write success plus sanitized note failure retries by reusing existing raw file.
* Sensitive capture writes exact raw text under Option A policy.
* Text + attachments capture appends `## Attachments` metadata section.
* Attachment-only capture creates raw metadata artifact with no binary body.
* Git-sync enabled filing adds raw file, sanitized note, and audit log to the same commit.
* `FileNoteResponse` returns `raw_capture_path` and `raw_sha256`.

## Implementation notes

Suggested writer-service structure:

```text
VaultWriter.write_note()
  acquire vault lock
  check git sync preconditions if enabled
  write_or_verify_raw_capture()
  render sanitized markdown with raw linkage
  write sanitized note
  append audit event with raw linkage
  git add raw + sanitized + audit log
  git commit/push if enabled
```

Suggested helper functions:

```python
build_raw_body(raw_text, attachments) -> str
raw_capture_path(capture_id, created_at) -> Path
compute_raw_sha256(raw_body) -> str
render_raw_markdown(...)
write_or_verify_raw_capture(...)
parse_raw_file(...)
```

Do not scan the whole vault to find raw files. Raw path is deterministic.

Do not base raw filename on title, folder, project, classification, or sanitized note path.

Do not overwrite raw files.

---

## Done when

Every successfully filed capture has:

* a deterministic immutable raw file in `00_raw/YYYY/MM/`
* a sanitized note linked to that raw file
* matching `raw_sha256` values in both files
* idempotent retry behavior
* hard failure on hash mismatch
* tests proving raw/sanitized linkage and failure behavior

TD-03 is done when the raw input is no longer lost after sanitized note generation, without falsely claiming the broader raw-before-classification pipeline guarantee.
