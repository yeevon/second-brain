# SB-141 — Immutable raw vault substrate

**Branch:** `feature/raw-vault-substrate`
**Source:** TD-P1.5-001 / GitHub #20

---

## Goal

Sanitized notes are human-readable, but technical, admin, and debug captures need a lossless vault-visible raw artifact that is independent of LLM classification or writer-service transformation. This closes the auditability gap: every capture must have a raw counterpart in the vault.

---

## Sensitive-capture policy

**Option A is the chosen policy:** raw vault files may contain sensitive content. The vault is treated as trusted private storage. Sensitive captures get the same lossless raw artifact as non-sensitive ones. No redacted stubs, no encryption.

This is a security posture decision: the vault is already the authoritative private store. Extending that trust to raw files is consistent with the existing model. If this assumption ever changes, raw file generation for sensitive captures must be revisited in a separate ticket.

---

## Data-flow and API ownership

The raw file is written by **writer-service**, not capture-service. This preserves the invariant that writer-service is the sole vault writer in capture-only / Docker mode.

Sequence:

1. capture-service receives Discord message → inserts row (`status=RECEIVED`)
2. capture-service dispatches to n8n / writer-service (capture-only) with `raw_text` included in the `FileNoteRequest`
3. writer-service writes the raw file first (`00_raw/YYYY/MM/<capture_id>.md`)
4. Only after a successful raw write does writer-service proceed with LLM classification / sanitized note generation
5. writer-service calls back to capture-service with `raw_capture_path` + `raw_sha256` + `derived_note_path` + `git_commit_hash`

`FileNoteRequest` gains a `raw_text: str` field. This is the exact original message text — no preprocessing, no truncation.

The ordering guarantee in step 3–4 is enforced within a single writer-service request handler. Classification does not run until the raw file is durably written (fsync is sufficient; Git commit of the raw file is not required before classification proceeds, but the raw file must be flushed to disk).

---

## Raw file path

```
00_raw/YYYY/MM/<capture_id>.md
```

- `YYYY/MM` is derived from the capture's `created_at` UTC timestamp (not the write time).
- Filename is derived solely from `capture_id` — classification output is not available at write time.
- Example: `00_raw/2026/06/SB-20260621-0003.md`

This path is deterministic from `capture_id` alone, which makes idempotency checks trivial.

---

## Raw file format

```markdown
---
capture_id: "SB-YYYYMMDD-NNNN"
created_at: "<ISO-8601 UTC>"
discord_message_id: "<snowflake>"
raw_sha256: "<hex digest>"
schema_version: 1
---

<exact original message text, no modification>
```

Rules:
- Frontmatter is minimal — only fields known before classification.
- Body is the exact original message text. No normalization. No trimming. No code-fence wrapping.
- Line endings are preserved exactly as received from Discord (typically `\n`).
- `raw_sha256` is `SHA-256(body.encode("utf-8"))` — hash of the body bytes after frontmatter, before any frontmatter is prepended. Whitespace and line endings are not normalized before hashing.

---

## Sanitized note frontmatter additions

Every sanitized note gains three new frontmatter fields:

```yaml
raw_capture_path: "00_raw/2026/06/SB-20260621-0003.md"
raw_sha256: "<hex digest — must match raw file>"
derived_from_capture_id: "SB-20260621-0003"
```

---

## Attachment behavior

TD-03 explicitly excludes S3 attachment storage. Raw file behavior per capture type:

| Capture type | Raw file body |
|---|---|
| Text-only | Exact message text |
| Text + attachments | Exact message text, followed by an `## Attachments` section listing filename and content-type per attachment. No binary bytes stored. |
| Attachment-only (no text body) | Empty body with `## Attachments` section listing filename and content-type. |

The `## Attachments` section format:

```markdown
## Attachments

- filename: "photo.png", content_type: "image/png"
```

Binary attachment bytes are not stored in TD-03.

## Idempotency and failure semantics

**Raw write failure:** if the raw file cannot be written (disk error, path guard violation, etc.), writer-service returns an error and does not proceed to classification or sanitized note generation. The capture-service row stays at `PENDING_FORWARD` and will be retried by the stale-lease reaper.

**Retry after partial success:** if a raw file exists for a given `capture_id` (path `00_raw/YYYY/MM/<capture_id>.md` already exists):
- Compute `SHA-256` of the incoming `raw_text`.
- If the hash matches the `raw_sha256` in the existing file's frontmatter: skip the write, proceed to classification. This is the idempotent retry path.
- If the hash does not match: fail hard with a logged error. Do not overwrite. This case indicates a data integrity violation and must not be silently resolved.

**Sanitized note failure after raw write success:** if the raw file is written but the sanitized note write fails, retry reuses the existing raw file (idempotent path above). The raw file is never rewritten on retry.

## Acceptance criteria

- [ ] Every capture writes a raw Markdown file under `00_raw/YYYY/MM/`.
- [ ] Raw file path is `00_raw/YYYY/MM/<capture_id>.md` — derived from `capture_id`, not from title or classification.
- [ ] Raw file is written before LLM classification or writer-service transformation runs.
- [ ] Raw content preserves the exact original input: terminal output, logs, code fences, whitespace, line breaks.
- [ ] Line endings are not normalized before hashing or writing.
- [ ] `raw_sha256` in the raw file frontmatter equals `SHA-256(body.encode("utf-8"))` — computed over the body bytes only, not including frontmatter.
- [ ] Sanitized note frontmatter includes `raw_capture_path`, `raw_sha256`, and `derived_from_capture_id`.
- [ ] Raw content is immutable after initial write — no overwrite on retry or re-delivery.
- [ ] Duplicate / idempotent delivery with matching hash does not create a second raw file.
- [ ] Duplicate delivery with mismatched hash fails hard (does not silently overwrite).
- [ ] Raw write failure prevents classification and sanitized note generation.
- [ ] `FileNoteRequest` carries `raw_text: str`.
- [ ] Sensitive captures receive the same lossless raw artifact as non-sensitive ones.
- [ ] Attachment-only captures produce a raw metadata artifact with an `## Attachments` section and no binary bytes.

---

## Tests

- Raw file exists after successful capture.
- Raw hash in raw file matches hash in sanitized note frontmatter.
- Re-delivery of same capture ID with same content does not create a second raw file (idempotent).
- Re-delivery of same capture ID with different content fails hard — no overwrite.
- Raw file content exactly matches original input (whitespace, code fences, line breaks).
- Raw file is present on disk before classification is invoked (ordering guarantee).
- Raw write failure: classification is not invoked; capture row stays retryable.
- Raw write success + sanitized write failure: retry reuses existing raw file without error.
- Sensitive capture: raw file is written with exact original text (Option A policy).
- Attachment-only capture: raw file has `## Attachments` section, no binary body.
- `raw_sha256` equals `SHA-256(body.encode("utf-8"))` — verified by recomputing from the file body.

---

**Done when:** every capture in the vault has a lossless raw counterpart, the hashes link them, re-delivery is idempotent, and all tests above pass.
