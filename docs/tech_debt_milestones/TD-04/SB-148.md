# SB-148 — SQLite contention instrumentation

**Branch:** `feature/sqlite-contention-instrumentation`
**Source:** TD-P2-007

---

## Goal

Before changing the serialized SQLite worker design, measure what is actually happening. Instrument the runtime first so architectural decisions are based on real data, not speculation.

---

## Acceptance criteria

- [ ] Add structured log events for:
  - SQLite queue depth (at enqueue time)
  - Queue wait duration (time from enqueue to worker pickup)
  - Job execution duration (time inside the worker thread)
  - Retry count per job
  - Busy-exhaustion count
- [ ] All logged values are numeric metadata — no raw exception bodies or query content.
- [ ] Do not implement an async SQLite adapter or change the serialized worker design in this ticket.
- [ ] Tests verify that instrumentation events are emitted with correct field names.

**Done when:** SQLite queue depth, wait duration, and execution duration are visible in structured logs and measurements can inform any future architecture decision.
