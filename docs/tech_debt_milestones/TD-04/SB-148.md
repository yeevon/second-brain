# SB-148 — SQLite contention instrumentation

**Branch:** `feature/sqlite-contention-instrumentation`
**Source:** TD-P2-007

---

## Goal

Before changing the serialized SQLite worker design, measure what is actually happening. Instrument the runtime first so architectural decisions are based on real data, not speculation.

---

## Current state

`sqlite_runtime.py` already emits several events. This ticket specifies exactly which events are required, their canonical names, fields, and units — and closes any remaining gaps.

---

## Required events and fields

All values are in **milliseconds** (never seconds, never mixed units). All fields are numeric metadata — no raw exception bodies, no SQL query content.

| Event name | When emitted | Fields |
|---|---|---|
| `sqlite_queue_depth` | At enqueue time (before `queue.put_nowait`) | `depth: int`, `operation_name: str` |
| `sqlite_queue_wait_ms` | When worker dequeues a job | `wait_ms: float`, `operation_name: str` |
| `sqlite_job_duration_ms` | After job completes (success or failure) | `duration_ms: float`, `operation_name: str` |
| `sqlite_busy_retry_count` | On each SQLITE_BUSY retry within `_execute_with_retry` | `attempt: int`, `operation_name: str`, `retrying: bool` |
| `sqlite_busy_exhausted_count` | When all retry attempts fail with SQLITE_BUSY | `operation_name: str` |

These event names are canonical. Do not add variants or aliases.

---

## Gap check against current implementation

Current `sqlite_runtime.py` emits:
- `sqlite_queue_depth` ✓ (at enqueue, includes `depth`)
- `sqlite_queue_wait_ms` ✓ (at dequeue, includes `wait_ms`)
- `sqlite_job_duration_ms` ✓ (after job, includes `duration_ms`)
- `sqlite_busy_retry_count` ✓ (inside retry loop, includes `attempt`, `retrying`)
- `sqlite_busy_exhausted_count` ✓ (after retry exhaustion)
- `sqlite_queue_full` — extra event for full-queue rejection (keep as-is)
- `sqlite_queue_wait_timeout` — extra event for wait timeout (keep as-is)
- `sqlite_job_slow` — extra event for slow job (keep as-is)

**If all five required events are already present with the correct names and fields, this ticket is a verification + test coverage task, not a new instrumentation task.**

---

## What this ticket delivers

1. Audit `sqlite_runtime.py` against the table above. If any event is missing, misnamed, or missing a required field, fix it.
2. Add unit tests that verify each of the five required events is emitted with correct field names (not just that no exception is raised).
3. Confirm all field values are in milliseconds (convert any seconds-based values).
4. Do **not** implement an async SQLite adapter or change the serialized worker design.

---

## Acceptance criteria

- [ ] All five events in the table above are emitted with the exact names and fields specified.
- [ ] All duration/wait fields are in milliseconds (float or int).
- [ ] Unit tests verify each event is emitted with correct field names under the relevant conditions.
- [ ] No raw exception bodies or SQL query content in any event payload.
- [ ] No architectural changes to `SQLiteRuntime`.

**Done when:** SQLite queue depth, wait duration, execution duration, retry count, and busy exhaustion are all visible in structured logs with consistent names and units, and each is covered by a test.
