# SB-142 — Durable receipt-repair tracking

**Branch:** `feature/durable-receipt-repair`
**Source:** TD-P2-001

---

## Goal

When terminal capture state commits but the Discord receipt update fails, the visible receipt is out of sync with the actual state. Operators currently have no durable way to detect or repair this divergence.

---

## Schema changes

Add three columns to the `captures` table via a new migration in `migrations.py`:

```sql
ALTER TABLE captures ADD COLUMN receipt_sync_status TEXT NOT NULL DEFAULT 'clean';
ALTER TABLE captures ADD COLUMN receipt_sync_last_attempt_at TEXT;
ALTER TABLE captures ADD COLUMN receipt_sync_last_error_type TEXT;
```

**`receipt_sync_status` values (exhaustive):**

| Value | Meaning |
|---|---|
| `clean` | Receipt is in sync with capture state (default) |
| `failed` | All fallback delivery paths exhausted; receipt is out of sync |
| `pending_repair` | A repair attempt is queued or in progress |
| `not_applicable` | Capture was never eligible for a Discord receipt (e.g., API-only ingestion) |

**State transition rules:**

1. On successful receipt delivery: set `receipt_sync_status = 'clean'`.
2. On receipt failure after all fallback paths are exhausted: set `receipt_sync_status = 'failed'`, `receipt_sync_last_attempt_at` (ISO-8601 UTC), `receipt_sync_last_error_type` (safe error type string, no raw body).
3. Receipt failure **must not** roll back or modify `status`, `delivery_status`, or any other capture state column. The write to receipt sync columns is a separate Ledger call after the terminal state is already committed.
4. A subsequent successful repair sets `receipt_sync_status = 'clean'` and clears `receipt_sync_last_error_type`.

**TD-04 scope boundary:** this ticket tracks repair _visibility_ only. An automated repair job or operator command is out of scope.

---

## `status` output

`uv run python -m secondbrain status` must include receipt sync state. For any capture where `receipt_sync_status != 'clean'`, surface:
- `capture_id`
- `receipt_sync_status`
- `receipt_sync_last_attempt_at`
- `receipt_sync_last_error_type`

Captures with `receipt_sync_status = 'clean'` or `'not_applicable'` are omitted from this output.

---

## Acceptance criteria

- [ ] Migration adds the three columns with the correct defaults (backwards-compatible; existing rows default to `'clean'`).
- [ ] Receipt failure after committed terminal state writes `receipt_sync_status = 'failed'` and does **not** modify capture status.
- [ ] Successful receipt delivery writes `receipt_sync_status = 'clean'`.
- [ ] `status` output surfaces out-of-sync receipts.
- [ ] Tests cover:
  - Successful receipt update — `receipt_sync_status` is `'clean'`
  - Discord receipt failure after committed state — `receipt_sync_status` is `'failed'`, committed state is unchanged
  - Receipt repair (simulate late success) — `receipt_sync_status` resets to `'clean'`

**Done when:** a failed receipt update is durably visible in `status` output and the committed capture state is never affected.
