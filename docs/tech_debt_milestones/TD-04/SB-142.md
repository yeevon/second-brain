# SB-142 — Durable receipt-repair tracking

**Branch:** `feature/durable-receipt-repair`
**Source:** TD-P2-001

---

## Goal

When terminal capture state commits but the Discord receipt update fails, the visible receipt is out of sync with the actual state. Operators currently have no durable way to detect or repair this divergence.

---

## Acceptance criteria

- [ ] Add durable receipt sync marker columns:
  - `receipt_sync_status`
  - `receipt_sync_last_attempt_at`
  - `receipt_sync_last_error_type`
- [ ] Mark receipt sync failure after all fallback delivery paths fail.
- [ ] Include receipt sync status in `uv run secondbrain status` output.
- [ ] Receipt failure never rolls back committed capture state.
- [ ] Tests cover:
  - Successful receipt update — sync status is clean
  - Discord receipt failure after committed state — sync status is marked failed
  - Committed capture state is preserved regardless of receipt outcome

**Done when:** a failed receipt update is durably visible in `status` output and the committed capture state is never affected.
