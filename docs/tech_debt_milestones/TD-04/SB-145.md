# SB-145 — Explicit invalid-classifier fallback in n8n Intake

**Branch:** `feature/invalid-classifier-fallback`
**Source:** TD-P2-004 / GitHub #16

---

## Goal

When the Gemini classifier returns `valid=false` or `route=null`, the capture currently waits for stale-lease recovery to pick it up. This is unnecessary latency and obscures the root cause. Invalid classifier output must route immediately to a defined failure path.

---

## Failure taxonomy and routing

| Condition | Class | Retryable | n8n callback endpoint | `reason_type` |
|---|---|---|---|---|
| `valid=false` in parsed response | Invalid output | `true` | `schedule-retry` | `invalid_classifier_output` |
| `route=null` in parsed response | Invalid output | `true` | `schedule-retry` | `invalid_classifier_output` |
| Gemini response is not valid JSON | Malformed output | `true` | `schedule-retry` | `malformed_gemini_output` |
| `valid=true`, low confidence, `route=inbox` | Normal inbox path | n/a | existing inbox filing path | n/a |

**Retry vs terminal:** `invalid_classifier_output` and `malformed_gemini_output` are retryable. The stale-lease reaper already applies a `max_retry_attempts` ceiling and marks `FAILED` at exhaustion — no new terminal-failure logic is needed here.

**What this ticket does NOT change:** schema-valid but low-confidence classifier output with `route=inbox` continues through the existing inbox path unchanged.

---

## Implementation surface

This is implemented in the **n8n Intake workflow** only. After the Gemini response parse step:

1. Add a validation node that checks `valid`, `route`, and JSON parse success.
2. Add a Switch/If branch: if validation fails → call `schedule-retry` with appropriate `reason_type`.
3. If validation passes → existing classification routing unchanged.

No capture-service schema changes are required. `reason_type` is persisted via the existing `schedule-retry` callback.

---

## Logging

Each invalid-classifier route must emit a structured log event from capture-service (written on the callback handler), e.g.:
```json
{"event": "invalid_classifier_routed", "capture_id": "...", "reason_type": "invalid_classifier_output"}
```
No raw Gemini response content in the log or in `captures.last_error`.

---

## Acceptance criteria

- [ ] `valid=false` routes immediately to `schedule-retry` with `reason_type = 'invalid_classifier_output'`.
- [ ] `route=null` routes immediately to `schedule-retry` with `reason_type = 'invalid_classifier_output'`.
- [ ] Non-JSON Gemini response routes to `schedule-retry` with `reason_type = 'malformed_gemini_output'`.
- [ ] Valid classifier output with `route=inbox` continues through existing path unchanged.
- [ ] Route decision is logged with a safe event type; no raw classifier output persisted.
- [ ] Architecture test (`tests/architecture/test_n8n_*.py`) verifies the Intake workflow has the invalid-output branch.
- [ ] Tests cover:
  - `valid=false` — immediate retry route
  - `route=null` — immediate retry route
  - Malformed JSON — immediate retry route
  - Valid output with `route=file` — normal filing path unchanged
  - Valid output with `route=inbox` — normal inbox path unchanged

**Done when:** schema-invalid Gemini output is handled immediately without waiting for stale-lease recovery, and the routing is covered by tests.
