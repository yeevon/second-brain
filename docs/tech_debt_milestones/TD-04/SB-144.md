# SB-144 — Handle attachment-only captures before text security screening

**Branch:** `feature/attachment-only-capture-path`
**Source:** TD-P2-003 / GitHub #16

---

## Goal

Attachment-only Discord messages produce empty `raw_text`. `SecurityScreenRequest.text` has `min_length=1` (`api_models.py:214`), so an attachment-only capture with `raw_text == ""` will fail validation before reaching classification.

---

## Decision

**Attachment-only captures skip text secret screening.**

The routing rule is:
- If `raw_text == ""` AND `has_attachments == True`: bypass `SecurityScreenRequest` entirely and proceed directly to classification/delivery.
- If `raw_text` is non-empty: existing screening path unchanged.

This is a pre-screening gate, not a change to `SecurityScreenRequest` itself.

### Why not change `SecurityScreenRequest.min_length`?

Changing `min_length=1` to `min_length=0` on `SecurityScreenRequest.text` would allow the endpoint to accept empty strings for all callers, which is incorrect. The skip must be a routing decision at the capture handler level, not a schema relaxation.

---

## Classifier behavior for attachment-only captures

- `raw_text` passed to n8n / in-process classifier is `""` (empty string) or `None`.
- The classifier (Gemini prompt / n8n payload) receives attachment metadata from `has_attachments` and the attachment list — no binary bytes.
- Expected classifier outcome: route to `inbox` with an `inbox_reason` of `attachment_only` or similar, unless the attachment metadata is sufficient to classify confidently.
- Operator-visible outcome: capture accepted, receipt shows ⏳ then routes to inbox on callback.

---

## Raw substrate behavior

Attachment-only captures must produce a raw substrate file (aligning with TD-03 behavior). `build_raw_body("", attachments)` in `writer.py` already handles this case — it writes an `## Attachments` section when `raw_text` is empty. No change needed in writer-service.

---

## Acceptance criteria

- [ ] Attachment-only capture (`raw_text == ""`, `has_attachments == True`) is accepted and does not hit `SecurityScreenRequest` validation.
- [ ] Routing gate is implemented in `capture_service.py` (or `worker.py`), not in `api_models.py`.
- [ ] Classification proceeds with attachment metadata; `raw_text` passed as empty string or omitted per existing classifier contract.
- [ ] Capture is accepted and produces a receipt; routes to inbox.
- [ ] Architecture test or workflow test covers the attachment-only path explicitly.
- [ ] Tests cover:
  - Attachment-only message — accepted and routed
  - Empty `raw_text` alone (no attachments) — existing rejection behavior preserved
  - Mixed (text + attachment) — existing path unchanged

**Done when:** an attachment-only Discord message is captured successfully and the path is covered by an architecture or workflow test.
