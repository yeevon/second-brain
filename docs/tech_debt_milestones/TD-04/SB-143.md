# SB-143 — Split Gemini error handling by failure class

**Branch:** `feature/gemini-error-classification`
**Source:** TD-P2-002 / GitHub #8

---

## Goal

All Gemini HTTP failures currently map to a single generic retry path. Rate limits, server errors, and credential failures have different causes and require different handling — collapsing them into one path masks problems and makes remediation harder.

---

## Failure class definitions

| HTTP status | Class | `reason_type` | Endpoint | Action |
|---|---|---|---|---|
| `429` | Rate limited | `gemini_rate_limited` | `schedule-retry` | Backoff and schedule retry |
| `5xx` or timeout | Server / transient | `gemini_server_or_timeout` | `schedule-retry` | Backoff and schedule retry |
| `401` / `403` | Auth / credential | `gemini_auth_failed` | `acknowledge-failed` | Terminal; log event; do not retry |

`reason_type` is a safe string — no raw HTTP bodies, no response content persisted anywhere.

---

## Implementation surface

This is implemented in the **n8n Intake workflow** only. The capture-service callback endpoints (`schedule-retry`, `acknowledge-failed`, `report-workflow-error`) already accept `reason_type`. No capture-service schema changes are required.

The n8n Switch node after the Gemini HTTP call branches on response status:
- `429` → rate-limit branch
- `500`, `502`, `503`, `504` → server/timeout branch
- `401`, `403` → auth-failed branch
- timeout (n8n execution error) → server/timeout branch (same path)
- All other non-2xx → server/timeout branch (safe default)

Each branch calls the appropriate capture-service callback with the correct `reason_type`. Retryability is encoded by endpoint choice — do not add a `retryable` field to existing callback schemas:
- `schedule-retry` → retryable path (`ScheduleRetryRequest` takes `delivery_attempt`, `error_type`, `reason_type`)
- `acknowledge-failed` → terminal path (`AcknowledgeDeliveryFailedRequest` takes `delivery_attempt`, `reason_type`)
- `report-workflow-error`, if used, sets `disposition` to `retryable` or `terminal`

---

## Acceptance criteria

- [ ] `429` routes to retry with `reason_type = 'gemini_rate_limited'`.
- [ ] `5xx` and timeout route to retry with `reason_type = 'gemini_server_or_timeout'`.
- [ ] `401` / `403` route to terminal failure via `acknowledge-failed` with `reason_type = 'gemini_auth_failed'`.
- [ ] Error payloads are sanitized — no raw HTTP response bodies are persisted in `captures.last_error` or `capture_events`.
- [ ] Architecture test (`tests/architecture/test_n8n_*.py`) verifies the Intake workflow has the three-branch Switch node.
- [ ] Tests cover each failure class with a distinct outcome and `reason_type`.

**Done when:** each Gemini failure class produces a distinct, correctly handled outcome and the paths are separately tested.
