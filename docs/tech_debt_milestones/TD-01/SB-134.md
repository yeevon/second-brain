# SB-134 — Preserve no-op correction move results

**Branch:** `feature/correction-noop-move`
**Source:** TD-P0-005

---

## Goal

A same-folder correction (no actual move) should not be reported as a real move and must not erase the prior delivery commit metadata stored on the capture. Currently a no-op move may overwrite the original `git_commit_hash` with a new value or null, making the audit trail inaccurate.

---

## Acceptance criteria

- [ ] Writer-service move response distinguishes `moved` from `no_op`.
- [ ] Capture-service preserves the original commit hash when the move response is `no_op`.
- [ ] Correction history records the outcome as `no_op` accurately.
- [ ] Tests cover:
  - Same-folder correction — original hash preserved, history shows no_op
  - Real move — new hash stored, history shows moved

**Done when:** a same-folder correction leaves the capture's original commit hash and audit history intact.
