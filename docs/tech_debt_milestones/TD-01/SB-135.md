# SB-135 — Verify writer-service production startup and deploy-key handling

**Branch:** `feature/writer-service-deploy-key`
**Source:** TD-P0-006

---

## Goal

The EC2 hotfix removed the `cap_drop: ALL` blocker for writer-service startup, but end-to-end Git sync has not been verified in production. The security posture is incomplete until deploy key handling is confirmed and documented.

---

## Acceptance criteria

- [ ] EC2 writer-service starts healthy.
- [x] Runtime user can access the copied SSH deploy key and `known_hosts` safely (correct path and permissions).
- [ ] Writer-service can complete: `git fetch`, fast-forward, commit, push with `GIT_SYNC_ENABLED=true`.
- [x] Service fails fast with a clear error when key or `known_hosts` are missing.
- [x] Local named-volume / fake-remote mode does not require GitHub deploy keys.
- [x] Compose/config tests cover both modes:
  - Local fake remote (no deploy key)
  - GitHub-backed mode (deploy key present)
- [x] Production security posture documented: which secrets live where, what file permissions are required.

**Done when:** full Git sync is verified on EC2 end-to-end and the security posture is documented.
