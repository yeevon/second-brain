# SB-149 — Startup reconciliation deferred-capture log clarity

**Branch:** `fix/startup-reconciliation-log-message`
**Source:** TD-P3-001 / GitHub #5

---

## Goal

Startup/history reconciliation logs `downstream processing disabled` for intentionally deferred historical messages. This looks like an error when it is normal behavior.

---

## Root cause

In `src/secondbrain/capture_service.py`, `_persist_accepted_capture` logs `capture_deferred` with `reason="downstream processing disabled"` whenever `notify_downstream=False` OR when `self._notify_capture is None` even if downstream *is* enabled at the service level. The two cases are currently conflated into one log line.

Startup reconcile always calls `make_capture_handler(notify_downstream=False)`, so every capture replayed at startup emits the same `reason` as a service-level disablement.

---

## Implementation

**File:** `src/secondbrain/capture_service.py` — `_persist_accepted_capture`

Change the `elif not self.settings.downstream_delivery_enabled` branch to emit a more granular reason. The `queued` variable already tracks whether this capture was actually forwarded. Add a second condition:

```python
queued = notify_downstream and self._notify_capture is not None
if queued:
    await self._notify_capture(capture.capture_id)
elif not notify_downstream:
    log_metadata(
        "capture_deferred",
        capture_id=capture.capture_id,
        discord_message_id=capture.discord_message_id,
        reason="historical_reconciliation_deferred",
    )
elif not self.settings.downstream_delivery_enabled:
    log_metadata(
        "capture_deferred",
        capture_id=capture.capture_id,
        discord_message_id=capture.discord_message_id,
        reason="downstream processing disabled",
    )
```

The priority of the two `elif` branches matters: check `not notify_downstream` first, because `notify_downstream=False` is the caller's intent (reconcile path), whereas `downstream_delivery_enabled=False` is a service-level config. The two can overlap but the caller's intent is the more specific signal.

**Effect on `CaptureDisposition.queued`:** no change. `queued` is already `False` in both cases; callers are unaffected.

**Affects both startup reconcile and periodic reconcile.** Periodic reconcile also calls `make_capture_handler(notify_downstream=True)` in `capture-only` mode, so periodic captures do queue normally and will not emit `capture_deferred` at all.

---

## Acceptance criteria

- [ ] Historical/startup messages (reconcile path, `notify_downstream=False`) log `capture_deferred` with `reason="historical_reconciliation_deferred"`.
- [ ] Service-level disabled (`downstream_delivery_enabled=False` + `notify_downstream=True`) still logs `reason="downstream processing disabled"`.
- [ ] `CaptureDisposition.queued` is unchanged in both cases.
