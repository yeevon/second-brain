# SB-127 — Configure n8n private access layer

**Branch:** `feature/n8n-private-access`

---

## Goal

Expose the n8n UI over a private, authenticated channel so it is not directly reachable from the public internet. Architecture section 9.1 requires this before production operation.

---

## Acceptable access methods

Choose one:

**Option A — Tailscale (recommended)**
- Install Tailscale on EC2 and the operator's machine.
- Access n8n at `http://<ec2-tailscale-ip>:5678` over the Tailscale network.
- Port 5678 remains closed on the EC2 security group for all non-Tailscale traffic.

**Option B — Cloudflare Tunnel**
- Install `cloudflared` on EC2.
- Configure a Cloudflare Tunnel to forward to `localhost:5678`.
- Protect the tunnel URL with Cloudflare Access (email or One-Time Pin).

**Option C — SSH tunnel (for rare admin access)**
- Forward port 5678 over SSH: `ssh -L 5678:localhost:5678 ec2-host`.
- Acceptable for infrequent access; does not require ongoing infrastructure.

---

## Checklist

- [ ] n8n UI is accessible through the chosen private access method from the operator's machine.
- [ ] n8n UI is NOT accessible via `http://<public-ec2-ip>:5678` from an external network.
- [ ] If using a reverse proxy, `WEBHOOK_URL`, `N8N_PROXY_HOPS`, and forwarded header configuration are set and verified.
- [ ] n8n `N8N_SECURE_COOKIE=false` setting is removed or overridden if the access layer now provides HTTPS.
- [ ] The chosen access method and URL are documented in the operations runbook (SB-129).

---

## Acceptance criteria

- The n8n UI loads through the private access layer.
- A direct request to `http://<ec2-public-ip>:5678` returns no response (connection refused or security-group drop).
- Workflow webhooks called from within the Docker network (container-to-container) continue to work.

**Done when:** n8n is only accessible via the private channel and the webhook internal traffic is unaffected.
