# SB-137 — Vault-update-service (validate and apply proposals)

**Branch:** `feature/v3-vault-update-service`

---

## Goal

Extend `writer-service` with a proposal-apply endpoint. When `capture-service` marks a proposal APPROVED, `writer-service` fetches the proposal, validates it, applies the change under the existing `flock` writer lock, commits a Git commit, and reports back.

The V3 write path reuses the same lock and Git serialization used for filing and corrections. No new concurrency mechanisms are introduced.

---

## writer-service internal API extension

```text
POST /internal/vault/apply-proposal
    Body: { proposal_id }
    Response: { proposal_id, changed_path, commit_hash, audit_record }
```

The endpoint:
1. Fetches the full proposal from `capture-service` via `GET /internal/vault/proposals/:proposal_id`.
2. Verifies proposal status is `APPROVED` — rejects any other status.
3. Acquires the OS advisory `flock` writer lock.
4. Runs the apply sequence under lock.
5. Releases the lock.
6. Calls back to `capture-service` to update proposal status.

---

## Apply sequence (under flock)

```text
git fetch origin
git merge --ff-only origin/main
verify clean worktree
resolve target_note_path against vault root
reject path traversal and hidden paths
verify target file exists
if target_anchor_json is present: verify anchor still matches current file content
execute operation-specific mutation
append audit record to 99_log/events.ndjson
git add changed files
git commit with structured commit message
git push origin main
return changed_path, commit_hash, audit_record
```

---

## Operation implementations (initial set)

Each operation receives the validated proposal and the resolved absolute file path.

| Operation | What it does |
|-----------|-------------|
| `mark_task_done` | Changes `- [ ] <task>` to `- [x] <task>` for the anchor task text |
| `mark_task_open` | Changes `- [x] <task>` to `- [ ] <task>` for the anchor task text |
| `set_task_due_date` | Sets or replaces `due_date` in the note frontmatter |
| `set_task_priority` | Sets or replaces `priority` in the note frontmatter |
| `append_task` | Appends `- [ ] <new task>` under the `## Actions` section |
| `append_note_section` | Appends a new named section at the end of the note body |
| `move_note_to_folder` | Moves the file to a different allowed vault folder via `git mv` |
| `add_project_tag` | Adds a tag to the `tags:` frontmatter list |
| `add_weekly_review_entry` | Appends a structured review entry under `90_digests/` |

### Invariants for all operations

- Reject path traversal (`../`, absolute paths outside vault root, hidden paths starting with `.`).
- Reject unsupported file types (only `.md` files are valid targets).
- Reject raw full-file rewrites (no operation may replace the entire file content).
- Verify the anchor still matches current file content before mutation. If the anchor is stale, return failure without applying the change.
- All operations must be idempotent where semantically reasonable (e.g. marking an already-done task as done is a no-op, not an error).

## Audit record

Append to `99_log/events.ndjson`:

```json
{
  "event": "VAULT_UPDATE_APPLIED",
  "proposal_id": "VUP-20260616-0001",
  "operation": "mark_task_done",
  "target_path": "20_projects/second-brain/example.md",
  "commit": "abc123",
  "timestamp": "2026-06-16T10:30:00-05:00"
}
```

## Acceptance criteria

- `POST /internal/vault/apply-proposal` with a PENDING (not APPROVED) proposal returns 422.
- An approved `mark_task_done` proposal changes the correct task checkbox and creates a Git commit.
- An approved `move_note_to_folder` proposal moves the file to the new folder and updates `captures.derived_note_path`.
- A stale anchor (task text no longer matches file content) returns failure without modifying the vault.
- Path traversal in `target_note_path` is rejected before any file is opened.
- A hidden path (starting with `.`) is rejected.
- Two near-simultaneous apply requests serialize correctly under the `flock` lock.
- A failed apply leaves the vault unchanged and marks the proposal FAILED.
- The audit record is appended to `99_log/events.ndjson` on every successful apply.

**Done when:** all acceptance criteria pass with tests covering apply success, stale anchor, path rejection, and concurrent apply serialization.
