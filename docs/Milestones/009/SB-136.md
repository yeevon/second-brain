# SB-136 — Vault-update-proposal schema and storage

**Branch:** `feature/v3-proposal-schema`

---

## Goal

Define the data contract for vault update proposals and the storage layer that persists them. A proposal is structured data that describes a desired change — it is never executable code. Nothing in this ticket applies changes to the vault.

---

## SQLite schema

Add a new table to the migration sequence:

```sql
CREATE TABLE vault_update_proposals (
    proposal_id TEXT PRIMARY KEY,
    source TEXT NOT NULL,
    requested_by TEXT NOT NULL,
    operation TEXT NOT NULL,
    target_note_path TEXT NOT NULL,
    target_anchor_json TEXT,
    change_json TEXT NOT NULL,
    reason TEXT,
    status TEXT NOT NULL DEFAULT 'PENDING',
    requires_approval INTEGER NOT NULL DEFAULT 1,
    submitted_at TEXT NOT NULL,
    reviewed_at TEXT,
    reviewed_by TEXT,
    applied_at TEXT,
    rejected_reason TEXT,
    git_commit_hash TEXT,
    last_error TEXT
);

CREATE INDEX idx_vault_update_proposals_status
    ON vault_update_proposals(status, submitted_at);
```

### Status lifecycle

```text
PENDING      — proposal submitted, awaiting approval
APPROVED     — user approved; apply has not run yet
REJECTED     — user rejected; never applied
APPLYING     — apply is in progress under writer lock
APPLIED      — writer-service committed the change
FAILED       — apply attempt failed; proposal is retryable or requires repair
```

### Proposal ID format

```text
VUP-YYYYMMDD-NNNN
```

Generated with the same transactional counter pattern used for `capture_id`.

---

## Allowed operations (initial set)

```text
mark_task_done
mark_task_open
set_task_due_date
set_task_priority
append_task
append_note_section
move_note_to_folder
add_project_tag
add_weekly_review_entry
```

Any `operation` value not in this list must be rejected at proposal-creation time.

---

## Internal API

Add to `capture-service` internal API (shared-secret auth required):

```text
POST /internal/vault/proposals
    Body: proposal JSON (see architecture section 18 for example shape)
    Response: { proposal_id, status }
    Validates: operation is in allowlist, target_note_path does not escape vault root

GET  /internal/vault/proposals/:proposal_id
    Response: full proposal record

GET  /internal/vault/proposals?status=PENDING
    Response: list of proposals matching status filter, most recent first, limit 50
```

---

## Acceptance criteria

- `vault_update_proposals` table exists and migration applies cleanly.
- `POST /internal/vault/proposals` with an unsupported `operation` returns 422.
- `POST /internal/vault/proposals` with a path containing `..` or an absolute path that escapes the configured vault root returns 422.
- `GET /internal/vault/proposals/:proposal_id` returns 404 for an unknown ID.
- `GET /internal/vault/proposals?status=PENDING` returns only PENDING proposals.
- Proposal IDs use the `VUP-YYYYMMDD-NNNN` format and are unique.
- Tests cover: create proposal, read proposal, list by status, unsupported operation rejection, path traversal rejection.

**Done when:** the schema is migrated, the API endpoints exist, and all acceptance criteria pass.
