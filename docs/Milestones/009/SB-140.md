# SB-140 — V3 security hardening and acceptance tests

**Branch:** `feature/v3-security-tests`

---

## Goal

Harden the V3 write path against the threat model defined in the architecture and verify all V3 acceptance criteria from `docs/ARCHITECTURE.md` section 18.

This ticket is the validation gate for Milestone 9. Do not declare V3 complete until all items here pass.

---

## Threat model: what V3 must defend against

From the architecture:

| Threat | Defense |
|--------|---------|
| Prompt injection inside vault notes | Treat note text as untrusted input when generating proposals; do not re-evaluate vault content as LLM instructions |
| LLM hallucinated file paths | Path validation at proposal creation and at apply time; require exact path match, not fuzzy match |
| LLM marking tasks complete based on implication | Completion rule: only propose completion when user explicitly says so, the note has `done:` prefix, or user approves a proposal that clearly marks it complete |
| Accidental edits to archived notes | Check `lifecycle_status` frontmatter; reject apply on notes with `lifecycle_status: archived` |
| Hidden file or `.git` access | Reject any path starting with `.` or containing `/.git/` |
| Broad regex replacements | Operations are surgical (specific task text, specific field); no regex-replace-all operations |
| Silent movement of notes between folders | `move_note_to_folder` always confirms the source and destination in the approval message |
| Update races against human Obsidian edits | Anchor verification checks current content before mutation; mismatch → failure, not overwrite |

## Security test cases

### Path validation

- [ ] Proposal with `../` in `target_note_path` is rejected at creation.
- [ ] Proposal with an absolute path outside the vault root is rejected at creation.
- [ ] Proposal targeting a hidden path (`.obsidian/config.md`) is rejected at creation.
- [ ] Proposal targeting `.git/COMMIT_EDITMSG` is rejected at creation.
- [ ] Apply attempt with a path that resolves outside vault root after joining is rejected.

### Anchor verification

- [ ] Apply with a task text anchor that has been edited since proposal creation → apply fails, vault unchanged.
- [ ] Apply with a task text anchor that has been completed since proposal creation → apply returns stale-anchor error.
- [ ] Apply with a valid anchor → task state changes correctly, commit produced.

### Lifecycle protection

- [ ] Proposal targeting a note with `lifecycle_status: archived` frontmatter is rejected at apply time.
- [ ] Proposal targeting a note with `lifecycle_status: superseded` frontmatter is rejected at apply time.
- [ ] Proposal targeting `lifecycle_status: active` applies normally.

### Rejected proposals do not modify the vault

- [ ] A REJECTED proposal cannot be approved after rejection.
- [ ] Calling apply-proposal with a REJECTED proposal ID returns 422.
- [ ] The vault is unchanged after a proposal is rejected.

### Unsupported operations

- [ ] Proposal with `operation: delete_note` is rejected at creation.
- [ ] Proposal with `operation: overwrite_note` is rejected at creation.
- [ ] Proposal with `operation: git_push_direct` is rejected at creation.
- [ ] Proposal with `operation: modify_raw_capture` is rejected at creation.
- [ ] Proposal with `operation: bulk_edit` is rejected at creation.

### Completion rule

- [ ] `propose_task_completion` with `reason` that contains no explicit completion signal logs a warning and includes the reason verbatim in the proposal (user sees it before approving).
- [ ] The apply operation never infers completion — it only applies what is in the approved proposal.

### Invariant verification (from architecture)

- [ ] Read-only MCP profile (`brain-mcp`) remains unchanged and read-only.
- [ ] Proposal tools in `brain-mcp-propose` cannot trigger apply directly.
- [ ] Every applied proposal creates a Git commit.
- [ ] Every proposal has an audit trail (event in `capture_events` or `99_log/events.ndjson`).
- [ ] Rejected proposals produce an audit event but no vault change.
- [ ] Bulk edits (multiple notes in one proposal) are not supported.
- [ ] The raw capture ledger (`captures` table) is never modified by V3 tooling.

## V3 acceptance criteria (from architecture section 18)

- [ ] Read-only MCP remains available and unchanged.
- [ ] Proposal tools cannot mutate files directly.
- [ ] At least one approval surface exists (Discord — SB-138).
- [ ] Approved proposals are applied only by `writer-service`/update-service.
- [ ] Every applied proposal creates a Git commit.
- [ ] Every proposal has an audit trail.
- [ ] Rejected proposals do not change the vault.
- [ ] Tests prove unsupported operations are rejected.
- [ ] Tests prove path traversal and hidden paths are rejected.
- [ ] Tests prove stale anchors are rejected.
