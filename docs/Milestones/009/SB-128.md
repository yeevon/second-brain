# SB-128 — Schedule encrypted nightly backups on EC2

**Branch:** `feature/ec2-backup-schedule`

---

## Goal

Put `deploy/backup.sh` on a nightly cron schedule on EC2, confirm it produces encrypted output, and confirm restore validation succeeds. The architecture treats GitHub as a remote replica, not the only backup — off-host encrypted backups are required.

SB-119 wrote the scripts. This ticket schedules and validates them in the production environment.

---

## What must be backed up

| Target | Script | Notes |
|--------|--------|-------|
| SQLite ledger | `deploy/backup.sh` | Uses SQLite `.backup` API — never a raw `cp` of a live WAL-mode file |
| EC2 vault clone | `deploy/backup.sh` | Creates a portable `git bundle`; same invocation as above |
| n8n data volume | `deploy/backup.sh` | Optional; set `N8N_DATA_DIR` env var to include |

---

## Checklist

### Cron schedule

- [ ] `deploy/backup.sh` runs nightly via cron (e.g. 2 am EC2 local time) with required env vars set.
- [ ] n8n volume snapshot is included (set `N8N_DATA_DIR` in the cron environment).
- [ ] Cron output is logged to a file readable by the operator.

### Encryption

- [ ] Backup archives are encrypted before being stored off-host (e.g. GPG with a strong key, or an encrypted S3 bucket policy).
- [ ] The encryption key or passphrase is stored separately from the backup destination.
- [ ] No plaintext backup of the SQLite ledger is left unencrypted on the EC2 host beyond the retention window.

### Off-host destination

- [ ] Encrypted backups are transferred to an off-EC2 destination (S3, a second host, or a local machine via SCP/rsync).
- [ ] Backup destination is not the same EC2 host (a host failure must not destroy both live data and backup).

### Restore validation

- [ ] `deploy/restore-validate.sh` decrypts and validates a recent backup into a temporary directory.
- [ ] The restore test runs weekly (cron or manual verification).
- [ ] The restore test never touches live volumes.
- [ ] A successful restore is confirmed by `deploy/restore-validate.sh`'s integrity check and row count output.

### Status visibility

- [ ] `secondbrain status` reports: last successful backup timestamp (ledger, vault), last successful restore validation timestamp.

---

## Acceptance criteria

- A nightly cron job produces an encrypted backup of the ledger and vault.
- The backup lands off-host.
- `deploy/restore-validate.sh` successfully decrypts and validates a backup into a temp directory without touching live data.
- `secondbrain status` shows backup timestamps.

**Done when:** the first nightly cron run completes successfully, the restore test passes, and `secondbrain status` reports the timestamps.
