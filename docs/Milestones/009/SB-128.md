# SB-128 — Schedule encrypted nightly backups on EC2

**Branch:** `feature/ec2-backup-schedule`

---

## Goal

Put the backup scripts from `scripts/` on a nightly cron schedule on EC2, confirm they produce encrypted output, and confirm restore validation succeeds. The architecture treats GitHub as a remote replica, not the only backup — off-host encrypted backups are required.

SB-119 wrote the scripts. This ticket schedules and validates them in the production environment.

---

## What must be backed up

| Target | Script | Notes |
|--------|--------|-------|
| SQLite ledger | `scripts/backup-ledger.sh` | Must use `.backup` or equivalent — never a raw `cp` of a live WAL-mode file |
| EC2 vault clone | `scripts/backup-vault.sh` | Full clone or incremental archive |
| n8n data volume | Manual or script | Encryption required before leaving host |

---

## Checklist

### Cron schedule

- [ ] `scripts/backup-ledger.sh` runs nightly via cron (e.g. 2 am EC2 local time).
- [ ] `scripts/backup-vault.sh` runs nightly via cron (e.g. 2:15 am).
- [ ] n8n volume snapshot is scheduled.
- [ ] Cron output is logged to a file readable by the operator.

### Encryption

- [ ] Backup archives are encrypted before being stored off-host (e.g. GPG with a strong key, or an encrypted S3 bucket policy).
- [ ] The encryption key or passphrase is stored separately from the backup destination.
- [ ] No plaintext backup of the SQLite ledger is left unencrypted on the EC2 host beyond the retention window.

### Off-host destination

- [ ] Encrypted backups are transferred to an off-EC2 destination (S3, a second host, or a local machine via SCP/rsync).
- [ ] Backup destination is not the same EC2 host (a host failure must not destroy both live data and backup).

### Restore validation

- [ ] `scripts/restore-test.sh` restores a recent backup into a temporary directory.
- [ ] The restore test runs weekly (cron or manual verification).
- [ ] The restore test never touches live volumes.
- [ ] A successful restore is confirmed by inspecting the restored ledger (e.g. row count matches expected).

### Status visibility

- [ ] `sb status` reports: last successful backup timestamp (ledger, vault), last successful restore validation timestamp.

---

## Acceptance criteria

- A nightly cron job produces an encrypted backup of the ledger and vault.
- The backup lands off-host.
- `scripts/restore-test.sh` successfully restores a backup into a temp directory without touching live data.
- `sb status` shows backup timestamps.

**Done when:** the first nightly cron run completes successfully, the restore test passes, and `sb status` reports the timestamps.
