# SB-103 — Add the internal capture-service API

**Branch:** `feature/capture-service-api`

## Goal

Expose the in-process `CaptureService` boundary created in SB-102 through a small authenticated HTTP API.

The API exists so future n8n workflows and `writer-service` can interact with capture state without opening SQLite directly.

```text
future n8n or writer-service
    ↓
private authenticated HTTP API
    ↓
CaptureService
    ↓
Ledger
    ↓
SQLite
```

The API must not bypass `CaptureService`.

---

## Precondition

Do not begin SB-103 until SB-102 is complete and approved.

SB-102 must already guarantee:

```text
CaptureService is the only production owner of Ledger
worker.py does not mutate Ledger directly
app.py does not open Ledger directly
reconcile.py does not mutate Ledger directly
receipt coordination lives behind CaptureService
```

SB-103 wraps that boundary. It must not repair an incomplete SB-102 extraction by reaching around it.

---

## Why this ticket exists

The canonical architecture requires a private internal API bound to localhost or a private Docker network. It defines the initial route surface and requires authentication for state-changing operations.

The production ownership rule is stricter:

```text
Only capture-service opens SQLite directly.
n8n and writer-service use the capture-service API.
```

SB-103 prepares that interaction boundary without adding n8n, Docker, or EC2 deployment yet.

---

## Design decisions

## 1. Keep one process

The local runtime remains one Python process:

```text
single Python process
├── Discord Gateway client
├── background classifier worker
├── CaptureService
├── Ledger and SQLite connection
└── internal FastAPI server
```

Do not run:

```bash
uvicorn secondbrain.capture_api:app
```

as a separate second process if that would instantiate another `CaptureService` and open a second SQLite owner.

Instead, start Uvicorn programmatically inside the existing runtime process.

---

## 2. Bind locally by default

Use:

```dotenv
CAPTURE_API_HOST=127.0.0.1
CAPTURE_API_PORT=8000
```

Do not bind to:

```text
0.0.0.0
```

during SB-103 unless deliberately testing Docker networking later.

The API is internal. It should not be reachable from other computers on the local network during this ticket.

SB-104 may override the host inside Docker while still avoiding public host-port publication.

---

## 3. Authenticate every `/internal/*` route

The architecture explicitly requires a shared-secret header for state-changing routes.

SB-103 should go slightly further:

```text
GET /health
    → no token required

all /internal/* routes
    → token required
```

`GET /internal/captures/{capture_id}` returns accepted raw text so downstream classification can occur. That data should not be available merely because a caller can reach localhost or the future private Docker network.

Use one header:

```text
X-Second-Brain-Internal-Token
```

Use one environment variable:

```dotenv
CAPTURE_SERVICE_INTERNAL_TOKEN=
```

Generate a local value with:

```bash
openssl rand -hex 32
```

Never commit the real token.

---

## 4. Keep HTTP concerns outside CaptureService

The API layer owns:

```text
HTTP routing
request validation
token validation
HTTP status-code mapping
response serialization
```

`CaptureService` owns:

```text
capture retrieval
state transitions
transition validation
SQLite mutation
receipt delivery
receipt replacement fallback
safe error handling
```

The router must not import `Ledger`.

---

## 5. Separate state mutation from receipt editing

Future n8n orchestration will perform these as distinct steps:

```text
mark capture FILED
    ↓
edit Discord receipt
```

The canonical n8n flow reflects that ordering: update capture-service state, then edit the Discord receipt through capture-service.

Therefore:

```text
POST /mark-filed
```

must mutate state only.

```text
POST /internal/receipts/{capture_id}/edit
```

must deliver the requested receipt separately.

The local worker may continue using convenience methods such as:

```python
await capture_service.complete_filed(...)
```

Internally, the convenience method should call:

```text
mark_filed()
    ↓
edit_receipt()
```

This preserves existing local behavior while making the API usable by future orchestration.

## Expected state transitions

Add:

```python
FORWARDED = "FORWARDED"
```

to the capture-domain status constants introduced in SB-102.

Support this minimal transition table:

| Endpoint           | Allowed starting state                 | Resulting state |
| ------------------ | -------------------------------------- | --------------- |
| `mark-forwarded`   | `RECEIVED`                             | `FORWARDED`     |
| `mark-classifying` | `RECEIVED`, `FORWARDED`                | `CLASSIFYING`   |
| `mark-filed`       | `CLASSIFYING`                          | `FILED`         |
| `mark-inbox`       | `CLASSIFYING`                          | `INBOX`         |
| `mark-failed`      | `RECEIVED`, `FORWARDED`, `CLASSIFYING` | `FAILED`        |
| `retry`            | `FAILED`                               | `RECEIVED`      |

Why allow:

```text
RECEIVED → CLASSIFYING
```

directly?

The current local worker still classifies captures without forwarding them through n8n. Once n8n replaces the local worker, the normal remote path becomes:

```text
RECEIVED → FORWARDED → CLASSIFYING
```

Do not add lease timestamps or retry counters yet. Those belong to the later delivery-hardening ticket.

## Replay and idempotency rules

HTTP callbacks may be retried. The API must handle replays deliberately.

For each state transition:

```text
current state is valid source state
    → apply transition
    → append one transition event
    → return changed=true

current state already equals requested target state
    → do not append another event
    → return changed=false

current state conflicts with requested target state
    → return HTTP 409
```

For terminal routes with payloads:

```text
same target state and same stored payload
    → idempotent success
    → changed=false

same target state but conflicting note_path or classification payload
    → HTTP 409
```

Examples:

```text
CLASSIFYING → FILED with note_path A
FILED → FILED with note_path A
    → safe replay

FILED → FILED with note_path B
    → conflict
```

This protects against duplicated n8n callbacks later.

## HTTP status-code mapping

Use:

| Condition                                      | HTTP status |
| ---------------------------------------------- | ----------- |
| Healthy service                                | `200`       |
| Missing or incorrect token                     | `401`       |
| Unknown capture ID                             | `404`       |

## Token validation

Create a dependency closure inside:

```text
capture_api.py
```

Use constant-time token comparison.

Example shape:

```python
from secrets import compare_digest
from typing import Annotated

from fastapi import Header, HTTPException


INTERNAL_TOKEN_HEADER = "X-Second-Brain-Internal-Token"


def build_require_internal_token(expected_token: str):
    async def require_internal_token(
        supplied_token: Annotated[
            str | None,
            Header(alias=INTERNAL_TOKEN_HEADER),
        ] = None,
    ) -> None:
        if supplied_token is None or not compare_digest(
            supplied_token,
            expected_token,
        ):
            raise HTTPException(
                status_code=401,
                detail="unauthorized",
            )

    return require_internal_token
```

Do not log the supplied token.

## Programmatic Uvicorn server

Create:

```text
src/secondbrain/api_server.py
```

Use a wrapper such as:

```python
import uvicorn


class InternalApiServer:
    def __init__(self, app, *, host: str, port: int) -> None:
        config = uvicorn.Config(
            app,
            host=host,
            port=port,
            log_level="info",
            access_log=False,
        )
        self._server = uvicorn.Server(config)

    async def serve(self) -> None:
        await self._server.serve()

    async def stop(self) -> None:
        self._server.should_exit = True
```

Disable access logging unless there is a concrete need for it. Request paths contain capture IDs, and internal access logs add little value during the MVP.

Do not enable:

```text
reload=True
workers > 1
```

Multiple Uvicorn workers would create multiple process-level service instances and undermine the single-owner SQLite boundary.

## App startup integration

Update:

```text
src/secondbrain/app.py
```

Keep one runtime process.

Target sequence:

```python
async def run_service() -> None:
    settings = Settings()

    queue = CaptureQueue(
        maxsize=settings.classifier_queue_maxsize,
    )

    capture_service = CaptureService.open(
        settings,
        notify_capture=queue.enqueue,
    )

    api = create_capture_api(
        capture_service=capture_service,
        internal_token=settings.capture_service_internal_token,
    )

    api_server = InternalApiServer(
        api,
        host=settings.capture_api_host,
        port=settings.capture_api_port,
    )

    client = create_discord_client(
        capture_service.handle_gateway_message,
        on_ready_callback=start_background_worker_once,
    )

    capture_service.attach_receipt_client(client)

    api_task = asyncio.create_task(api_server.serve())
    discord_task = asyncio.create_task(
        client.start(settings.discord_bot_token)
    )

    try:
        await asyncio.gather(api_task, discord_task)
    finally:
        await api_server.stop()
        await client.close()
        capture_service.close()
```

The exact composition may vary, but preserve these rules:

```text
one CaptureService instance
one Ledger owner
one SQLite connection owner
one Discord client
one API server in the same process
```

If either the API server or Discord client exits unexpectedly, the process should shut down rather than continuing in a partially broken state.

## Acceptance criteria

SB-103 is complete when:

* FastAPI runs inside the existing capture-service process.
* The API binds to `127.0.0.1` by default.
* `/health` returns a minimal healthy response without authentication.
* Every `/internal/*` route requires the shared-secret header.
* `GET /internal/captures/{capture_id}` returns accepted raw text only to authenticated callers.
* Sensitive rejected captures never return raw plaintext.
* The HTTP layer does not import `Ledger`.
* The HTTP layer does not open SQLite.
* All state transitions go through `CaptureService`.
* Invalid transitions return `409`.
* Identical repeated callbacks are idempotent.
* Conflicting callback replays return `409`.
* Receipt editing occurs through `CaptureService`.
* Receipt-edit fallback sends at most one replacement receipt.
* The fake downstream integration test fetches, transitions, and edits a receipt without direct SQLite access.
* The local Discord-to-vault path remains functional.
* All SB-101 and SB-102 regressions still pass.
* No Docker, EC2, n8n, writer-service, leases, backoff, retry caps, or stale-lease reaper work has been added.

---

## Definition of done

A downstream HTTP client can retrieve a capture, transition it through processing states, and edit its Discord receipt through a private authenticated interface.

SQLite remains an implementation detail of `CaptureService`.

The next ticket can Dockerize and deploy the capture service to EC2 without redesigning its internal API.

---

One point to watch during implementation: do not let the API endpoints mutate status with direct repository calls merely because it seems simpler. The whole value of SB-102 and SB-103 is that the same CaptureService rules govern the local worker, future n8n callbacks, retries, and receipts.
