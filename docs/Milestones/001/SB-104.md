# SB-104 — Dockerize and deploy capture-service to EC2

**Branch:** `feature/ec2-capture-deployment`

## Goal

Deploy the durable Discord intake service to an always-on EC2 instance.

By the end of this ticket:

```text
desktop powered off
    ↓
message posted from phone to #brain-dump
    ↓
capture-service running on EC2 receives Discord Gateway event
    ↓
allowlist and secret screen run
    ↓
accepted raw capture committed to SQLite on persistent storage
    ↓
immediate Discord receipt confirms durable capture
```

Classification and Markdown filing remain disabled on EC2 during this ticket.

The result is intentionally incomplete but useful: thoughts are durably captured while the local desktop is offline.

---

## Precondition

Do not begin SB-104 until SB-102 and SB-103 are complete and approved.

SB-102 must already guarantee:

```text
CaptureService is the only production Ledger owner
worker.py does not mutate Ledger directly
app.py does not open Ledger directly
reconcile.py does not mutate Ledger directly
receipt coordination lives behind CaptureService
```

SB-103 must already guarantee:

```text
internal authenticated API wraps CaptureService
GET /health exists
API runs inside the capture-service process
internal API does not import Ledger
SQLite still has one production owner
```

SB-104 deploys the existing service boundary. It must not reintroduce direct SQLite access to work around an incomplete refactor.

---

## Why this ticket exists

The local MVP proves the capture workflow but does not provide continuous availability.

When the desktop is asleep, disconnected, rebooting, or powered off:

```text
Discord messages still exist
    ↓
but the local listener is not running
    ↓
capture is delayed until local startup reconciliation
```

SB-104 moves the durable intake boundary to EC2:

```text
Discord Gateway
    ↓
capture-service on EC2
    ↓
SQLite ledger on persistent EBS-backed storage
```

The EC2 service must remain small and focused. It captures safely first. Downstream orchestration comes later.

---

## Architecture rules

These rules must remain true:

```text
persist before downstream processing
reject secret-like plaintext before persistence
Discord message ID is the idempotency key
capture-service is the only SQLite owner
internal API is not publicly exposed
container retains outbound internet access
desktop listener and EC2 listener must not run simultaneously
```

The final rule matters during migration. Running both listeners against the same Discord channel would produce duplicate receipts and separate local and EC2 ledgers.

## Persistent storage design

SQLite must not live only inside the container filesystem.

Use:

```text
EC2 encrypted EBS volume
    ↓
mounted host directory
    ↓
bind-mounted into capture-service container
    ↓
SQLite database file
```

Recommended paths:

```text
EC2 host:
    /opt/second-brain/data

container:
    /var/lib/second-brain

SQLite:
    /var/lib/second-brain/ledger.sqlite3
```

Environment value:

```dotenv
LEDGER_PATH=/var/lib/second-brain/ledger.sqlite3
```

Compose mount:

```yaml
volumes:
  - /opt/second-brain/data:/var/lib/second-brain
```

Do not store SQLite at:

```text
/app/ledger.sqlite3
/tmp/ledger.sqlite3
container writable layer only
```

A Docker named volume would survive container recreation, but an explicit EBS-backed host path is preferred because it is easier to inspect, snapshot, restore, and move to another instance.

## Internal networking design

The internal API must remain available to the container health check and future Compose services.

It must not be published to the EC2 host.

Inside the container:

```dotenv
CAPTURE_API_HOST=0.0.0.0
CAPTURE_API_PORT=8000
```

In Compose:

```yaml
expose:
  - "8000"
```

Do not add:

```yaml
ports:
  - "8000:8000"
```

Target network shape:

```text
public internet
    ──X──> capture-service API

EC2 host network
    ──X──> capture-service API host port

future n8n container
    ─────> http://capture-service:8000

capture-service container
    ─────> Discord Gateway through outbound internet
```

Use a normal Compose bridge network. Do not set:

```yaml
internal: true
```

The container requires outbound internet access for Discord.

# Acceptance criteria

SB-104 is complete when:

- `capture-only` runtime mode exists.
- `local-full` runtime mode still works.
- Capture-only mode does not require Gemini configuration.
- Capture-only mode does not require a vault path.
- Capture-only mode does not start the classifier worker.
- Capture-only mode does not construct `VaultWriter`.
- Capture-only receipts accurately state that downstream filing is disabled.
- Discord allowlists still apply.
- Secret screening still happens before plaintext persistence.
- SQLite lives on an EBS-backed bind-mounted host path.
- SQLite survives container recreation.
- SQLite survives EC2 reboot.
- Startup reconciliation recovers a message posted while the container is stopped.
- Duplicate reconciliation does not create duplicate rows or receipts.
- The internal API starts inside the capture-service process.
- `GET /health` works inside the container.
- Docker reports the container as healthy.
- Compose uses `restart: unless-stopped`.
- The container runs as a non-root user.
- The container drops unnecessary Linux capabilities.
- The internal API port is not published to the EC2 host.
- No public security-group rule exposes port `8000`.
- SSH ingress is restricted to an intentional source.
- The EC2 instance requires IMDSv2.
- The desktop listener is stopped while EC2 owns Discord intake.
- A phone message receives a durable-capture receipt while the desktop is offline.
- All SB-101, SB-102, and SB-103 regression tests continue to pass.
- No n8n, writer-service, Git sync, retry lease, backoff, or backup automation work has been added.
