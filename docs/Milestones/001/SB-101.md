# SB-101 — Add regression tests for the proven local slice

**Branch:** `feature/mvp-regression-suite`

## Goal

Create an automated regression suite around the working local MVP before extracting `capture-service`.

The suite must prove that the existing capture-to-vault loop still behaves correctly after future refactors. Tests must run locally without connecting to Discord or Gemini.

This ticket freezes current behavior. It does **not** redesign the architecture.

---

## Why this comes first

The next tickets will move Discord intake, SQLite ownership, receipts, and reconciliation behind a `capture-service` boundary. Without regression coverage, a refactor could silently break durability or idempotency while still appearing to work during a basic manual test.

The most important behavior to preserve is:

```text
Discord-like message received
    ↓
filters and secret screen run
    ↓
accepted raw capture committed to SQLite
    ↓
immediate saved receipt sent
    ↓
background classification runs
    ↓
Markdown note written or routed to Inbox
    ↓
original receipt edited into final state
```

The tests should exercise the same application code paths used by the runtime, but replace external services with deterministic fakes.

---

## Scope

### In scope

* Add test fixtures and fake clients.
* Add regression tests for the proven MVP behavior.
* Add minimal dependency-injection seams only where required for testing.
* Verify SQLite rows, Markdown files, audit events, and Discord receipt calls.
* Verify startup reconciliation without connecting to Discord.
* Verify asynchronous background processing completes deterministically in tests.

### Out of scope

* Extracting `capture-service`.
* Adding an HTTP API.
* Deploying to EC2.
* Adding Docker.
* Adding n8n.
* Adding delivery leases or the stale-lease reaper.
* Changing the SQLite schema unless a small test-support change is unavoidable.
* Rewriting working production logic merely to make the code look cleaner.

---

## Test strategy

Use three layers:

```text
unit tests
    → validate isolated rules

integration tests with fake clients
    → validate the local capture-to-vault flow

manual smoke test
    → verify one real Discord message still works after the suite passes
```

Most of SB-101 is the integration layer. The purpose is to lock down the behavior that the later `capture-service` extraction must preserve.

## Implementation plan

## Step 1 — Create isolated temporary runtime fixtures

Add shared fixtures in:

```text
tests/conftest.py
```

Each test must run against its own temporary filesystem and SQLite database.

Use pytest's `tmp_path` fixture to create:

```text
<tmp_path>/
├── runtime/
│   └── ledger.sqlite3
└── vault/
```

Create a settings fixture that points the application to those temporary paths.

Example fixture responsibilities:

```text
test_settings
    → temporary LEDGER_PATH
    → temporary VAULT_PATH
    → fixed allowed guild ID
    → fixed allowed channel ID
    → fixed allowed user ID
    → fixed Gemini model string
    → fixed confidence threshold
```

No test may write to the real `.runtime/ledger.sqlite3` or the real Obsidian vault.

### Required result

Running the suite repeatedly leaves no files behind outside pytest's temporary directory.

## Step 4 — Add controllable vault-write failure behavior

Prefer using the real deterministic Markdown writer for successful tests. This validates the generated Markdown file rather than merely checking that a method was called.

Add either:

```text
FakeFailingVaultWriter
```

or an injectable failure mode in a test wrapper around the real writer.

The failure fake should raise a predictable exception such as:

```text
OSError("simulated vault write failure")
```

Do not add a production-only failure flag merely for tests if dependency injection can solve the problem.

### Required result

The suite can prove that a vault-write failure does not delete or corrupt the committed raw capture.

---

## Step 5 — Add a deterministic worker-drain helper

The production application classifies captures in a background worker. Tests must wait for processing without using arbitrary sleep calls.

Add a helper such as:

```text
await drain_worker(app)
```

It should:

```text
wait until queued background work completes
fail with a short timeout if processing hangs
surface worker exceptions to the test
```

Prefer:

```python
await queue.join()
```

with a timeout rather than:

```python
await asyncio.sleep(1)
```

Sleep-based tests become slow and flaky.

### Required result

Integration tests complete consistently and fail quickly when the worker stalls.

## Step 6 — Add the core regression tests

## Test 1 — Normal message creates exactly one SQLite capture and one Markdown note

**File:**

```text
tests/integration/test_capture_pipeline.py
```

**Test name:**

```text
test_normal_message_creates_one_capture_and_one_note
```

### Arrange

* Create an allowlisted fake Discord message.
* Use the default successful fake classifier.
* Use a temporary SQLite ledger.
* Use a temporary vault path.
* Start the background worker.

### Act

* Pass the fake message through the same ingestion method used for live Discord messages.
* Wait for the worker queue to drain.

### Assert

* Exactly one `captures` row exists.
* The row contains the original accepted `raw_text`.
* The row reaches `FILED`.
* Exactly one Markdown file exists under the expected vault folder.
* The Markdown frontmatter contains the generated `capture_id`.
* The source Discord message ID appears in the frontmatter.
* Exactly one audit event was appended.
* One immediate receipt was sent after commit.
* The original receipt was edited into the filed state.

---

## Test 2 — Duplicate Discord event does not create a duplicate

**Test name:**

```text
test_duplicate_discord_event_is_idempotent
```

### Arrange

* Create one fake Discord message with a fixed Discord message ID.

### Act

* Submit the same message twice.
* Wait for background processing.

### Assert

* Exactly one SQLite capture row exists for the Discord message ID.
* Exactly one derived Markdown note exists.
* The classifier is not allowed to create a second note.
* The audit log does not contain a second `FILED` event for the same capture.
* Routine duplicate receipt spam is not produced.

The Discord message ID is the real idempotency key. The human-readable `capture_id` is not a substitute.

---

## Test 3 — Bot receipt is ignored

**Test name:**

```text
test_bot_authored_receipt_is_ignored
```

### Arrange

* Create a fake message authored by the bot.
* Use content resembling a receipt:

```text
✅ SB-20260607-0001 filed.
Location: 20_projects / halo
```

### Act

* Pass the fake bot message into the Discord ingestion method.

### Assert

* No capture row is created.
* No classifier call occurs.
* No Markdown note is created.
* No new receipt is sent.

This prevents an ingestion loop in which the bot captures its own receipts.

## Test 4 — Secret-like input stores only a redacted rejection

**Test name:**

```text
test_secret_like_input_stores_redacted_rejection_only
```

### Arrange

* Create an allowlisted fake Discord message containing a known test-only secret pattern:

```text
api_key=TEST_ONLY_DO_NOT_USE_123456
```

Never place a real credential in the test suite.

### Act

* Submit the message through the normal ingestion path.

### Assert

* One rejection row exists.
* `status == REJECTED_SENSITIVE`.
* `is_sensitive == 1`.
* `raw_text IS NULL`.
* `redacted_text` exists.
* The original test secret does not appear in SQLite.
* The original test secret does not appear in logs.
* The classifier call count remains zero.
* No Markdown note is created.
* A rejection receipt is sent.

Use `caplog` or equivalent log capture to verify the plaintext does not leak into logs.

## Minimal dependency-injection rule

If the current code directly instantiates Discord, Gemini, or filesystem dependencies internally, add the smallest seam necessary to replace them in tests.

Preferred shape:

```python
app = SecondBrainApp(
    settings=test_settings,
    discord_client=fake_discord,
    classifier=fake_classifier,
    vault_writer=test_writer,
)
```

Production startup can still construct the real dependencies:

```python
app = SecondBrainApp.from_settings(settings)
```

Do not begin the `capture-service` extraction in this ticket. The goal is testability, not a structural rewrite.

## Acceptance criteria

SB-101 is complete when:

* All eight required regression tests pass locally.
* Tests do not call the real Discord API.
* Tests do not call the real Gemini API.
* Tests do not write to the real vault.
* Tests do not modify the real runtime SQLite ledger.
* Background-worker tests do not rely on arbitrary sleep delays.
* Secret-like plaintext is absent from SQLite and captured logs.
* Duplicate Discord events cannot create duplicate rows or notes.
* Restart reconciliation recovers a missed message exactly once.
* Receipt-edit failure creates exactly one replacement receipt.
* One manual real-Discord smoke test still passes.
* No EC2, HTTP API, n8n, or service-extraction work has been added to the branch.

---

## Definition of done

The current local MVP behavior is protected by a repeatable automated regression suite. A later refactor can move intake logic behind `capture-service` while immediately exposing any change in durability, idempotency, recovery, or receipt behavior.
