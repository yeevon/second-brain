# SB-102 — Introduce a capture-service boundary inside the existing repo

**Branch:** `feature/capture-service-boundary`

## Goal

Refactor the working local monolith so all durable intake responsibilities sit behind a single in-process `CaptureService` interface.

The local application must continue behaving exactly as it did before the refactor:

```text
Discord message
    ↓
allowlist and secret screening
    ↓
durable SQLite commit
    ↓
immediate Discord receipt
    ↓
background classification
    ↓
deterministic Markdown write or Inbox fallback
    ↓
final receipt edit
```

The change is architectural, not user-facing.

By the end of SB-102:

* Discord intake reaches the ledger only through `CaptureService`.
* Reconciliation reaches the ledger only through `CaptureService`.
* The background worker reads and mutates capture state only through `CaptureService`.
* Receipt delivery and editing are coordinated through `CaptureService`.
* CLI status queries go through `CaptureService`.
* `app.py`, `worker.py`, `discord_capture.py`, and `reconcile.py` no longer open, mutate, or query SQLite directly.
* The application remains one local Python process.

---

## Why this ticket exists

The current MVP works, but its responsibilities are distributed across orchestration modules:

```text
app.py
    → opens Ledger
    → screens secrets
    → inserts rows
    → stores receipt IDs
    → advances reconciliation markers

worker.py
    → mutates Ledger
    → edits final receipts
    → records FILED, INBOX, and FAILED transitions

reconcile.py
    → reads and writes SQLite system state directly
```

That is acceptable for proving the local MVP. It is not safe to wrap with an HTTP API yet.

SB-102 introduces the boundary that SB-103 will expose over HTTP and SB-104 will deploy to EC2.

---

## Architecture rule

Only the capture boundary may own or mutate the SQLite ledger.

```text
Allowed:
    CaptureService
        ↓
    Ledger

Forbidden:
    Worker       ──X──> Ledger
    app.py       ──X──> Ledger
    Discord      ──X──> Ledger
    reconcile.py ──X──> Ledger
    future n8n   ──X──> SQLite file
    future writer-service ──X──> SQLite file
```

Unit tests for `Ledger` may continue to instantiate it directly. Production modules outside the capture boundary may not.

## Temporary downstream notifier

The current local worker queue is the temporary replacement for the future private n8n webhook.

`CaptureService` should not import `CaptureQueue` directly. Inject a small asynchronous callback:

```python
notify_capture: Callable[[str], Awaitable[None]] | None
```

Local composition:

```python
queue = CaptureQueue(maxsize=settings.classifier_queue_maxsize)

capture_service = CaptureService.open(
    settings,
    notify_capture=queue.enqueue,
)
```

After an accepted capture is durably persisted and the immediate receipt attempt completes:

```python
await self._notify_capture(capture.capture_id)
```

Later, the callback can become a private webhook forwarder without rewriting ingestion.

For recovered unfinished rows:

```python
await capture_service.enqueue_unfinished_captures()
```

This preserves the current startup behavior while keeping queue knowledge outside the durable boundary.

## Ordering invariants

Preserve these invariants exactly.

## Accepted message ordering

```text
run filter
    ↓
run secret screen
    ↓
commit accepted raw capture to SQLite
    ↓
attempt immediate saved receipt
    ↓
persist receipt_message_id if available
    ↓
notify downstream queue
```

Never send:

```text
Your note is saved.
```

before the SQLite commit succeeds.

## Sensitive message ordering

```text
run filter
    ↓
run secret screen
    ↓
commit redacted rejection record only
    ↓
attempt rejection receipt
    ↓
persist receipt_message_id if available
    ↓
stop
```

Never enqueue or classify a rejected sensitive message.

## Filed completion ordering

```text
VaultWriter successfully writes Markdown
    ↓
CaptureService commits FILED state and note path to SQLite
    ↓
CaptureService attempts final receipt edit
    ↓
if edit fails:
    send one replacement receipt
    persist replacement receipt ID
```

Do not hold an SQLite transaction open while calling Discord.

## Inbox completion ordering

```text
VaultWriter writes Inbox note
    ↓
CaptureService commits INBOX state and reason
    ↓
CaptureService attempts final receipt edit
```

## Failed completion ordering

```text
worker detects vault failure or unexpected processing error
    ↓
CaptureService commits FAILED state and safe error summary
    ↓
CaptureService attempts visible Discord failure receipt
```

Example approach:

```python
from pathlib import Path
import ast

FORBIDDEN_LEDGER_IMPORTERS = {
    "app.py",
    "worker.py",
    "discord_capture.py",
    "reconcile.py",
}

def imported_modules(path: Path) -> set[str]:
    tree = ast.parse(path.read_text(encoding="utf-8"))
    modules = set()

    for node in ast.walk(tree):
        if isinstance(node, ast.Import):
            modules.update(alias.name for alias in node.names)
        elif isinstance(node, ast.ImportFrom) and node.module:
            modules.add(node.module)

    return modules

def test_non_boundary_modules_do_not_import_ledger():
    src = Path("src/secondbrain")

    for filename in FORBIDDEN_LEDGER_IMPORTERS:
        modules = imported_modules(src / filename)
        assert "secondbrain.ledger" not in modules
```

Do not ban direct ledger use in:


That test intentionally validates the repository.
