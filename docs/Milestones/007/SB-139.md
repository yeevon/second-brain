# SB-139 — V3 proposal-only MCP tools

**Branch:** `feature/v3-mcp-proposal-tools`

---

## Goal

Add a second MCP profile with proposal tools that LLM clients can use to suggest vault changes. These tools call the proposal API in `capture-service` — they do not write vault files directly. The read-only MCP profile remains unchanged and remains the default.

---

## Two MCP profiles

### Profile 1 (unchanged): `brain-mcp` — read-only

Existing tools: `search_notes`, `read_note`, `list_recent_notes`, `list_open_tasks`, `get_sync_status`.

This profile does not change.

### Profile 2 (new): `brain-mcp-propose` — proposal-only

New tools listed below. This profile requires the user to opt in explicitly — it is not the default MCP server.

---

## New proposal tools

### `propose_task_completion`

```json
{
  "name": "propose_task_completion",
  "description": "Propose marking a task as done. Requires explicit user confirmation that the task is complete — never infer completion from vague prose.",
  "inputSchema": {
    "note_path": "string (relative vault path)",
    "task_text": "string (exact task text as it appears in the note)",
    "completion_note": "string (optional — brief note on how it was completed)",
    "reason": "string (why this task is believed to be complete)"
  }
}
```

### `propose_due_date_change`

Propose setting or changing the `due_date` frontmatter field on a note.

### `propose_priority_change`

Propose setting or changing the `priority` frontmatter field on a note.

### `propose_note_move`

Propose moving a note to a different allowed vault folder. The target folder must be in the allowlist.

### `propose_task_append`

Propose appending a new open task (`- [ ] <text>`) to the `## Actions` section of a note.

### `propose_review_entry`

Propose appending a weekly review entry to the designated digest note.

### `list_pending_update_proposals`

Return proposals in PENDING status. The LLM may use this to show the user what is waiting for approval.

### `read_update_proposal`

Return the full detail of a single proposal by ID, including operation, target, change, and current status.

---

## Tool implementation rules

All proposal tools must:

- Call `POST /internal/vault/proposals` on `capture-service` — never write files directly.
- Pass the exact `note_path` as provided — do not resolve or rewrite paths inside the tool.
- Return the `proposal_id` and `status` to the LLM client on success.
- Return a clear error if the proposal was rejected at creation (unsupported operation, path escape, etc.).
- Never call the apply endpoint — proposals are applied only after user approval through the Discord surface.

The following tools are explicitly disallowed in this profile:

```text
write_note
delete_note
replace_note
move_note_directly
git_commit
git_push
shell
```

---

## MCP server configuration

The existing `brain-mcp` server is started with the vault-only read-only tool set.

The new `brain-mcp-propose` server is a separate entry point or configuration flag that adds the proposal tools.

The `gembrain` CLI is updated:

```bash
gembrain ask "..."       # uses brain-mcp (read-only, default)
gembrain propose "..."   # uses brain-mcp-propose (proposal mode, opt-in)
```

---

## Acceptance criteria

- `brain-mcp-propose` is a distinct profile from `brain-mcp`.
- All proposal tools call the proposal API — none write vault files or call the apply endpoint.
- `propose_task_completion` rejects an empty `task_text` or empty `note_path`.
- `propose_note_move` rejects a target folder not in the vault folder allowlist.
- `list_pending_update_proposals` returns only PENDING proposals.
- `read_update_proposal` returns 404 for an unknown ID.
- The disallowed tools (`write_note`, `shell`, etc.) are not present in the proposal profile.
- Tests cover: each proposal tool submits to the API, disallowed tools absent, invalid inputs rejected.

**Done when:** the proposal MCP profile is functional and all acceptance criteria pass.
