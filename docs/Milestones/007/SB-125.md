# SB-125 — Merge V2 staging to main and review production configuration

**Branch:** `release/v2-production`

---

## Goal

Merge the `staging` branch to `main` and verify the compose configuration is safe to deploy on EC2.

The `staging` branch was declared a V2 release candidate in the commit "v2 ready for update to main line but not ready for prod rollout". The blockers before prod are security hardening, private n8n access, and backup scheduling — those are separate tickets. This ticket handles the merge and config review.

---

## Checklist

### Compose configuration review

- [ ] `compose.yaml` uses a bridge network with no published host ports for `capture-service` and `writer-service`.
- [ ] `n8n` port `5678` is not published to `0.0.0.0` in any compose file committed to the repository.
- [ ] `compose.override.yaml` is listed in `.gitignore` and is not committed with production secrets.
- [ ] `.env.example` documents all required variables without containing real values.
- [ ] No plaintext secrets appear in any committed compose or env file.
- [ ] All service images either have pinned versions or build from the local Dockerfile.

### Branch merge

- [ ] Open a pull request from `staging` → `main`.
- [ ] Confirm CI passes (linting, tests).
- [ ] Merge with merge commit — do not squash; the milestone history is meaningful.
- [ ] Tag the merge commit: `v2.0.0`.

---

## Acceptance criteria

- `main` contains the full V2 feature set.
- No secrets are present in the merged commit.
- Internal service ports are not published to the EC2 host in the committed compose configuration.
- The `v2.0.0` tag is present on the merge commit.

**Done when:** the PR is merged, the tag is created, and the configuration review above shows no violations.
