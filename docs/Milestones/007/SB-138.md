# SB-138 — Discord approval surface

**Branch:** `feature/v3-discord-approval`

---

## Goal

When a vault-update proposal is submitted, post an approval request to Discord showing the target file, operation, and summary of the intended change. The user approves or rejects by reply. At least one approval surface must exist before any V3 write MCP tools are enabled (architecture invariant).

---

## Approval request message

Sent to `#brain-dump` or `#brain-digest` when a proposal is created:

```text
📋 VUP-20260616-0001 — vault update proposal

Operation:  mark_task_done
File:       20_projects/second-brain/example.md
Task:       "Expose second brain to local MCP server"
Reason:     User stated this task is complete.

Reply with:
  approve VUP-20260616-0001  — to apply this change
  reject VUP-20260616-0001   — to discard without applying
```

The message must show enough context for the user to make an informed decision without opening the file.

---

## Reply routing in capture-service

`capture-service` must recognize two new reply patterns:

```text
approve VUP-YYYYMMDD-NNNN
reject VUP-YYYYMMDD-NNNN
```

These patterns are checked before the `fix:` prefix check. They are not treated as new captures.

### Approve flow

1. Validate the proposal ID exists and is in PENDING status.
2. Transition proposal to APPROVED in `vault_update_proposals`.
3. Call `POST /internal/vault/apply-proposal` on `writer-service`.
4. On success: transition to APPLIED, edit the approval-request message to show ✅ Applied + commit hash.
5. On failure: transition to FAILED, edit the message to show ❌ Apply failed + error type.

### Reject flow

1. Validate the proposal ID exists and is in PENDING or APPROVED status.
2. Transition proposal to REJECTED with `rejected_reason` set to "User rejected via Discord".
3. Append a `VAULT_UPDATE_REJECTED` event.
4. Edit the approval-request message to show ❌ Rejected.
5. Do not apply any vault change.

---

## Approval message storage

Store the Discord approval-request message ID in `vault_update_proposals`:

```sql
ALTER TABLE vault_update_proposals ADD COLUMN approval_message_id TEXT;
```

This allows the approval-request message to be edited after apply or reject.

---

## What the approval message must show

For each operation type, the approval message summary must include:

| Operation | Summary fields |
|-----------|---------------|
| `mark_task_done` / `mark_task_open` | task text, file, current status → new status |
| `set_task_due_date` | file, current due date (or none), new due date |
| `set_task_priority` | file, current priority (or none), new priority |
| `append_task` | file, new task text |
| `append_note_section` | file, new section heading |
| `move_note_to_folder` | current folder, new folder, file name |
| `add_project_tag` | file, new tag |
| `add_weekly_review_entry` | review period, entry summary |

---

## Acceptance criteria

- A submitted proposal triggers an approval request message in Discord.
- `approve VUP-YYYYMMDD-NNNN` in a non-reply message is recognized and applies the proposal.
- `reject VUP-YYYYMMDD-NNNN` transitions the proposal to REJECTED without touching the vault.
- Approving a REJECTED or APPLIED proposal returns a visible error ("this proposal is already closed").
- The approval-request Discord message is edited to show the final outcome (applied or rejected).
- A failed apply edits the message to show the failure and leaves the proposal in FAILED status.
- These reply patterns do not create new captures.
- Tests cover: approve success, approve with stale anchor, reject, approve already-closed proposal, invalid proposal ID.

**Done when:** all acceptance criteria pass and the approval surface is wired to the apply endpoint.
