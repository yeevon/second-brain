# SB-113 — Add the n8n error workflow

**Branch:** `feature/n8n-error-workflow`

Replace the placeholder n8n Error Trigger workflow with a real last-resort downstream recovery workflow. The error workflow must preserve the durable capture boundary:

```text
capture-service SQLite ledger = source of truth
```

n8n coordinates error recovery but does not own durability.

---

## Goal

Implement:

```text
Second Brain - Intake workflow fails unexpectedly
    ↓
n8n Error Trigger runs
    ↓
extract safe workflow metadata
    ↓
recover capture_id and delivery_attempt when available
    ↓
report safe failure metadata to capture-service
    ↓
capture-service schedules retry or marks terminal failure
    ↓
capture-service edits the existing Discord receipt
    ↓
raw capture remains preserved in SQLite
```

The user must never experience:

```text
Discord capture saved
    ↓
workflow fails silently
    ↓
no visible receipt update
    ↓
note appears to disappear
```

---

## Architecture rule: Error Trigger is a fallback, not the retry engine

Keep the explicit error branches from SB-112. Expected recoverable failures must still call capture-service directly through `POST /internal/captures/:capture_id/delivery/schedule-retry`:

```text
Gemini timeout
Gemini 429
Gemini 5xx
writer-stub timeout
writer-stub 5xx
classification validation failure
capture-service temporary 503
```

The Error Trigger workflow exists for:

```text
unexpected node exception
unhandled workflow exception
deliberately forced test failure
future workflow defect
unexpected response shape that escapes explicit routing
```

Do not route every expected retry through the Error Trigger. Do not create two competing retry owners.

---

## Failure ownership rule

For any single failure: either the explicit SB-112 error branch handles it, or the SB-113 Error Trigger workflow handles it. Never both. If an explicit SB-112 branch successfully calls `schedule-retry`, `acknowledge-failed`, or `acknowledge-inbox`, the branch must terminate successfully. Do not throw another error afterward — the Error Trigger may attempt to schedule the same failure twice.

---

## Preserve the validated durable boundary

Keep `capture-service` as the sole owner of:

```text
raw capture text
SQLite ledger
delivery attempts
retry count
processing lease
next retry timestamp
terminal state
Discord receipt delivery
```

The Error Trigger workflow must never open `ledger.sqlite3`, mount capture-service storage, call Discord directly, write a vault note, write a Git repository, call Gemini, submit to writer stub, store raw capture text, store stack traces, or store arbitrary exception messages.

---

## Correlation problem

The n8n Error Trigger receives workflow and execution information. It does not automatically guarantee `capture_id` or `delivery_attempt`. The intake workflow must provide those values intentionally for known failure paths.

Use two recovery modes:

**Mode A — correlated workflow error**: `capture_id` and `delivery_attempt` present → report retryable or terminal failure to capture-service.

**Mode B — orphan workflow error**: correlation missing or invalid → record safe n8n metadata only, do not guess capture ID, stale-lease reaper eventually recovers the durable capture.

## 1. Add a safe downstream workflow-error API route

Update `src/secondbrain/api_models.py`, `src/secondbrain/capture_api.py`, `src/secondbrain/capture_service.py`, and `src/secondbrain/ledger.py`. Add:

```text
POST /internal/captures/:capture_id/delivery/report-workflow-error
```

Require the existing internal auth header `X-Second-Brain-Internal-Token`.

### Request contract

```json
{
  "delivery_attempt": 1,
  "disposition": "retryable",
  "error_type": "gemini_timeout",
  "reason_type": "workflow_error",
  "workflow_id": "abc123",
  "workflow_name": "second_brain_intake",
  "execution_id": "456",
  "stage": "gemini"
}
```

Allowed dispositions: `retryable`, `terminal`.

### Response contracts

Retryable failure:

```json
{ "capture_id": "SB-YYYYMMDD-NNNN", "delivery_attempt": 1, "delivery_status": "RETRY_WAIT", "retry_attempts": 1, "outcome": "retry_scheduled" }
```

Exhausted retries:

```json
{ "capture_id": "SB-YYYYMMDD-NNNN", "delivery_attempt": 3, "delivery_status": "FAILED", "retry_attempts": 3, "outcome": "terminal_failure" }
```

Stale callback:

```json
{ "capture_id": "SB-YYYYMMDD-NNNN", "delivery_attempt": 1, "delivery_status": "CLASSIFYING", "retry_attempts": 0, "outcome": "ignored_stale_attempt" }
```

Already completed:

```json
{ "capture_id": "SB-YYYYMMDD-NNNN", "delivery_attempt": 1, "delivery_status": "COMPLETE", "retry_attempts": 0, "outcome": "ignored_already_terminal" }
```

## 2. Restrict workflow-error metadata to safe values

Validate `error_type`, `reason_type`, `workflow_id`, `execution_id`, and `stage` against the pattern `^[A-Za-z0-9_.:-]{1,100}$`.

Store a normalized workflow name slug only:

```text

```

Do not store arbitrary workflow display text.

Reject request bodies containing any of:

```text
raw_text  redacted_text  stack  traceback  exception_message
headers   credentials    token  api_key    password
classification  writer_payload  discord_message
```

Configure Pydantic with `ConfigDict(extra="forbid")`. Do not persist n8n's raw `execution.error.message` or `execution.error.stack` — those values may accidentally contain capture text or sensitive data.

## 5. Add a dedicated transactional ledger operation for workflow-error reporting

Add a new method `report_workflow_error()` to `src/secondbrain/ledger.py`. Do not implement this as a thin wrapper around existing methods with exception mapping outside the transaction. The replay-safety contract from section 6 requires atomically checking current state, detecting duplicate reports, and writing the transition — all within a single SQLite transaction.

The method signature:

```python
def report_workflow_error(
    self,
    *,
    capture_id: str,
    delivery_attempt: int,
    disposition: str,
    error_type: str,
    workflow_id: str,
    workflow_name: str,
    execution_id: str | None,
    stage: str,
) -> WorkflowErrorOutcome:
    ...
```

Within a single transaction the method must:

1. Fetch the current capture row with a lock.
2. Return `ignored_stale_attempt` if `request.delivery_attempt < current delivery_attempt`.
3. Return `ignored_already_terminal` if `delivery_status` is `COMPLETE` or `FAILED`.
4. Return `ignored_retry_already_scheduled` if `delivery_status` is already `RETRY_WAIT` for the same attempt.
5. Return `ignored_conflicting_replay` if a prior mutation already applied a different disposition to this attempt.
6. Append `N8N_WORKFLOW_ERROR_REPORTED` to the capture-event log.
7. For `retryable`: apply bounded retry logic — increment `retry_attempts`, calculate backoff, populate `next_attempt_at`, clear the processing lease, move to `RETRY_WAIT`, or mark `FAILED` if retry limit is exhausted.
8. For `terminal`: mark `FAILED`, clear the processing lease, preserve `raw_text`.
9. Commit. Return a typed `WorkflowErrorOutcome` containing `delivery_status`, `retry_attempts`, and the `outcome` slug.

The existing `schedule_delivery_retry()` and `acknowledge_delivery_failed()` behaviors remain unchanged. `report_workflow_error()` shares their transition logic but wraps it in a single atomic transaction with the additional idempotency checks.

## 6. Make error reporting idempotent

The Error Trigger workflow may itself retry or be delivered twice. The route must tolerate replay:

| Condition | Outcome |
| --- | --- |
| `request.delivery_attempt < current delivery_attempt` | `ignored_stale_attempt` — do not mutate |
| Capture already `COMPLETE` or `FAILED` | `ignored_already_terminal` — do not mutate |
| Same attempt already moved to `RETRY_WAIT` | `ignored_retry_already_scheduled` — do not increment again |
| Replay attempts to change `retryable` → `terminal` after mutation | `ignored_conflicting_replay` — do not corrupt state |

These outcomes are enforced transactionally inside `report_workflow_error()` in `src/secondbrain/ledger.py`. Do not implement idempotency through HTTP-layer exception catching around non-transactional calls. A race between two concurrent error reports for the same capture must not result in two retry events, two `N8N_WORKFLOW_ERROR_REPORTED` audit entries, or a double retry-count increment.


The n8n Error Trigger exposes two payload shapes depending on how the error occurred:

Treat `execution_id` as optional in all cases. When it is absent or unavailable, set `execution_id = null` in the normalized output. Do not fail or set `correlated = false` solely because `execution_id` is missing — correlation depends on `capture_id` and `delivery_attempt`, not on execution ID.

Do not forward arbitrary node names. Do not forward arbitrary error messages. Do not forward stack traces.

## 12. Handle orphan workflow errors safely

When correlation is missing:

```text
do not mutate capture-service state
do not send a Discord receipt update
do not guess a capture ID
do not fetch raw captures
do not retry arbitrary capture IDs
```

Record only safe metadata through n8n logs: `workflow_id`, `workflow_name`, `execution_id` if present, `stage`, `timestamp`, `correlated = false`. Then end with No Operation.

The capture-service stale-lease reaper remains the safety net. Orphan recovery is eventual, not immediate. With default configuration, the worst-case recovery delay is approximately the processing-lease duration plus the reaper interval — around 300 seconds plus up to 30 seconds. Do not describe orphan recovery as a fast fallback in documentation or receipts.

```text
FORWARDED or CLASSIFYING lease expires
    ↓ (worst case: ~330 seconds at default settings)
capture-service reaper detects stale lease
    ↓
capture-service schedules retry
    ↓
capture-service edits visible retry receipt
    ↓
raw capture remains preserved
```

Acceptance criteria that mention a "visible receipt update" for orphan failures must tolerate this delay in test assertions.
