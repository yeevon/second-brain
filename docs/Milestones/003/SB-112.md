# SB-112 — Add the n8n intake workflow with a stub writer

**Branch:** `feature/n8n-intake-workflow`

Build the first real downstream-processing workflow behind the durable `capture-service` boundary. The workflow must process captures without giving n8n direct access to the capture ledger, Discord Gateway intake, or the Obsidian vault.

---

## Goal

Implement:

```text
Discord message
    ↓
capture-service persists raw capture
    ↓
capture-service sends immediate durable receipt
    ↓
capture-service delivery dispatcher posts minimal webhook envelope
    ↓
n8n fetches capture through authenticated internal API
    ↓
defense-in-depth secret scan
    ↓
mark CLASSIFYING
    ↓
call Gemini with structured output schema
    ↓
validate normalized classification
    ↓
confidence gate
    ↓
submit normalized filing request to HTTP stub writer
    ↓
mark FILED or INBOX through capture-service API
    ↓
capture-service edits Discord receipt
```

The key offline behavior:

```text
n8n stopped
    ↓
Discord capture still persists immediately
    ↓
saved receipt still appears immediately
    ↓
capture remains queued in SQLite
    ↓
n8n starts later
    ↓
capture-service dispatcher retries webhook delivery
    ↓
queued capture processes with at-least-once delivery and produces one observable terminal result
```

---

## Architecture correction: do not rely on webhook availability

A webhook by itself cannot meet the offline-processing requirement. If n8n is stopped, `POST http://n8n:5678/webhook/second-brain-intake` fails with a connection error. The durable queue must remain owned by the capture-service SQLite ledger.

Use the existing delivery state machine:

```text
PENDING_FORWARD → FORWARDING → FORWARDED → CLASSIFYING → COMPLETE
```

Retry path:

```text
FORWARDING / FORWARDED / CLASSIFYING
    ↓
webhook failure, workflow failure, or expired lease
    ↓
RETRY_WAIT
    ↓
next_attempt_at reached
    ↓
new delivery_attempt
```

Do not create a second queue inside n8n. Do not use n8n execution storage as the source of truth.

---

## Delivery guarantee

Do not claim mathematically strict exactly-once execution across HTTP and process-crash boundaries. Implement:

```text
at-least-once webhook delivery
+ attempt-aware idempotent callbacks
+ idempotent stub-writer behavior
= one observable filing result
```

Duplicate webhook deliveries are expected and must be harmless. A repeated delivery with the same `capture_id` and `delivery_attempt` must not call Gemini twice or create a second writer side effect. A stale delivery with an older `delivery_attempt` must be acknowledged as ignored and must not mutate state.

---

## Preserve the validated durable boundary

Keep `capture-service` as the sole owner of:

```text
Discord Gateway intake
SQLite ledger
delivery state
delivery attempts
retry schedule
processing leases
receipt delivery
startup reconciliation
periodic reconciliation
```

n8n may interact with capture-service only through authenticated internal HTTP routes. n8n must never open `ledger.sqlite3`, mount `/var/lib/second-brain`, write Discord receipts directly, read Discord Gateway history, or claim intake durability.

## 14. Use acknowledge-forwarded as the atomic winner gate

Call:

```text
POST http://capture-service:8000/internal/captures/:capture_id/delivery/acknowledge-forwarded
```

Body: `{ "delivery_attempt": 1 }`.

Read the `outcome` field from the response. Branch on its value before proceeding:

**`changed`** — this execution wins the transition. Continue to `Respond 202 Accepted`, then `Acknowledge Classifying`, then Gemini. Only the execution receiving `changed` may proceed past this node.

**`idempotent_replay`**, **`stale_attempt`**, or **`invalid_state`** — another execution already claimed this attempt, or the capture is no longer in a claimable state. Return `Respond 202 Ignored` immediately:

```json
{ "accepted": true, "capture_id": "...", "delivery_attempt": 1, "ignored": true, "reason": "idempotent_replay" }
```

Do not continue to `Acknowledge Classifying`, Gemini, or the writer stub. Do not mutate any state. End the execution.

The initial n8n concurrency limit of one reduces the chance of a concurrent race, but must not be the correctness mechanism. This outcome branch is the actual correctness guarantee. If the n8n execution crashes after returning 202, the processing lease eventually expires and the stale-lease reaper schedules a new attempt.

**Capture API contract requirement**: for the workflow to branch on `outcome`, the `acknowledge-forwarded` route must return HTTP 200 with a typed outcome body for all four states — `changed`, `idempotent_replay`, `stale_attempt`, and `invalid_state`. HTTP 409 must not be used for `invalid_state` when the workflow expects to handle it silently. See section 37 for the full API contract change and required tests.

## 23. Add explicit retry branches inside n8n

Do not rely only on the Error Trigger workflow for recoverable processing failures. For recoverable failures after webhook acceptance, call `POST /internal/captures/:capture_id/delivery/schedule-retry` with:

```json
{ "delivery_attempt": 1, "error_type": "gemini_timeout", "reason_type": "explicit_workflow_failure" }
```

Use lowercase canonical slugs throughout. Match the taxonomy from `src/secondbrain/downstream_errors.py`.

### Recoverable failures — schedule retry

```text
Gemini timeout
Gemini 429
Gemini 5xx
capture-service temporary 503
writer-stub timeout
writer-stub 5xx
classification validation failure
unexpected response schema
unhandled workflow exception
```

When capture-service itself is unavailable, n8n cannot reliably reach the retry endpoint. The recovery path depends on when the failure occurs:

**Before HTTP 202 handoff**: if n8n cannot reach capture-service to fetch the capture or call `acknowledge-forwarded`, return a non-202 response to the dispatcher. The dispatcher treats non-202 as a delivery failure and schedules retry through SQLite. No state mutation occurs inside n8n.

**After HTTP 202 handoff**: if capture-service becomes unavailable after `acknowledge-forwarded` returned `changed`, n8n cannot report the failure. Allow the processing lease to expire. The stale-lease reaper detects the expired lease and schedules a new delivery attempt when capture-service returns.

### Terminal failures — acknowledge-failed

```text
defense-in-depth secret detection
invalid webhook envelope
impossible contract violation
repeated retry exhaustion (handled by capture-service)
```

Use safe category slugs only:

```text
gemini_timeout
gemini_rate_limit
gemini_server_error
writer_stub_timeout
writer_stub_unavailable
capture_service_unavailable
classification_validation_failure
unexpected_response_shape
workflow_unhandled_exception
defense_in_depth_secret_detected
contract_violation
```

Clarify invalid-envelope handling. A webhook envelope may be untrustworthy in two distinct ways:

**No trustworthy `capture_id` or `delivery_attempt`** — return a non-202 response immediately. Do not attempt to identify or mutate a capture row. The dispatcher treats non-202 as a delivery failure and schedules local retry. Do not call `acknowledge-failed`.

**Valid capture reference, but impossible contract violation** (e.g., `capture_id` is valid and routable, but an internal assertion fails after `acknowledge-forwarded`) — call `POST /delivery/acknowledge-failed` with `error_type = contract_violation`. This path terminates a specific, identified capture rather than guessing.

## 37. Align acknowledge-forwarded API with the winner-gate branch requirement

The n8n winner-gate branches on the `outcome` field read from a normal HTTP 200 response. The workflow cannot branch on HTTP status codes. Update `src/secondbrain/capture_api.py` so the `acknowledge-forwarded` route returns HTTP 200 for all four expected state-machine outcomes:

| Outcome | Meaning | HTTP status |
| --- | --- | --- |
| `changed` | this request won the transition | 200 |
| `idempotent_replay` | capture is already FORWARDED — safe replay | 200 |
| `stale_attempt` | request `delivery_attempt` is behind the current value | 200 |
| `invalid_state` | capture is in a state that cannot transition to FORWARDED | 200 |

Reserve HTTP 409 for a genuine contract conflict that represents a programming error, not for an expected state-machine outcome that workflows handle silently.

### Response shape for all four outcomes

```json
{
  "capture_id": "SB-YYYYMMDD-NNNN",
  "delivery_attempt": 1,
  "outcome": "invalid_state",
  "delivery_status": "COMPLETE"
}
```

## 38. Make explicit failure callbacks replay-safe across the HTTP boundary

A network callback can succeed inside capture-service while its response is lost before n8n receives it. If n8n retries, the system must recognize the replay and terminate quietly rather than returning HTTP 409 and potentially triggering an unintended Error Handler execution.

Apply the same typed HTTP 200 outcome pattern from section 37 to the explicit failure callbacks:

### schedule-retry

| Replay condition | Outcome | HTTP status |
| --- | --- | --- |
| Same attempt already in `RETRY_WAIT` | `ignored_retry_already_scheduled` | 200 |
| Request `delivery_attempt` is older than current | `ignored_stale_attempt` | 200 |
| Capture already `COMPLETE` or `FAILED` | `ignored_already_terminal` | 200 |
| Retry successfully scheduled | `retry_scheduled` | 200 |
| Retry limit exhausted — capture now `FAILED` | `terminal_failure` | 200 |

### acknowledge-failed

| Replay condition | Outcome | HTTP status |
| --- | --- | --- |
| Same terminal failure already recorded | `idempotent_replay` | 200 |
| Request `delivery_attempt` is older than current | `ignored_stale_attempt` | 200 |
| Capture already `COMPLETE` | `ignored_already_terminal` | 200 |
| Terminal failure successfully recorded | `terminal_failure` | 200 |

Reserve HTTP 409 for a genuine conflicting contract mutation that represents a programming error, not for a normal idempotent replay.

### Failure callback API unit tests

```text

schedule-retry: no replay case returns HTTP 409
acknowledge-failed: same-attempt replay returns idempotent_replay and HTTP 200
acknowledge-failed: stale-attempt replay returns ignored_stale_attempt and HTTP 200
```
