# SB-132 — WAL-aware backup procedure (TD-006)

**Branch:** `tech-debt/wal-aware-backup`

Resolves:
- [TD-006](../../tech_debt_1/TD-006-wal-aware-backup-restore.md) — Define a WAL-aware backup and restore procedure

---

## Problem

The existing `scripts/backup-ledger.sh` was written before WAL mode was fully validated in production. A raw `cp` of a live WAL-mode SQLite database is not safe — the WAL file may contain transactions that have not been checkpointed into the main database file, leaving the backup in an inconsistent state.

---

## Required backup method

Use one of the following safe approaches:

**Option A — SQLite `.backup` API (preferred)**

Call `VACUUM INTO '/path/to/backup.sqlite3'` from within a running process, or use Python's `sqlite3.Connection.backup()` to produce a consistent point-in-time copy with WAL transactions included.

**Option B — `sqlite3` CLI with `.backup`**

```bash
sqlite3 /var/lib/second-brain/ledger.sqlite3 ".backup /tmp/ledger-backup.sqlite3"
```

This uses the SQLite backup API and is WAL-safe.

**Do not use:**

- `cp` of a live `.sqlite3` or `.sqlite3-wal` file.
- `rsync` without stopping writes first.
- Stopping the service to take a backup (availability requirement).

---

## Checklist

### Backup script

- [ ] `scripts/backup-ledger.sh` uses the SQLite backup API (not `cp`).
- [ ] The script produces a consistent backup even while `capture-service` is writing.
- [ ] The WAL file and SHM file do not need to be separately copied.
- [ ] The script logs success with a timestamp and the backup file's size.
- [ ] The script exits non-zero on failure and logs the error.

### Restore validation

- [ ] `scripts/restore-test.sh` restores the backup into a temporary directory.
- [ ] The restore test opens the backup database and runs a basic integrity check (`PRAGMA integrity_check`).
- [ ] The restore test does NOT use the backup as the live database.
- [ ] The test exits non-zero if integrity check fails.

### Documentation

- [ ] `docs/OPERATIONS.md` (from SB-129) describes the backup method and why raw `cp` is unsafe for WAL-mode SQLite.

---

## Acceptance criteria

- `scripts/backup-ledger.sh` uses the SQLite backup API.
- A backup produced while `capture-service` is actively writing passes `PRAGMA integrity_check`.
- `scripts/restore-test.sh` passes against the produced backup.
- The scripts never use `cp` or `rsync` on a live WAL-mode database file.
