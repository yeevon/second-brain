## TD-003: Event loop contention

### Phase 1 — Instrumentation (required)

Add structured log events emitted by the `SQLiteRuntime` worker:

- `sqlite_queue_depth` — number of pending jobs at time of measurement.
- `sqlite_queue_wait_ms` — milliseconds a job waited in the queue before execution began.
- `sqlite_job_duration_ms` — milliseconds from job start to completion.
- `sqlite_busy_retry_count` — number of SQLITE_BUSY retries for a given job.
- `sqlite_busy_exhausted_count` — number of times the retry budget was fully exhausted.

All fields must be numeric metadata. No exception messages or raw SQL in log output.

### Phase 2 — Async adapter (only if phase 1 data justifies it)

If measured queue wait times or job durations show that blocking ledger calls are delaying Discord heartbeats or API responses:

- Move blocking ledger calls to `asyncio.to_thread(...)` or an equivalent async boundary.
- Do not introduce `aiosqlite` or any async database library without evidence of a real problem.
- Do not change the serialized `SQLiteRuntime` worker design speculatively.

Phase 2 may be deferred indefinitely if production measurements do not show contention.

## TD-004: Startup and shutdown watchdogs

### Startup watchdog

- `SQLiteRuntime.start()` must raise within a configurable timeout (e.g. `SQLITE_STARTUP_TIMEOUT_S`, default 10 seconds) if the worker thread does not signal ready.
- A timeout must produce a clear log event and a startup failure — it must never leave the service in a half-started state silently.

### Shutdown watchdog

- `SQLiteRuntime.stop()` must drain pending jobs and shut down the worker thread within a configurable timeout (e.g. `SQLITE_SHUTDOWN_TIMEOUT_S`, default 10 seconds).
- Jobs that remain in the queue after the timeout are logged with their count and types.
- The shutdown watchdog must not block graceful service shutdown indefinitely.

---

## TD-007: Constructor validation

Add defensive checks in `SQLiteRuntime.__init__` or an equivalent constructor:

- Raise `ValueError` immediately if `db_path` is empty or None.
- Raise `ValueError` immediately if `queue_maxsize` is zero or negative.
- Raise `ValueError` immediately if `busy_retry_attempts` is negative.
- Log the resolved absolute `db_path`, `queue_maxsize`, and retry configuration at INFO level during construction.

These checks prevent a misconfigured runtime from failing silently at first use.

---

## Acceptance criteria

- TD-003 phase 1: structured log events for queue depth, wait, duration, and busy retries appear in the service log under normal operation.
- TD-004: `SQLiteRuntime.start()` fails with a clear error if the worker thread does not become ready within the timeout.
- TD-004: `SQLiteRuntime.stop()` completes or logs remaining jobs within the shutdown timeout.
- TD-007: `SQLiteRuntime` raises `ValueError` immediately for invalid constructor arguments.
- No exception message text (stack traces, SQL, raw error strings) appears in structured log events.

**Done when:** all three TD items pass their acceptance criteria and are moved to Resolved in `docs/tech_debt_1/README.md`.
