# SB-131 — Schema migration safety (TD-005)

**Branch:** `tech-debt/schema-migration-safety`

Resolves:
- [TD-005](../../tech_debt_1/TD-005-validate-legacy-schema-adoption.md) — Validate adopted legacy schemas before recording migration adoption

---

## Problem

The current migration runner records a schema version as applied when the SQL executes without error. It does not verify that the resulting schema matches the expected shape before marking the migration done.

For migrations that adopt a pre-existing legacy schema (tables created outside the migration framework), this means the recorded version may not correspond to any validated schema state.

---

## What to implement

After each migration's SQL executes successfully and before writing the version number to `schema_migrations`:

1. Run a schema assertion for that migration version.
2. The assertion checks that the expected tables, columns, and indexes exist.
3. If the assertion fails, roll back the migration recording and raise a startup error.

### Assertion approach

Use SQLite's `PRAGMA table_info(table_name)` and `PRAGMA index_list(table_name)` to inspect the live schema. Compare against an expected list of columns and indexes embedded alongside the migration definition.

Assertions must be:
- Read-only (no DDL inside the assertion).
- Idempotent (safe to re-run).
- Bounded (cover only the tables changed by the migration).

### Legacy adoption migrations

For any migration that adopts a table that existed before the migration framework, the assertion must additionally verify the exact column set, column types, and NOT NULL constraints match the canonical definition in the architecture.

---

## What not to do

- Do not add assertions that rerun on every startup against every migration. Assertions run only once, when the migration is first applied.
- Do not alter the schema inside an assertion.
- Do not write migration assertions for resolved migrations that have already been validated by the existing V2/V3 milestone checkpoints unless the migration adopts legacy schema state.

---

## Acceptance criteria

- A migration that produces incorrect schema (e.g. missing column, wrong type) causes startup to fail with a clear error before the version is recorded.
- A correctly applied migration passes the assertion and records its version.
- Legacy adoption migrations assert the expected column set before marking the version applied.
- No DDL executes inside an assertion function.
- Tests cover: assertion pass, assertion fail (wrong column), assertion fail (missing table), legacy adoption with correct schema, legacy adoption with missing column.
