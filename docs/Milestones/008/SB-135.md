## TD-011: Configuration preflight command

### Goal

Add a CLI command that validates configuration before starting the service. Currently, misconfigured values are discovered at first use, potentially mid-operation.

### Command

```bash
secondbrain preflight
```

### Checks

- All required environment variables are present (Discord token, Gemini key, etc.).
- `VAULT_PATH` resolves to an existing directory.
- `LEDGER_PATH` directory exists and is writable.
- SQLite can be opened at `LEDGER_PATH` (non-destructive open).
- `GEMINI_MODEL` is set to the pinned model string (not a floating alias like `gemini-flash-latest`).
- Discord token format passes a basic structural check (does not verify authentication against Discord).

### Output

```text
✓ DISCORD_BOT_TOKEN         — set
✓ GEMINI_API_KEY            — set
✓ VAULT_PATH                — /opt/second-brain/vault (exists)
✓ LEDGER_PATH               — /var/lib/second-brain/ledger.sqlite3 (directory writable)
✓ GEMINI_MODEL              — gemini-3.5-flash (pinned)
✗ DISCORD_CAPTURE_CHANNEL_ID — missing
```

Exit code 0 = all checks pass. Exit code 1 = one or more checks failed. Failures list which checks failed.

The preflight command must never start the Discord Gateway or make network calls to Discord or Gemini.

## TD-012: Background task liveness monitoring

### Goal

The stale-lease reaper and periodic reconciliation loop are background asyncio tasks. If one crashes and is not restarted, the service continues running silently without that background function. There is currently no mechanism to detect or alert on this.

### What to implement

- Each long-running background task records a heartbeat timestamp in `system_state` after every successful pass.
- The existing service heartbeat loop (already writing `capture_service_state`) also checks that background task heartbeats are within expected intervals.
- If a background task heartbeat is older than a configurable threshold (e.g. `REAPER_LIVENESS_THRESHOLD_S`, `RECONCILE_LIVENESS_THRESHOLD_S`), the service logs a structured warning event and sets a `background_task_stale` flag readable by `sb status`.
- The liveness check must not restart crashed tasks automatically — it must alert so the operator can investigate.
- Dead-task detection (already partially implemented) is the restart mechanism; liveness monitoring is the alert mechanism.

In `sb status` and `secondbrain status`:

- Report last reaper heartbeat timestamp and whether it is within the expected interval.
- Report last reconciliation heartbeat timestamp and whether it is within the expected interval.
- Report `background_task_stale: true/false`.

## TD-013: Persist last-successful-vault-write metadata

### Goal

The current `sb status` output does not report when the last successful vault write occurred (note filed or correction applied). This information is useful for diagnosing a stalled writer-service or detecting a classification pipeline that stopped filing notes.

### What to implement

- When `writer-service` successfully files a note or applies a correction and calls back to `capture-service`, `capture-service` writes a `last_vault_write_at` key to `system_state` with the current timestamp.
- This is a single row update — not a new table.
- The `secondbrain status` read-only command reads `last_vault_write_at` from `system_state` and includes it in the Note lifecycle section.
- `sb status` includes `last_vault_write_at` in the Note lifecycle output.

- TD-011: `secondbrain preflight` exits 0 when all checks pass and 1 when any check fails.
- TD-011: The preflight command does not start the Discord Gateway or make Discord/Gemini network calls.
- TD-011: Missing required variables and invalid paths are identified by name in the failure output.
- TD-012: Stale background task heartbeats produce a structured log warning.
- TD-012: `sb status` reports last reaper and reconciliation heartbeat timestamps and staleness flag.
- TD-013: `last_vault_write_at` is updated in `system_state` after every successful vault write callback.
- TD-013: `sb status` and `secondbrain status` display `last_vault_write_at`.
- Tests cover: preflight with missing env var, preflight with missing vault dir, background task heartbeat freshness check, `last_vault_write_at` update after successful filing callback.
