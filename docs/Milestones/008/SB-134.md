## TD-002: Shutdown cleanup resilience

### Problem

The current shutdown path attempts to cancel background tasks and close resources in sequence. If one cleanup step raises an exception, it can prevent subsequent cleanup steps from running, leaving resources in an uncertain state.

### What to implement

The shutdown sequence must:

- Run each cleanup step independently, catching and logging exceptions without stopping later steps.
- Log a structured warning for every step that fails, including the step name and error type.
- Complete gracefully regardless of which individual steps fail.
- Not raise from the top-level shutdown handler (the OS or Compose restart policy handles terminal failures).

Example pattern:

```python
for step_name, step_fn in shutdown_steps:
    try:
        await step_fn()
    except Exception as exc:
        log.warning("shutdown_step_failed", step=step_name, error_type=type(exc).__name__)
```

- The shutdown sequence must attempt all steps even if early steps fail.
- Steps are still attempted in dependency order where possible (e.g. close Discord before closing SQLite).
- A test must confirm: a cleanup step that raises does not prevent later steps from running.

## TD-009: Durable receipt-repair tracking

### Problem

When the original receipt message cannot be edited (e.g. Discord API failure), the service sends a replacement receipt and stores the new `receipt_message_id`. This replacement is not explicitly recorded as a distinct event in `capture_events`, making it invisible in the audit trail and in `sb status`.

### What to implement

When a receipt replacement occurs:

- Append a `RECEIPT_REPLACED` event to `capture_events` with:
  - `old_receipt_message_id`
  - `new_receipt_message_id`
  - `reason` (error type that caused the replacement — sanitized, no raw message)
- Update `captures.receipt_message_id` to the new message ID atomically with the event append, in the same SQLite transaction.

In `sb status`:

- Report the count of `RECEIPT_REPLACED` events in the last 24 hours as `receipt_repairs_today`.
- A non-zero count is informational, not an alert condition.

In `secondbrain status` (read-only command):

- Include `receipt_repairs_today` in the Note lifecycle section.

## Acceptance criteria

- TD-002: A shutdown step that raises does not prevent subsequent shutdown steps from completing.
- TD-002: Every failed shutdown step produces a structured log warning with step name and error type.
- TD-002: The shutdown handler does not propagate exceptions to the OS signal handler.
- TD-009: A forced receipt-edit failure produces a `RECEIPT_REPLACED` event in `capture_events`.
- TD-009: `captures.receipt_message_id` is updated atomically with the `RECEIPT_REPLACED` event.
- TD-009: `sb status` and `secondbrain status` report `receipt_repairs_today`.
- Tests cover: multiple failed shutdown steps, receipt replacement with event append, receipt replacement with failed transaction (no partial state).
