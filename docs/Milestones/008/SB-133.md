## TD-001: Exception message sanitization

### Problem

Exception messages from SQLite worker jobs are currently persisted as-is into `captures.last_error` and `capture_events.event_payload_json`. These may include raw SQL text, stack traces, exception class names, or other internal details that should not be stored in the ledger.

Design invariant: the ledger must never contain a raw exception body.

### What to implement

- Replace raw exception messages with a sanitized error summary before persisting to `last_error` or `capture_events`.
- The sanitized summary contains: error type (e.g. `SQLiteBusyError`, `ValidationError`), a short human-readable description, and numeric metadata (retry count, duration) where available.
- The sanitized summary must not contain: full stack traces, raw SQL statements, full exception `.args`, or Python module paths.
- A separate structured log event (written to the service log, not the ledger) may include fuller diagnostic detail at DEBUG level, without raw user content.

### Sanitization rule

```python
# Acceptable in last_error / capture_events
"SQLiteBusyError: write queue exhausted after 5 retries (job: insert_capture)"

# Not acceptable
"sqlite3.OperationalError: database is locked\nTraceback (most recent call last):\n  ..."
```

## TD-008: Centralized structured logging

### Problem

Log output is currently emitted inconsistently: some modules use `print()`, some use `logging.getLogger()` directly, some emit plain strings, and some emit structured dicts. This makes filtering and monitoring difficult.

### What to implement

- Create one application-level logging configuration (e.g. `secondbrain/logging_config.py`) applied at service startup.
- All modules use `logging.getLogger(__name__)` — no bare `print()` for operational events.
- Log records include: `timestamp`, `level`, `logger` (module path), `capture_id` (where available), and a `message` field.
- Sensitive fields are never included: `raw_text`, `redacted_text`, Discord token, Gemini API key.
- Log level is configurable via `LOG_LEVEL` environment variable (default `INFO`).
- DEBUG level may include fuller diagnostic detail but must still exclude sensitive field values.
- The structured log format is consistent enough that `grep` and `jq` can filter by `capture_id` or `level`.

---

## Acceptance criteria

- TD-001: No raw stack traces or raw SQL appear in `captures.last_error` or `capture_events.event_payload_json` in any test or production scenario.
- TD-001: Error type and short description appear in `last_error` when a worker job fails.
- TD-008: All operational log output uses the `logging` module (no bare `print()` statements in production code paths).
- TD-008: Log records are consistently structured (timestamp, level, logger, message).
- TD-008: `LOG_LEVEL=DEBUG` produces fuller diagnostic output without exposing raw message text or credentials.
- Tests confirm: a forced SQLite failure persists a sanitized error summary, not a raw exception message.
