# SB-116 — Add Git conflict and stale-lock failure handling

**Branch:** `feature/writer-safe-failure`

Add explicit failure detection and safe failure behavior for every bad state the Git-backed write sequence can reach. The guiding principle: fail visibly, preserve the raw SQLite capture, and leave the vault in a state the operator can inspect and resolve manually.

---

## Goal

After SB-116, every Git-layer failure produces a typed error response that:

```text
returns a non-2xx HTTP status with a structured error body
never destroys the raw capture in capture-service SQLite
never partially commits a note (no orphaned unstaged changes)
never blindly deletes .git/index.lock
never auto-resolves a merge conflict
leaves the vault working tree in a known, documented state
```

The n8n intake workflow routes each failure type through the appropriate existing error path (retry or terminal), exactly as it handles Gemini and writer-stub failures.

---

## Failure inventory

### 1. Non-fast-forward merge failure

`git merge --ff-only origin/main` exits non-zero because the local clone has diverged from the remote.

**Cause:** A previous write committed locally but the push failed or was interrupted. The local HEAD is now ahead of the remote in a way that cannot fast-forward.

**Safe response:** Return HTTP 409 with `error_type: git_merge_conflict`. Do not auto-resolve. Do not run `git merge` without `--ff-only`. Do not run `git rebase`. Leave the vault working tree as-is for operator inspection.

**Recovery path:** n8n routes to `acknowledge-failed` — not `schedule-retry` — because a retry cannot resolve a diverged local clone. The operator must inspect and resolve the divergence manually:

```bash
git -C /opt/second-brain/vault fetch origin
git -C /opt/second-brain/vault log --oneline origin/main..HEAD  # see diverged commits
# Decide: push the diverged commit, or reset and re-file through capture-service
```

After resolving, the raw capture remains in capture-service SQLite and can be re-filed.

### 2. Rejected push

`git push origin main` exits non-zero because the remote has advanced since the last fetch.

**Cause:** A concurrent writer-service instance (e.g., during a rolling deploy) or a manual commit on the remote. The write sequence fetched and merged at time T, but another commit landed on the remote between the merge and the push.

**Safe response:** Return HTTP 409 with `error_type: git_push_rejected`. The note file and audit event are already written to the local clone but not committed. Roll back the local filesystem write before releasing the flock. Do not leave a written file without a corresponding commit.

**Recovery path (automatic):** capture-service schedules retry. The retry attempt fetches, merges (now including the intervening commit), and pushes successfully. No operator action required unless push rejection persists.

### 3. Existing `.git/index.lock`

The Git index lock file `/opt/vault/.git/index.lock` exists when the write sequence begins.

**Cause:** A previous `git add` or `git commit` was interrupted while the lock was held. Git creates this file at the start of index-modifying operations. Unlike the advisory flock on `.writer.lock`, `.git/index.lock` is managed by Git and may legitimately exist.

**Safe response:** Return HTTP 503 with `error_type: git_index_locked`. Do not delete `.git/index.lock`. Stale index locks are sometimes legitimately held by a running Git process. Deleting a live lock file corrupts the in-progress Git operation.

**Recovery path (operator):** Verify no Git process is using the vault (check `lsof /opt/vault/.git/index.lock`). If no process holds it, delete the stale lock file manually. Log the event so the operator is notified.

**Why not auto-delete:**

```text
Deleting an index lock held by a live process corrupts Git index state.
A crashed write leaves a stale lock, but a concurrent read (git status, git log)
also holds a lock briefly. Auto-deletion is not safe without verifying the lock
is actually stale.
```

### 4. Duplicate capture_id detected in vault

The idempotency scan finds a file with matching `capture_id` frontmatter at the expected path. This is the normal idempotent replay path from SB-114 — return the existing path.

A separate failure case: the scan finds the `capture_id` frontmatter in more than one file.

**Cause:** A previous write committed twice due to a retry race or a manual vault edit. The vault is now inconsistent.

**Safe response:** Return HTTP 409 with `error_type: capture_id_duplicate`. Do not write a third copy. Do not delete either existing copy. Log which files contain the duplicate.

**Recovery path:** n8n routes to `acknowledge-failed` — not `acknowledge-inbox` — because no valid `note_path` exists (the write was refused). The operator must inspect both vault files, remove the duplicate manually, commit and push, then re-file the capture through capture-service if needed.

### 5. Dirty working tree at write time

The vault working tree contains uncommitted changes when the write begins (after fetch + merge).

**Cause:** A previous write crashed after `write_note_file` or `append_audit_event` but before the `git commit`. The note file or audit event is on disk but not committed.

**Safe response:** Return HTTP 503 with `error_type: git_worktree_dirty`. Do not overwrite or delete the existing uncommitted changes. They may represent a note that needs to be committed manually.

**Recovery path (operator):** Inspect the uncommitted changes. Commit or discard them. Once the working tree is clean, the next retry will proceed.

### 6. Path escape attempt

A derived note path contains `../`, an absolute path segment, a null byte, or any component that would resolve outside the vault root.

**Cause:** Classification produced a folder or project value that passed Pydantic validation but generates a dangerous path when combined with the vault root. This should not happen in normal operation; its presence indicates either a classification model defect or an attempted injection.

**Safe response:** Return HTTP 422 with `error_type: path_traversal_attempt`. Do not write any file. Log the rejected path in a safe, non-revealing way (log the path shape, not the raw classification values).

**Recovery path:** No operator action required for the capture. The raw capture remains in capture-service SQLite. n8n routes to `acknowledge-failed` — not `acknowledge-inbox` — because no note was written and there is no valid `note_path` to pass to `acknowledge-inbox`. The failure is terminal and deterministic for that classification.

## 1. Define typed Git exception hierarchy

Add `src/writerservice/git_errors.py`:

```python
class WriterError(Exception):
    """Base class for all writer-service failures."""
    error_type: str
    http_status: int
    retryable: bool

class GitMergeConflictError(WriterError):
    error_type = "git_merge_conflict"
    http_status = 409
    retryable = False

class GitPushRejectedError(WriterError):
    error_type = "git_push_rejected"
    http_status = 409
    retryable = True

class GitIndexLockedError(WriterError):
    error_type = "git_index_locked"
    http_status = 503
    retryable = True

class CaptureDuplicateError(WriterError):
    error_type = "capture_id_duplicate"
    http_status = 409
    retryable = False

class PathTraversalError(WriterError):
    error_type = "path_traversal_attempt"
    http_status = 422
    retryable = False

class GitWorkdirDirtyError(WriterError):
    error_type = "git_worktree_dirty"
    http_status = 503
    retryable = False  # operator must inspect and resolve stale working-tree changes
```

Map each exception to its HTTP response in the FastAPI exception handler. Never expose raw Git subprocess output in the error response body.

```python
def check_index_lock(vault_path: Path) -> None:
    lock_file = vault_path / ".git" / "index.lock"
    if lock_file.exists():
        raise GitIndexLockedError(
            f"Git index lock exists at {lock_file}. "
            "A previous Git operation may have been interrupted. "
            "Verify no Git process is running, then delete the lock file manually."
        )
```

Call `check_index_lock()` immediately after acquiring the advisory flock, before any Git operation.

```python
def git_merge_ff_only(vault_path: Path) -> None:

        )
```

## 4. Detect push rejection

Update `git_push()` in `git_ops.py`:

```python
def git_push(vault_path: Path) -> None:
    result = subprocess.run(
        ["git", "push", "origin", "main"],
        cwd=vault_path,
        capture_output=True,
        text=True,

```

## 5. Roll back local writes on any post-filesystem failure

The write sequence is: commit → resolve hash → push. If any step from `git add` onward fails, the local clone may have an orphaned commit that diverges from the remote. Before writing, assert the working tree is clean and record the pre-write HEAD; on failure, use `git reset --hard` plus `git clean` to restore the original state.

Add `GitWorkdirDirtyError` to `git_errors.py`:

```python
class GitWorkdirDirtyError(WriterError):
    error_type = "git_worktree_dirty"
    http_status = 503
    retryable = False  # operator must inspect and commit or discard the stale change
```

Update `writer.py`:

```python
with vault_write_lock(vault_path / ".writer.lock"):
    check_index_lock(vault_path)
    git_fetch(vault_path)
    git_merge_ff_only(vault_path)   # raises GitMergeConflictError → no files written yet

    # Refuse to write if the working tree is dirty.
    # A dirty tree indicates a previous write crashed after filesystem mutation
    # but before commit. Do not erase those changes automatically.
    check_working_tree_clean(vault_path)  # raises GitWorkdirDirtyError if dirty

    pre_write_head = git_rev_parse_head(vault_path)
    audit_log_existed = audit_log_path.exists()

    try:
        write_note_file(vault_path, note_path, request)
        append_audit_event(audit_log_path, request, note_path)

        git_add(vault_path, note_path, "99_log/events.ndjson")
        git_commit(vault_path, f"note: {request.capture_id} via writer-service")
        commit_hash = git_rev_parse_head(vault_path)
        git_push(vault_path)        # raises GitPushRejectedError if rejected
    except Exception:
        _rollback_to_head(
            vault_path, pre_write_head, note_path,
            audit_log_existed, audit_log_path,
        )
        raise
```

Implement `check_working_tree_clean()`:

```python
def check_working_tree_clean(vault_path: Path) -> None:
    result = subprocess.run(
        ["git", "status", "--porcelain"],
        cwd=vault_path,
        capture_output=True,
        text=True,
    )
    if result.stdout.strip():
        raise GitWorkdirDirtyError(
            "Vault working tree is not clean. "
            "A previous write may have crashed after filesystem mutation. "
            "Inspect uncommitted changes before retrying."
        )
```

Implement `_rollback_to_head()`:

```python
def _rollback_to_head(
    vault_path: Path,
    pre_write_head: str,
    written_note: Path,
    audit_log_existed: bool,
    audit_log_path: Path,
) -> None:
    # Hard reset restores tracked files (index and working tree) to pre_write_head.
    # This handles: failure after git add, after git commit, and after git push.
    subprocess.run(
        ["git", "reset", "--hard", pre_write_head],
        cwd=vault_path,
        check=False,
    )
    # git clean removes untracked files that git reset --hard leaves behind.
    # Use targeted clean rather than -fdx to avoid removing unrelated files.
    rel_note = str(written_note.relative_to(vault_path))
    subprocess.run(
        ["git", "clean", "-f", rel_note],
        cwd=vault_path,
        check=False,
    )
    # Remove the audit log only if this write created it.
    # If it already existed, truncating or removing it would destroy prior audit history.
    if not audit_log_existed and audit_log_path.exists():
        audit_log_path.unlink(missing_ok=True)
```

`git reset --hard <pre_write_head>` handles tracked-file state:

```text
failure after git add    → unstages files, discards working-tree changes to tracked files
failure after git commit → removes the local commit, restores working tree
failure after git push   → removes the local commit that the remote rejected
```

`git clean -f <note_path>` removes the newly created note file when it is still untracked (e.g. after a failure during `git add`).

The rollback does not raise. If `git reset --hard` itself fails, log the failure and still re-raise the original exception. The vault will require operator inspection in that case.

Do not apply rollback to `git_merge_ff_only` or `check_working_tree_clean` failures — no filesystem mutations have occurred at those points.

## 6. Detect duplicate capture_id in vault

Update `find_note_by_capture_id()` in `vault.py`:

```python
def find_note_by_capture_id(vault_path: Path, capture_id: str) -> Path | None:
    matches = []
    for md_file in vault_path.rglob("*.md"):
        if _frontmatter_contains_capture_id(md_file, capture_id):
            matches.append(md_file)

    if len(matches) == 0:
        return None
    if len(matches) == 1:
        return matches[0]
    raise CaptureDuplicateError(
        f"Found {len(matches)} vault files with capture_id {capture_id}. "
        "Manual deduplication required. "
        f"Files: {', '.join(str(p.relative_to(vault_path)) for p in matches)}"
    )
```

Log the relative file paths. Do not log file contents.

## 7. Update the error response format

All `WriterError` subclasses produce this HTTP error body:

```json
{
  "error_type": "git_push_rejected",
  "retryable": true,
  "message": "git push rejected: remote has advanced. Retry will fetch and re-attempt."
}
```

Do not include:

```text
raw Git subprocess output
classification body
raw capture text
writer token
vault deploy key path
```

Register a FastAPI exception handler:

```python
@app.exception_handler(WriterError)
async def writer_error_handler(request: Request, exc: WriterError):
    return JSONResponse(
        status_code=exc.http_status,
        content={
            "error_type": exc.error_type,
            "retryable": exc.retryable,
            "message": str(exc),
        },
    )
```

## 8. Update the n8n intake workflow

### HTTP Request node options

By default, n8n's HTTP Request node throws on non-2xx responses and does not expose the status code to downstream nodes. Configure the `Submit to Writer Service` node with:

```text
Response → Never Error       (enabled)
Response → Include Response Headers and Status  (enabled)
Response Format → JSON
```

This makes the full response available — `statusCode` and `body` — regardless of the HTTP status code, so downstream nodes can branch on both.

### Error routing graph

```text
Submit to Writer Service
    ↓
Evaluate Writer Response
    ├── statusCode 2xx → Acknowledge Filed / Acknowledge Inbox
    ├── statusCode 409, body.error_type = git_push_rejected   → schedule-retry
    ├── statusCode 503, body.error_type = git_index_locked    → schedule-retry
    ├── statusCode 409, body.error_type = git_merge_conflict  → acknowledge-failed
    ├── statusCode 503, body.error_type = git_worktree_dirty  → acknowledge-failed
    ├── statusCode 409, body.error_type = capture_id_duplicate → acknowledge-failed
    ├── statusCode 422, body.error_type = path_traversal_attempt → acknowledge-failed
    └── unexpected → Stop And Error (workflow_unhandled_exception)
```

Branch on `body.error_type` first, then confirm `statusCode` as a sanity check. Do not branch on `statusCode` alone — a future writer-service change could reuse the same status code for a different error type.

Map writer-service `error_type` values to the capture-service `error_type` taxonomy:

| `git_worktree_dirty` | `writer_git_worktree_dirty` |

## Out of scope

Do not implement:

```text
automatic conflict resolution (git merge, git rebase)
automatic .git/index.lock deletion
automatic deduplication of vault files
alerting or PagerDuty integration
Git revert automation
```

All recovery for vault corruption requires operator action. The design goal is that capture durability (SQLite) is never compromised — vault inconsistency is an operator concern, not an automatic recovery concern.
