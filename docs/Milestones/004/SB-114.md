# SB-114 — Create writer-service with a local-only vault

**Branch:** `feature/writer-service-api`

Extract deterministic Markdown generation into a dedicated HTTP service that replaces the `writer-stub` container. The initial implementation writes notes to a local filesystem only — no Git operations in this story.

---

## Goal

After SB-114:

```text
n8n Intake workflow
    ↓
POST http://writer-service:8001/internal/notes/file
    ↓
writer-service validates the filing request
    ↓
writer-service renders deterministic Markdown
    ↓
writer-service writes note to local vault filesystem
    ↓
writer-service returns real note path
    ↓
n8n calls acknowledge-filed with real path
    ↓
capture-service edits Discord receipt with filed status
```

The `writer-stub` container and its stub:// note paths are removed from the stack.

---

## Architecture rule: writer-service is the sole vault writer

Only `writer-service` may mutate files in the vault directory. No other container mounts or writes to the vault path. `capture-service` owns the SQLite ledger. `n8n` owns the classification workflow. `writer-service` owns vault file creation and updates.

---

## Preserve the validated Markdown generation behavior

The existing deterministic writer behavior from the MVP is the reference implementation. Preserve every constraint:

```text
schema validation on all classification fields
folder allowlist enforcement
title sanitization (strip dangerous characters)
project-slug sanitization
path traversal protection (reject ../ and absolute paths in derived paths)
deterministic filenames: YYYY-MM-DD--<capture_id>--<title-slug>.md
deterministic frontmatter field order
idempotency by capture_id (same capture_id always produces the same file path)
99_log/events.ndjson append for every vault write
```

Do not change the generated Markdown format or frontmatter structure in this story.

### Compose integration

Add `writer-service` to `compose.n8n.yaml` on the `backend` network only:

```yaml
services:
  writer-service:
    build:
      context: .
      dockerfile: writer-service/Dockerfile
    container_name: second-brain-writer-service
    restart: unless-stopped
    env_file:
      - ${WRITER_SERVICE_ENV_FILE:?WRITER_SERVICE_ENV_FILE must be set}
    volumes:

      - backend
    cap_drop:
      - ALL
    security_opt:
      - no-new-privileges:true
    healthcheck:
      test: ["CMD", "python", "-c", "import urllib.request; urllib.request.urlopen('http://127.0.0.1:8001/health')"]
      interval: 30s
      timeout: 5s
      retries: 5
      start_period: 20s
```

Do not publish any host ports. The service is reachable only over the `backend` Docker network.

Remove `writer-stub` from both `compose.n8n.yaml` and `compose.override.yaml` in the same commit that adds `writer-service`. The standard local workflow (`docker compose up -d`) loads `compose.override.yaml` automatically; if `writer-stub` remains there, local runs will still start the old container.

Add `writer-service` to `compose.override.yaml` with local-safe defaults:

```yaml
services:
  writer-service:
    environment:
      WRITER_SERVICE_TOKEN: ${WRITER_SERVICE_TOKEN:-dev-writer-service-token-change-me}
      VAULT_PATH: /opt/vault
      AUDIT_LOG_PATH: /opt/vault/99_log/events.ndjson
      GIT_SYNC_ENABLED: ${GIT_SYNC_ENABLED:-false}
    volumes:
      - second-brain-local-vault:/opt/vault

volumes:
  second-brain-local-vault:
```

This ensures `docker compose up -d` remains the standard local path with no extra flags.

## 4. Define the writer-service API

### `GET /health`

No authentication required. Returns:

```json
{ "status": "ok" }
```

Returns HTTP 200 when the service is running and the vault path is reachable. Returns HTTP 503 when the vault path is not writable.

### `POST /internal/notes/file`

Requires `X-Second-Brain-Writer-Token` header authentication.

Request:

```json
{
  "capture_id": "SB-YYYYMMDD-NNNN",
  "source_message_id": "123456789012345678",
  "created_at": "2026-06-12T18:00:00Z",
  "delivery_attempt": 1,
  "model": "gemini-2.5-flash",
  "prompt_version": "1",
  "classification": {
    "folder": "projects",
    "project": "second-brain",
    "note_type": "implementation-note",
    "title": "SB-114 writer service",
    "tags": ["writer", "second-brain"],
    "body": "Normalized note body.",
    "actions": [],
    "needs_clarification": false,
    "clarifying_question": null,
    "confidence": 0.92
  },
  "inbox_reason": null
}
```

Response (new note created):

```json
{
  "result": "FILED",
  "note_path": "20_projects/second-brain/2026-06-12--SB-20260612-0001--sb-114-writer-service.md",
  "git_commit_hash": null,
  "idempotent": false
}
```

Response (idempotent replay — same `capture_id`, same path already exists):

```json
{
  "result": "FILED",
  "note_path": "20_projects/second-brain/2026-06-12--SB-20260612-0001--sb-114-writer-service.md",
  "git_commit_hash": null,
  "idempotent": true
}
```

When `inbox_reason` is non-null, write to the `00_inbox/` folder regardless of the `classification.folder` value.

## 5. Define the request validation rules

### Authentication

Reject any request to an authenticated route without `X-Second-Brain-Writer-Token`. Return HTTP 401.

Reject any request with a token that does not match `WRITER_SERVICE_TOKEN`. Return HTTP 401.

Do not log the presented token value.

### Classification validation

Apply the same Pydantic `Classification` model used by `capture-service`. Reject:

```text
unknown fields (extra="forbid")
invalid folder enum value
invalid action status enum value
confidence < 0 or > 1
```

### Folder values and physical directory mapping

The shared `Classification` model uses logical enum values:

```text
inbox | people | projects | ideas | learning | admin
```

`writer-service` accepts only these logical values from n8n. Physical vault directory names (`00_inbox`, `20_projects`, etc.) are an internal implementation detail. Never expose or accept physical directory names in the API.

The internal mapping from logical value to vault subdirectory:

| `classification.folder` | Vault subdirectory |
| --- | --- |
| `inbox` | `00_inbox` |
| `people` | `10_people` |
| `projects` | `20_projects` |
| `ideas` | `30_ideas` |
| `learning` | `40_learning` |
| `admin` | `50_admin` |

Reject any `folder` value not in the logical enum. Return HTTP 422.

### Path traversal protection

Validate every derived path component before writing:

```text
reject components containing ..
reject components containing /
reject absolute path segments
reject null bytes
reject components starting with .
```

Return HTTP 422 with a non-revealing error message. Do not echo back the rejected path component.

### capture_id format

Validate `capture_id` matches `^SB-\d{8}-\d{4}$`. Return HTTP 422 for non-matching values.

### delivery_attempt

Must be an integer `>= 1`. Return HTTP 422 for values below 1.

## 6. Implement deterministic Markdown generation

Extract the `VaultWriter` rendering logic from the MVP unchanged. The existing implementation and a byte-for-byte golden fixture test (see section 12) are the authoritative specification — do not redefine rendering rules in prose.

### Filename format

```text
YYYY-MM-DD--<capture_id>--<title-slug>.md
```

The `capture_id` is embedded in the filename. This makes the file self-describing and prevents collisions when two captures share a similar title.

Title slug rules (match the MVP implementation exactly):

```text
lowercase
spaces → hyphens
strip: ! @ # $ % ^ & * ( ) + = [ ] { } | \ : ; " ' < > , ? /
collapse consecutive hyphens
strip leading and trailing hyphens
```

Example: `capture_id = SB-20260612-0001`, `title = "SB-114 writer service"` →

```text
2026-06-12--SB-20260612-0001--sb-114-writer-service.md
```

### Frontmatter

Copy the existing MVP frontmatter field set exactly. The fields and their order are:

```yaml
---
capture_id: SB-20260612-0001
source_message_id: "123456789012345678"
created_at: "2026-06-12T18:00:00Z"
area: projects
project: second-brain
note_type: implementation-note
tags:
  - writer
  - second-brain
actions:
  - text: "Review the writer-service API"
    status: open
lifecycle_status: active
model: gemini-2.5-flash
prompt_version: "1"
schema_version: "1"
---
```

Field order must be deterministic and match the above template. Do not use arbitrary dict iteration order.

Notes on field mapping:

- `area` ← `classification.folder` (logical value, e.g. `projects`)
- `source_message_id` ← `request.source_message_id`
- `created_at` ← `request.created_at`
- `lifecycle_status` is always `active` for new notes
- `model` and `prompt_version` are passed through from the filing request (add these to `FileNoteRequest`)
- `schema_version` is hardcoded by writer-service to `"1"`; do not accept it from n8n

### Body

Append the classification body verbatim. The exact separator format, any additional body sections, and their ordering are determined by the VaultWriter implementation. The golden fixture test is the authoritative correctness check.

### Vault subdirectory mapping

The `classification.folder` logical value determines the physical subdirectory (see section 5 for the mapping table). Project-level subdirectory creation, project slug sanitization, and inbox override logic are determined by the VaultWriter implementation — do not re-specify them in prose.

When `inbox_reason` is non-null, always write to `00_inbox/` regardless of `classification.folder`.

## 7. Implement idempotency by capture_id

Before writing a new note:

1. Search the vault for any existing file whose frontmatter contains `capture_id: <value>`.
2. Zero matches → write the new note, append the audit event, and return `idempotent: false`.
3. Exactly one match → return the existing file path as `idempotent: true`. Do not overwrite. Do not require the derived path to match the existing path — a retry may re-classify with a different title or project slug, producing a different derived path, but the existing note is still the correct filed result.
4. More than one match → return HTTP 409 with `error_type: capture_id_duplicate`. This case is handled explicitly in SB-116.

The `capture_id` frontmatter field is the authoritative idempotency key. The filename is derived and may drift across retries.

## 8. Implement the audit log

## 9. Update the n8n intake workflow

Rename the `Submit to Writer Stub` node to `Submit to Writer Service`. Rebuild both downstream branches to call `writer-service` and then `capture-service`.

### Extend the capture-service downstream response

`writer-service` requires `source_message_id` and `created_at` to render accurate frontmatter. These fields must be included in the `GET /internal/downstream/captures/:capture_id` response so n8n can pass them in the filing request.

Update the `DownstreamCaptureResponse` in `capture-service`:

| Response field | Source column |
| --- | --- |
| `source_message_id` | `discord_message_id` |
| `created_at` | `received_at` |

Both columns already exist in the `captures` table. No migration is required — only the response serializer needs updating.

### Classification routing and inbox_reason

The existing n8n intake workflow routes to inbox or file via a confidence gate Code node. Extend that node to also produce a typed `inbox_reason` slug so writer-service and acknowledge-inbox receive a structured reason.

Update the Code node to return `ClassificationValidationResponse`:

```javascript
// Output structure (extended with inbox_reason)
{
  valid: true,
  route: "file" | "inbox",
  confidence_met: true | false,
  inbox_reason: null | "classifier_selected_inbox" | "needs_clarification" | "low_confidence",
  errors: []
}
```

Routing rules — apply in this order:

| Condition | `route` | `inbox_reason` |
| --- | --- | --- |
| `classification.folder == "inbox"` | `"inbox"` | `"classifier_selected_inbox"` |
| `classification.needs_clarification == true` | `"inbox"` | `"needs_clarification"` |
| `classification.confidence < threshold` | `"inbox"` | `"low_confidence"` |
| none of the above | `"file"` | `null` |

`threshold` is the confidence gate value already configured in the workflow (e.g. `0.75`). Do not add a new variable.

n8n passes the `inbox_reason` slug directly to `POST /internal/notes/file` and, on the inbox branch, to `acknowledge-inbox` as `reason_type`. When `inbox_reason` is non-null, writer-service writes to `00_inbox/` regardless of `classification.folder`.

### File branch (confidence sufficient, no inbox_reason)

```text
Submit to Writer Service
    POST http://writer-service:8001/internal/notes/file
    Body:
      capture_id        ← from envelope
      delivery_attempt  ← from envelope
      classification    ← full Gemini classification object
      inbox_reason      ← null
      source_message_id ← from fetched capture
      created_at        ← from fetched capture
      model             ← pinned Gemini model name (workflow constant)
      prompt_version    ← pinned prompt version (workflow constant)
    Credential: Second Brain - Writer Service Header
    ↓
Acknowledge Filed
    POST http://capture-service:8000/internal/captures/{{capture_id}}/delivery/acknowledge-filed
    Body:
      delivery_attempt  ← from envelope
      note_path         ← body.note_path from writer-service response
      git_commit_hash   ← body.git_commit_hash from writer-service response (null in SB-114)
```

### Inbox branch (low confidence or inbox_reason present)

```text
Submit to Writer Service
    POST http://writer-service:8001/internal/notes/file
    Body:
      capture_id        ← from envelope
      delivery_attempt  ← from envelope
      classification    ← full Gemini classification object
      inbox_reason      ← ClassificationValidationResponse.inbox_reason slug
      source_message_id ← from fetched capture
      created_at        ← from fetched capture
      model             ← pinned Gemini model name (workflow constant)
      prompt_version    ← pinned prompt version (workflow constant)
    Credential: Second Brain - Writer Service Header
    ↓
Acknowledge Inbox
    POST http://capture-service:8000/internal/captures/{{capture_id}}/delivery/acknowledge-inbox
    Body:
      delivery_attempt  ← from envelope
      note_path         ← body.note_path from writer-service response
      git_commit_hash   ← null
      reason_type       ← inbox_reason slug
```

`writer-service` always receives the full classification payload so it can render the complete frontmatter. It does not call `capture-service` itself — that remains n8n's responsibility.

### New credential

Create in the n8n UI:

| Credential name | Header | Value |
| --- | --- | --- |
| Second Brain - Writer Service Header | `X-Second-Brain-Writer-Token` | same token as `WRITER_SERVICE_TOKEN` |

### Bootstrap upgrade path

The existing `bootstrap-n8n.sh` skips import when a workflow by that name already exists. That behavior leaves the installed `Second Brain - Intake` pointing at `writer-stub`.

Add an upgrade path to `deploy/bootstrap-n8n.sh`:

```text
1. Export all workflow names and IDs from the running container
2. Find the ID of "Second Brain - Intake" if it exists
3. If found:
   a. Sanitize the updated fixture (strip id and versionId from the JSON)
   b. Inject the found ID into the sanitized fixture
   c. Import the fixture in place — n8n updates the existing workflow by ID
   d. Leave the workflow inactive
   e. Print: "Updated Second Brain - Intake — rebind Second Brain - Writer Service Header credential and reactivate"
4. If not found:
   a. Follow the existing import path
```

Do not auto-activate. Do not overwrite credentials. Print the credential rebind instruction every time the intake workflow is upgraded.
