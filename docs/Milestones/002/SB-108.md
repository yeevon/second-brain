# SB-108 — Add the stale-processing-lease reaper

**Branch:** `feature/stale-lease-reaper`

## Objective

Implement one self-scheduling watchdog loop inside `capture-service`.

The watchdog recovers captures whose downstream processing lease expired without a terminal callback.

A successful webhook response does not prove that n8n, Gemini, `writer-service`, Git, or the final callback completed. The reaper must prevent accepted captures from remaining stuck forever in an in-progress state.

Required behavior:

```text
capture forwarded downstream
    ↓
processing lease assigned
    ↓
downstream process crashes or never calls back
    ↓
lease expires
    ↓
stale-lease reaper schedules bounded retry
    ↓
dispatcher forwards capture again
    ↓
eventually:
    terminal callback succeeds
        OR
    retry cap is exceeded
        ↓
    capture marked FAILED visibly
    manual retry remains available
```

## Design decision: separate delivery generations from retry count

Track two counters.

```text
delivery_attempts
    = number of actual downstream delivery claims

retry_attempts
    = number of failed or expired attempts converted into retries
```

Example:

```text
initial outbound claim
    delivery_attempts = 1
    retry_attempts = 0

lease expires
    delivery_attempts = 1
    retry_attempts = 1
    delivery_status = RETRY_WAIT

next outbound claim
    delivery_attempts = 2
    retry_attempts = 1

lease expires again
    delivery_attempts = 2
    retry_attempts = 2
```

The reaper increments only:

```text
retry_attempts
```

The dispatcher increments only:

```text
delivery_attempts
```

Do not increment both counters in the same layer.

## State model

The reaper examines only active downstream-processing states:

```text
FORWARDING
FORWARDED
CLASSIFYING
```

A row is stale when:

```text
processing_lease_until IS NOT NULL
AND
processing_lease_until <= now
```

### Retryable stale lease

When retry capacity remains:

```text
FORWARDING | FORWARDED | CLASSIFYING
    ↓
RETRY_WAIT
```

Update:

```text
retry_attempts = retry_attempts + 1
processing_lease_until = NULL
next_attempt_at = now + capped exponential backoff
last_error = safe stale-lease reason
```

Append:

```text
REQUEUED_STALE_LEASE
```

### Exhausted retry cap

When the incremented retry count exceeds or reaches the configured maximum:

```text
FORWARDING | FORWARDED | CLASSIFYING
    ↓
delivery_status = FAILED
status = FAILED
```

Update:

```text
retry_attempts = retry_attempts + 1
processing_lease_until = NULL
next_attempt_at = NULL
last_error = safe retry-limit reason
```

Append:

```text
RETRY_LIMIT_EXCEEDED
```

Return the capture ID so the caller can send a visible Discord alert **after commit**.

## Backoff calculation

Add:

```python
def calculate_retry_delay_seconds(
    *,
    retry_attempts: int,
    base_delay_seconds: int,
    max_delay_seconds: int,
) -> int:
    delay = base_delay_seconds * (
        2 ** max(retry_attempts - 1, 0)
    )
    return min(delay, max_delay_seconds)
```

Example:

```text
base delay = 10 seconds
maximum delay = 300 seconds
```

| Retry attempt |       Delay |
| ------------- | ----------: |
| 1             |  10 seconds |
| 2             |  20 seconds |
| 3             |  40 seconds |
| 4             |  80 seconds |
| 5             | 160 seconds |
| 6+            | 300 seconds |

Always cap the delay.

## Ledger operation

Add:

```python
reap_expired_processing_leases(
    *,
    now: datetime,
    batch_size: int,
    retry_max_attempts: int,
    retry_base_delay_seconds: int,
    retry_max_delay_seconds: int,
) -> LeaseReaperResult
```

Run the entire mutation through the serialized SQLite runtime.

### Query

Select only a bounded batch:

```sql
SELECT
    capture_id,
    delivery_status,
    delivery_attempts,
    retry_attempts
FROM captures
WHERE delivery_status IN (
    'FORWARDING',
    'FORWARDED',
    'CLASSIFYING'
)
AND processing_lease_until IS NOT NULL
AND processing_lease_until <= ?
ORDER BY processing_lease_until ASC, id ASC
LIMIT ?;
```

### For each stale row

Calculate:

```python
next_retry_attempts = row.retry_attempts + 1
```

Then choose one transition.

### Retry remains available

Condition:

```python
next_retry_attempts < retry_max_attempts
```

Update conditionally:

```sql
UPDATE captures
SET
    delivery_status = 'RETRY_WAIT',
    retry_attempts = ?,
    processing_lease_until = NULL,
    next_attempt_at = ?,
    last_error = ?,
    updated_at = ?
WHERE capture_id = ?
AND delivery_status = ?
AND processing_lease_until IS NOT NULL
AND processing_lease_until <= ?;
```

Use safe `last_error` text:

```text
stale processing lease expired
```

Append event:

```text
REQUEUED_STALE_LEASE
```

Event payload:

```json
{
  "previous_delivery_status": "FORWARDED",
  "delivery_attempts": 1,
  "retry_attempts": 1,
  "next_attempt_at": "2026-06-08T12:34:56+00:00"
}
```

### Retry cap reached

Condition:

```python
next_retry_attempts >= retry_max_attempts
```

Update conditionally:

```sql
UPDATE captures
SET
    status = 'FAILED',
    delivery_status = 'FAILED',
    retry_attempts = ?,
    processing_lease_until = NULL,
    next_attempt_at = NULL,
    last_error = ?,
    updated_at = ?
WHERE capture_id = ?
AND delivery_status = ?
AND processing_lease_until IS NOT NULL
AND processing_lease_until <= ?;
```

Use safe `last_error` text:

```text
retry limit exceeded after repeated stale processing leases
```

Append:

```text
RETRY_LIMIT_EXCEEDED
```

Event payload:

```json
{
  "previous_delivery_status": "CLASSIFYING",
  "delivery_attempts": 5,
  "retry_attempts": 5
}
```

### Conditional-update rule

If a selected row changed before its conditional update executes:

```text
rowcount == 0
    → skip it
```

Do not overwrite a late successful callback.

This protects against:

```text
reaper selects stale row
    ↓
terminal FILED callback commits
    ↓
reaper attempts conditional update
    ↓
conditional update affects zero rows
    ↓
FILED state remains authoritative
```

## Transaction boundary

The reaper ledger operation may contain only:

```text
bounded SELECT
conditional UPDATE statements
capture-event INSERT statements
commit
```

It must not contain:

```text
Discord API call
receipt edit
n8n webhook call
Gemini call
Git command
filesystem note write
sleep
```

Required ordering:

```text
open short SQLite transaction
    ↓
claim and update stale rows
    ↓
append events
    ↓
commit
    ↓
return LeaseReaperResult
    ↓
send Discord receipt edits outside transaction
```

## Reaper loop

Add:

```text
src/secondbrain/reaper.py
```

Implement:

```python
async def run_stale_lease_reaper(
    *,
    settings,
    ledger,
    receipt_client,
) -> None:
    ...
```

Use a self-scheduling sequential loop:

```python
async def run_stale_lease_reaper(
    *,
    settings,
    ledger,
    receipt_client,
) -> None:
    while True:
        await run_stale_lease_reaper_once(
            settings=settings,
            ledger=ledger,
            receipt_client=receipt_client,
        )

        await asyncio.sleep(
            settings.stale_lease_reaper_interval_seconds
        )
```

Do not use:

```text
asyncio.create_task(run_pass())
inside each interval

setInterval

Timer callbacks that can overlap
```

The next sleep begins only after the prior pass and its post-commit alert delivery finish.

## One-pass function

Add:

```python
async def run_stale_lease_reaper_once(
    *,
    settings,
    ledger,
    receipt_client,
) -> LeaseReaperResult:
    ...
```

Flow:

```text
call ledger.reap_expired_processing_leases(...)
    ↓
SQLite transaction commits
    ↓
log bounded result metadata
    ↓
for each requeued capture:
    edit receipt with retry scheduled warning
    ↓
for each permanently failed capture:
    edit receipt with retry exhausted alert
    ↓
return result
```

Receipt-edit failure must:

```text
log error type safely
continue processing remaining captures
```

Do not abort the entire pass because one Discord receipt could not be updated.

## Single-flight guarantee

The sequential loop already prevents self-overlap.

Add an explicit lock so future manual health hooks cannot overlap the scheduled pass:

```python
class StaleLeaseReaper:
    def __init__(...):
        self._lock = asyncio.Lock()

    async def run_once(self) -> LeaseReaperResult:
        if self._lock.locked():
            log_metadata(
                "stale_lease_reaper_overlap_skipped"
            )
            return LeaseReaperResult(
                scanned=0,
                requeued=(),
                failed=(),
            )

        async with self._lock:
            ...
```

A second pass must skip rather than wait indefinitely.

Log:

```text
stale_lease_reaper_overlap_skipped
```

## Application wiring

Start the watchdog exactly once after startup initialization.

Recommended startup order:

```text
initialize SQLite runtime
    ↓
run migrations
    ↓
run Discord startup reconciliation
    ↓
start delivery dispatcher
    ↓
start stale-lease reaper once
    ↓
continue Gateway operation
```

Discord may call `on_ready` more than once after reconnects.

Track:

```python
stale_lease_reaper_task: asyncio.Task | None = None
```

Guard:

```python
if (
    stale_lease_reaper_task is None
    or stale_lease_reaper_task.done()
):
    stale_lease_reaper_task = asyncio.create_task(
        run_stale_lease_reaper(...)
    )
```

Do not start a second watchdog because Discord reconnected.

## Acceptance criteria

SB-108 is complete when:

* [ ] `retry_attempts` is stored separately from actual `delivery_attempts`.
* [ ] Expired `FORWARDING`, `FORWARDED`, and `CLASSIFYING` leases are detected.
