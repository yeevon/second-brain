# SB-105 — Upgrade the SQLite runtime for service operation

**Branch:** `feature/sqlite-service-hardening`

## Objective

Harden the existing SQLite ledger so it can operate safely inside a long-running capture service.

The current MVP repository already uses SQLite as the durable spine. This task upgrades the runtime behavior without changing the ledger’s public responsibilities or moving to a different database.

The service must continue to guarantee:

```text
accepted Discord message
    ↓
durable SQLite commit
    ↓
saved receipt may be sent
```

A note must never be dropped because SQLite was briefly busy. The bot must never claim a note was saved before the transaction committed.

---

## Existing baseline

The current `Ledger` implementation already has:

* One SQLite connection.
* `PRAGMA foreign_keys = ON`.
* A process-local lock around mutations.
* Short transaction scopes using the SQLite connection context manager.
* A bootstrap `migrate()` method that creates tables and indexes.
* Discord-message idempotency through the unique `discord_message_id`.
* Transactional creation of the human-readable `capture_id`.

Preserve those behaviors while upgrading the runtime model.

---

## Design decision

Use **one dedicated SQLite runtime worker thread** with a bounded job queue.

The worker thread owns the SQLite connection for its entire lifetime.

Do **not** create the connection on the main Discord event-loop thread and pass it across threads. Do **not** use `check_same_thread=False` as a shortcut.

All database operations must run through the worker:

```text
Discord handler / classifier worker / status command
    ↓
Ledger repository method
    ↓
SQLiteRuntime.submit(...)
    ↓
bounded job queue
    ↓
single SQLite worker thread
    ↓
single owned SQLite connection
```

This gives the service one serialized database execution path while preserving the existing `Ledger` repository as the application-facing API.

SQLite WAL mode still permits external readers, such as a separate `python -m secondbrain status` process, while the capture service is running.

## Implementation plan

## 1. Add SQLite service configuration

Add environment-backed settings with safe defaults:

```dotenv
SQLITE_BUSY_TIMEOUT_MS=1000
SQLITE_BUSY_RETRY_ATTEMPTS=5
SQLITE_BUSY_RETRY_BASE_DELAY_MS=25
SQLITE_JOB_QUEUE_MAXSIZE=10000
```

Add the corresponding typed values to `Settings`.

Validate:

```text
SQLITE_BUSY_TIMEOUT_MS >= 0
SQLITE_BUSY_RETRY_ATTEMPTS >= 1
SQLITE_BUSY_RETRY_BASE_DELAY_MS >= 0
SQLITE_JOB_QUEUE_MAXSIZE >= 1
```

The retry budget should remain bounded. Do not retry forever.

---

## 2. Add the SQLite runtime worker

Create:

```text
src/secondbrain/sqlite_runtime.py
```

Add a `SQLiteRuntime` class responsible for:

* Starting one dedicated daemon thread.
* Creating the SQLite connection inside that worker thread.
* Applying connection-scoped PRAGMAs.
* Running migrations before accepting normal jobs.
* Executing queued read and write operations.
* Returning results or exceptions through `concurrent.futures.Future`.
* Closing the connection during shutdown.
* Rejecting new work after shutdown starts.

Recommended internal shape:

```python
@dataclass
class DatabaseJob:
    operation: Callable[[sqlite3.Connection], T]
    future: Future[T]
    write: bool
```

The runtime should expose a small API:

```python
class SQLiteRuntime:
    def read(self, operation): ...
    def write(self, operation): ...
    def close(self): ...
```

`read()` and `write()` may block the caller until the job completes. That is acceptable because each database operation must remain short.

The queue must be bounded:

```python
queue.Queue(maxsize=settings.sqlite_job_queue_maxsize)
```

Do not silently discard jobs when the queue fills.

---

## 3. Create the SQLite connection inside the worker thread

The runtime worker must create and configure its own connection:

```python
connection = sqlite3.connect(database_path)
connection.row_factory = sqlite3.Row
```

Apply:

```sql
PRAGMA journal_mode = WAL;
PRAGMA foreign_keys = ON;
PRAGMA busy_timeout = <configured milliseconds>;
```

Verify startup state:

```sql
PRAGMA journal_mode;
PRAGMA foreign_keys;
PRAGMA busy_timeout;
```

Fail startup with a clear exception when:

```text
journal_mode != wal
foreign_keys != 1
busy_timeout does not match configuration
```

The runtime must finish startup configuration and migrations before the bot begins listening for captures.

---

## 4. Add the SQLite version startup check

Use the Python SQLite runtime version:

```python
sqlite3.sqlite_version_info
```

Define:

```python
MIN_SQLITE_VERSION = (3, 24, 0)
```

SQLite `3.24.0` is the minimum because the current ledger uses UPSERT syntax:

```sql
ON CONFLICT(key) DO UPDATE
```

At startup:

```python
if sqlite3.sqlite_version_info < MIN_SQLITE_VERSION:
    raise RuntimeError(...)
```

The error must report:

```text
installed SQLite version
minimum required version
```

Do not start the Discord listener when the SQLite runtime is unsupported.

Use `CREATE TABLE IF NOT EXISTS` and `CREATE INDEX IF NOT EXISTS` so existing MVP databases can be adopted without data loss.

## 6. Add bounded retry handling for transient lock contention

Add a dedicated exception:

```python
class SQLiteBusyError(RuntimeError):
    pass
```

Retry only transient SQLite lock failures.

Recognize:

```text
SQLITE_BUSY
SQLITE_LOCKED
```

Do not retry arbitrary `sqlite3.OperationalError` failures such as malformed SQL or missing tables.

Recommended retry schedule:

```text
attempt 1: execute immediately
attempt 2: wait 25 ms
attempt 3: wait 50 ms
attempt 4: wait 100 ms
attempt 5: wait 200 ms
```

Apply retries around the entire short database operation.

For writes:

```text
begin transaction
    ↓
execute repository mutation
    ↓
append capture event when required
    ↓
commit
```

When a transient lock error occurs:

```text
rollback
    ↓
wait for bounded backoff
    ↓
retry complete operation
```

When the retry budget is exhausted:

```text
raise SQLiteBusyError
```

Do not claim the message was saved. Do not send a saved receipt.

Discord startup catch-up remains the durability fallback when a live capture could not commit.

## 7. Route all ledger operations through the runtime

Refactor `Ledger` so it owns a `SQLiteRuntime`, not a directly accessed connection.

Before:

```python
self._connection = sqlite3.connect(...)
self._lock = Lock()
```

After:

```python
self._runtime = SQLiteRuntime(...)
```

Repository methods should submit short closures to the runtime.

Example pattern:

```python
def insert_accepted_capture(...):
    return self._runtime.write(
        lambda connection: self._insert_accepted_capture(
            connection,
            ...
        )
    )
```

Move direct SQL into private methods that accept the worker-owned connection:

```python
def _insert_accepted_capture(
    self,
    connection: sqlite3.Connection,
    ...
) -> InsertResult:
    ...
```

Update private helpers similarly:

```text
_get_by_discord_message_id(connection, ...)
_get_by_capture_id(connection, ...)
_next_capture_id(connection, ...)
_append_event(connection, ...)
```

Route reads through the runtime too:

```text
get_capture
enqueueable_capture_ids
captures_by_status
status_counts
total_captures
last_successful_vault_write
get_system_state
```

Route mutations through `runtime.write()`:

```text
insert_accepted_capture
insert_sensitive_rejection
set_receipt_message_id
mark_classifying
reset_classifying_to_received
update_capture
set_system_state
migration execution
```

Remove the old repository-level `Lock` once all operations use the worker-owned connection.

Do not retain two competing serialization mechanisms.

## 8. Keep every write transaction short

The SQLite runtime job may contain:

```text
SQL query
row mapping
capture-event append
commit
```

The SQLite runtime job must never contain:

```text
Discord API call
Discord receipt send
Discord receipt edit
Gemini request
n8n webhook call
Git command
filesystem note write
sleep unrelated to SQLite retry backoff
network request of any kind
```

The existing application order must remain:

```text
ledger.insert_accepted_capture(...)
    ↓
SQLite commit returns successfully
    ↓
send_saved_receipt(...)
    ↓
queue capture ID
```

For final processing:

```text
vault write completes
    ↓
ledger.update_capture(...)
    ↓
SQLite commit returns successfully
    ↓
edit final Discord receipt
```

Document this as an invariant in:

```text
docs/ARCHITECTURE.md
```

## 9. Add safe shutdown

Update:

```text
Ledger.close()
```

so it shuts down the runtime worker and closes the SQLite connection.

Update the listener startup path:

```python
ledger = Ledger(...)

try:
    client.run(...)
finally:
    ledger.close()
```

The status command must continue closing its ledger instance after printing the report.

Runtime shutdown should:

```text
stop accepting new jobs
    ↓
finish queued jobs
    ↓
send worker sentinel
    ↓
close SQLite connection
    ↓
join worker thread
```

Do not abandon queued writes during a normal shutdown.
