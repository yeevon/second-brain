# SB-107 — Add delivery attempts, leases, and capped retry state

**Branch:** `feature/delivery-leases`

## Objective

Extend the durable SQLite ledger so `capture-service` can safely deliver accepted captures to downstream processing without assuming that webhook acceptance proves workflow completion.

A successful HTTP response from the n8n intake webhook means only:

```text
n8n accepted the request
```

It does **not** prove:

```text
classification completed
Markdown note was filed
Git push completed
final callback reached capture-service
```

The system must recover automatically when downstream processing stops after webhook acceptance.

The required behavior is:

```text
accepted Discord message
    ↓
persist immutable raw capture
    ↓
send immediate saved receipt
    ↓
forward capture_id to downstream workflow
    ↓
assign processing lease
    ↓
receive terminal callback
        OR
lease expires
    ↓
retry safely with capped backoff
        OR
mark FAILED visibly after retry cap
```

---

## Dependency

Implement after:

```text
SB-105 — Upgrade the SQLite runtime for service operation
```

SB-107 assumes:

* Versioned schema migrations exist.
* All SQLite mutations run through the serialized SQLite runtime.
* WAL mode, `foreign_keys`, and `busy_timeout` are configured.
* No network call occurs while a SQLite write transaction is open.

SB-106 periodic reconciliation may merge before or after this task. Any capture recovered by startup or periodic reconciliation must enter the same pending-delivery state as a live Gateway capture.

## Existing baseline

The current MVP ledger already stores:

```text
status
last_error
updated_at
```

The production architecture additionally requires:

```text
delivery_attempts
processing_lease_until
next_attempt_at
```

Preserve the existing `last_error` column. Do not add a duplicate field.

The MVP currently uses one `status` column for local classifier progress and terminal note state. Production delivery retry behavior must not overload that field further.

## Design decision: separate note lifecycle from delivery state

Keep:

```text
status
```

as the note lifecycle status.

Add:

```text
delivery_status
```

as the downstream-delivery status.

This separation makes the record answer two different questions cleanly:

```text
status
    = what happened to the note?

delivery_status
    = where is downstream delivery and processing?
```

### Note lifecycle statuses

Use:

```text
RECEIVED
FILED
INBOX
REJECTED_SENSITIVE
FAILED
```

Reserve for later correction work:

```text
NEEDS_CLARIFICATION
CORRECTED
```

### Delivery statuses

Add:

```text
NOT_APPLICABLE
PENDING_FORWARD
FORWARDING
FORWARDED
CLASSIFYING
RETRY_WAIT
COMPLETE
FAILED
```

### Meaning

| `delivery_status` | Meaning                                                                                                   |
| ----------------- | --------------------------------------------------------------------------------------------------------- |
| `NOT_APPLICABLE`  | Capture is not eligible for downstream delivery, such as a sensitive rejection.                           |
| `PENDING_FORWARD` | Accepted capture is durable but has not been delivered to n8n.                                            |
| `FORWARDING`      | `capture-service` claimed the row and is currently making the outbound webhook request.                   |
| `FORWARDED`       | n8n accepted the request; downstream completion callback has not arrived yet.                             |
| `CLASSIFYING`     | n8n reported that classifier processing began.                                                            |
| `RETRY_WAIT`      | Prior attempt failed or became stale; row is waiting for `next_attempt_at`.                               |
| `COMPLETE`        | Terminal callback arrived successfully: `FILED`, `INBOX`, or a later successful lifecycle terminal state. |
| `FAILED`          | Retry cap was exceeded or an explicitly non-retryable downstream failure occurred.                        |

`FORWARDING` is required even though it was not in the original abbreviated list. Without it, a service crash during the outbound HTTP request leaves no explicit state for the reaper to recover.

```sql

    ON captures(delivery_status, next_attempt_at);

    ON captures(delivery_status, processing_lease_until);
```

### Migrate existing rows safely

Normalize existing MVP rows:

```text
status = REJECTED_SENSITIVE
    → delivery_status = NOT_APPLICABLE

status = FILED
    → delivery_status = COMPLETE

status = INBOX
    → delivery_status = COMPLETE

status = FAILED
    → delivery_status = FAILED

status = RECEIVED
    → delivery_status = PENDING_FORWARD

status = CLASSIFYING
    → status = RECEIVED
    → delivery_status = PENDING_FORWARD
```

Existing MVP `CLASSIFYING` rows must be reset to a retryable state during migration because they do not represent a valid production n8n lease.

Run the service migration while the old local process is stopped.

## Insert behavior

### Accepted capture

A normal accepted Discord message must be inserted as:

```text
status = RECEIVED
delivery_status = PENDING_FORWARD
delivery_attempts = 0
processing_lease_until = NULL
next_attempt_at = NULL
```

Then:

```text
commit SQLite transaction
    ↓
send immediate saved receipt
    ↓
wake delivery dispatcher
```

### Sensitive rejection

A likely-secret rejection must be inserted as:

```text
status = REJECTED_SENSITIVE
delivery_status = NOT_APPLICABLE
delivery_attempts = 0
processing_lease_until = NULL
next_attempt_at = NULL
```

Sensitive rejections must never enter the downstream delivery queue.

### Reconciliation recovery

Captures inserted by startup or periodic Discord reconciliation use the same accepted insert path:

```text
status = RECEIVED
delivery_status = PENDING_FORWARD
```

Do not create a special reconciliation delivery state.

## Backoff policy

Use capped exponential backoff:

```python
def calculate_retry_delay_seconds(
    *,
    delivery_attempts: int,
    base_delay_seconds: int,
    max_delay_seconds: int,
) -> int:
    delay = base_delay_seconds * (2 ** max(delivery_attempts - 1, 0))
    return min(delay, max_delay_seconds)
```

Example with:

```text
base = 10 seconds
max = 300 seconds
```

| Attempt | Retry delay |
| ------- | ----------: |
| 1       |  10 seconds |
| 2       |  20 seconds |
| 3       |  40 seconds |
| 4       |  80 seconds |
| 5       | 160 seconds |

Do not retry forever.

## Attempt correlation

Use:

```text
delivery_attempts
```

as both the retry count and downstream attempt generation.

When `capture-service` forwards work, send:

```json
{
  "capture_id": "SB-20260607-0042",
  "delivery_attempt": 2
}
```

`capture_id` remains the idempotency key.

`delivery_attempt` is correlation metadata so late callbacks from an older workflow execution cannot overwrite newer state.

All downstream progress and terminal callbacks must include:

```text
capture_id
delivery_attempt
```

Reject or safely ignore callbacks whose attempt number does not match the capture’s current `delivery_attempts`.

Example:

```text
attempt 1 forwarded
    ↓
lease expires
    ↓
attempt 2 claimed
    ↓
late CLASSIFYING callback arrives from attempt 1
    ↓
ignore stale callback
```

Without attempt correlation, leases create a real stale-callback race.

## Find expired leases

```python
requeue_expired_leases(
    *,
    now: datetime,
    batch_size: int,
) -> LeaseReaperResult
```

Eligible rows:

```text
delivery_status IN (
    FORWARDING,
    FORWARDED,
    CLASSIFYING
)

AND processing_lease_until <= now
```

For each expired row:

```text
attempts remain
    → RETRY_WAIT
    → clear lease
    → set next_attempt_at
    → append REQUEUED_STALE_LEASE

retry cap exceeded
    → status = FAILED
    → delivery_status = FAILED
    → clear lease
    → append RETRY_LIMIT_EXCEEDED
```

Return capture IDs requiring visible failure alerts.

Use a bounded batch.

Perform all Discord alerts only after the SQLite transaction commits.

## Important attempt-count rule

Increment:

```text
delivery_attempts
```

only when a real downstream forwarding attempt is claimed.

Do **not** increment it merely because the reaper noticed an expired lease.

Correct sequence:

```text
expired lease found
    ↓
move to RETRY_WAIT
    ↓
wait until next_attempt_at
    ↓
dispatcher claims row
    ↓
increment delivery_attempts
    ↓
make next HTTP attempt
```

This prevents repeated reaper scans from consuming retry attempts without actually attempting delivery.

## Documentation

Update:

```text
docs/ARCHITECTURE.md
```

Document:

### Two-dimensional state

```text
status
    = note lifecycle

delivery_status
    = downstream transport and processing lease state
```

### Persist-before-forward invariant

```text
SQLite commit
    ↓
immediate saved receipt
    ↓
delivery claim
    ↓
HTTP webhook call
```

### Lease recovery

```text
FORWARDING, FORWARDED, or CLASSIFYING lease expires
    ↓
RETRY_WAIT
    ↓
bounded exponential backoff
    ↓
next dispatcher claim
    ↓
new delivery attempt
```

### Retry cap

```text
delivery_attempts >= DELIVERY_MAX_ATTEMPTS
    ↓
status = FAILED
delivery_status = FAILED
    ↓
visible Discord alert
```

### Callback correlation

```text
capture_id
    = stable idempotency key

delivery_attempt
    = workflow-attempt generation

stale callback
    = ignored safely
```

## Acceptance criteria

SB-107 is complete when:

* [ ] `delivery_status` is stored separately from note lifecycle `status`.
* [ ] `delivery_attempts` is persisted.
* [ ] `processing_lease_until` is persisted.
* [ ] `next_attempt_at` is persisted.
* [ ] Existing `last_error` is reused.
* [ ] Accepted captures begin as `PENDING_FORWARD`.
* [ ] Sensitive rejections begin as `NOT_APPLICABLE`.
* [ ] Delivery claims are bounded and atomic.
* [ ] Delivery claims increment `delivery_attempts` exactly once.
* [ ] HTTP delivery occurs only after the claim transaction commits.
* [ ] Webhook acceptance moves the row into `FORWARDED` with a processing lease.
* [ ] Classifier progress callbacks move the row into `CLASSIFYING` and renew the lease.
* [ ] Filed and Inbox callbacks clear the lease and mark delivery `COMPLETE`.
* [ ] Retryable failures move the row into `RETRY_WAIT`.
* [ ] Retry backoff is capped.
* [ ] The stale-lease reaper is bounded and single-flight.
* [ ] Expired leases become retryable rather than remaining stuck.
* [ ] The reaper does not increment attempts unless a new delivery is actually claimed.
* [ ] Retry exhaustion moves the capture into terminal `FAILED`.
* [ ] Retry exhaustion sends a visible safe Discord failure alert.
* [ ] Late callbacks from older attempts cannot regress current state.
* [ ] Duplicate callbacks remain idempotent.
* [ ] Raw accepted text remains immutable.
* [ ] No network request occurs inside a SQLite write transaction.
* [ ] Existing MVP tests still pass.
* [ ] A simulated downstream crash after webhook acceptance leaves the note retryable rather than stuck forever.

Run the local service:

```bash
uv run python -m secondbrain run
```

Then manually verify:

```text
1. Start capture-service in downstream mode.
2. Send one normal Discord brain dump.
3. Confirm the saved receipt appears immediately after persistence.
4. Simulate a downstream adapter that accepts the webhook but never calls back.
5. Wait for the configured processing lease to expire.
6. Confirm status reports RETRY_WAIT.
7. Wait until next_attempt_at.
8. Confirm a new delivery attempt occurs.
9. Run:
   uv run python -m secondbrain status
10. Confirm delivery attempts increased and the raw capture remains safe.
```
