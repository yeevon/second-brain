# SB-109 — Expand the operational status command

**Branch:** `feature/ops-status`

## Objective

Expand the existing operator command:

```bash
uv run python -m secondbrain status
```

so backlog and service health can be diagnosed without opening SQLite manually.

The command must answer:

```text
Is capture-service alive?
Are captures accumulating?
Are notes reaching FILED state?
Are uncertain notes accumulating in Inbox?
Are retryable captures waiting?
Are active downstream leases stale?
Did Discord reconciliation succeed recently?
```

Required report fields:

```text
captures received today
captures filed today
captures in inbox
captures failed
captures waiting for retry
stale leases
last successful reconciliation
capture-service health
```

Retain useful existing MVP fields such as ledger path, vault path, total captures, sensitive rejections, last reconciled Discord message ID, and last successful vault write.

## Design decisions

## 1. Keep status read-only

The status command must not instantiate the normal mutable `Ledger` runtime.

Do not:

```text
run migrations
create a missing database
start a SQLite worker
update system_state
modify capture rows
```

The operator command should open the SQLite file read-only:

```python
sqlite3.connect(
    f"file:{ledger_path}?mode=ro",
    uri=True,
)
```

Apply:

```sql
PRAGMA query_only = ON;
PRAGMA busy_timeout = 1000;
```

WAL mode allows this read-only operational query while `capture-service` is running.

If the database file does not exist or cannot be queried, report a clear operational failure. Do not silently create an empty database.

---

## 2. Add status-only configuration

The current status command should not require:

```text
Discord bot token
Discord guild ID
Discord channel ID
Gemini API key
vault write configuration
```

An operator must be able to inspect health precisely when runtime configuration is broken.

Add a small configuration object:

```python
@dataclass(frozen=True)
class StatusSettings:
    ledger_path: Path
    vault_path: Path | None
    status_timezone: str
    capture_service_health_stale_after_seconds: int
```

Load only the fields needed by the operator report.

The runtime `Settings` object remains unchanged for the service process.

---

## 3. Use explicit operational date boundaries

Add:

```dotenv
STATUS_TIMEZONE=UTC
```

Allow a local deployment override such as:

```dotenv
STATUS_TIMEZONE=America/Chicago
```

Use Python `zoneinfo.ZoneInfo`.

Calculate:

```text
local start of today
local start of tomorrow
```

then convert both timestamps to UTC before querying SQLite.

This makes:

```text
captures received today
captures filed today
```

predictable and explicit.

Print the timezone in the report.

---

## 4. Count received and filed captures differently

### Captures received today

Count rows by immutable intake timestamp:

```sql
SELECT COUNT(*)
FROM captures
WHERE received_at >= ?
AND received_at < ?;
```

This includes:

```text
accepted captures
sensitive rejection records
```

because both represent Discord messages received by `capture-service`.

### Captures filed today

Count successful filing events:

```sql
SELECT COUNT(*)
FROM capture_events
WHERE event_type = 'CAPTURE_FILED'
AND created_at >= ?
AND created_at < ?;
```

Do not use:

```text
captures.updated_at
```

A later receipt replacement, correction, retry, or metadata update may change `updated_at` and incorrectly make an old note appear newly filed.

### Current backlog fields

Use current row state:

```sql
captures in inbox
    WHERE status = 'INBOX'

captures failed
    WHERE status = 'FAILED'

captures waiting for retry
    WHERE delivery_status = 'RETRY_WAIT'

stale leases
    WHERE delivery_status IN (
        'FORWARDING',
        'FORWARDED',
        'CLASSIFYING'
    )
    AND processing_lease_until IS NOT NULL
    AND processing_lease_until <= ?
```

## 5. Add a capture-service heartbeat

The status command needs evidence that the long-running service is alive.

Add system-state keys:

```text
capture_service_instance_id
capture_service_state
capture_service_started_at
capture_service_last_heartbeat_at
capture_service_stopped_at
```

Use states:

```text
STARTING
RUNNING
STOPPED
```

### Runtime startup

When `capture-service` starts:

```text
generate instance UUID
    ↓
write instance ID
write state = STARTING
write started_at
clear stopped_at
```

After startup initialization completes:

```text
Discord client ready
startup reconciliation complete
classifier or delivery workers started
periodic reconciliation started
stale-lease reaper started
    ↓
write state = RUNNING
write heartbeat timestamp
```

### Heartbeat loop

Add:

```dotenv
CAPTURE_SERVICE_HEARTBEAT_INTERVAL_SECONDS=15
CAPTURE_SERVICE_HEALTH_STALE_AFTER_SECONDS=60
```

Validate:

```text
CAPTURE_SERVICE_HEARTBEAT_INTERVAL_SECONDS >= 1
CAPTURE_SERVICE_HEALTH_STALE_AFTER_SECONDS
    > CAPTURE_SERVICE_HEARTBEAT_INTERVAL_SECONDS
```

Implement one self-scheduling loop:

```python
async def run_capture_service_heartbeat(
    *,
    ledger,
    instance_id: str,
    interval_seconds: int,
) -> None:
    while True:
        ledger.record_capture_service_heartbeat(
            instance_id=instance_id,
            now=datetime.now(UTC),
        )

        await asyncio.sleep(interval_seconds)
```

The heartbeat write must be a short SQLite mutation.

Do not perform network calls inside the heartbeat transaction.

Start the heartbeat task exactly once. Discord may invoke `on_ready` multiple times after reconnecting.

### Graceful shutdown

On a normal shutdown:

```text
cancel heartbeat task
    ↓
write state = STOPPED
write stopped_at
    ↓
close ledger runtime
```

A hard crash will not write `STOPPED`. The stale heartbeat reveals the failure.

## 6. Add atomic system-state helpers

Extend the ledger repository with:

```python
set_system_states(
    values: dict[str, str],
) -> None
```

Use one short transaction.

Add:

```python
record_capture_service_start(
    *,
    instance_id: str,
    now: datetime,
) -> None

record_capture_service_ready(
    *,
    instance_id: str,
    now: datetime,
) -> None

record_capture_service_heartbeat(
    *,
    instance_id: str,
    now: datetime,
) -> bool

record_capture_service_stop(
    *,
    instance_id: str,
    now: datetime,
) -> bool
```

Heartbeat and shutdown updates must be conditional on:

```text
capture_service_instance_id = current instance ID
```

This prevents an older shutting-down process from overwriting the state of a newly started replacement instance.

## 9. Calculate capture-service health

Add:

```python
def calculate_capture_service_health(
    *,
    service_state: str | None,
    last_heartbeat_at: datetime | None,
    now: datetime,
    stale_after_seconds: int,
) -> str:
    ...
```

Return one of:

```text
HEALTHY
STARTING
STOPPED
STALE
UNKNOWN
```

Rules:

```text
service_state = RUNNING
and heartbeat exists
and heartbeat age <= stale threshold
    → HEALTHY

service_state = STARTING
and heartbeat exists
and heartbeat age <= stale threshold
    → STARTING

service_state = STOPPED
    → STOPPED

service_state is RUNNING or STARTING
and heartbeat missing or too old
    → STALE

missing service state
    → UNKNOWN
```

Do not define service health based on backlog counts.

Keep the concerns separate:

```text
capture-service health
    = is the service process alive?

backlog fields
    = is downstream work flowing normally?
```

A running service may be healthy while downstream n8n or writer processing is failing. The retry and stale-lease counts expose that condition.

## 11. Update CLI wiring

Keep:

```bash
uv run python -m secondbrain status
```

Replace the current mutable-ledger status path with:

```python
def run_status() -> int:
    settings = StatusSettings.from_env()

    try:
        snapshot = read_operational_status(
            settings=settings,
        )
    except OperationalStatusUnavailable as exc:
        print(
            "Second Brain operational status unavailable"
        )
        print(f"reason: {exc.safe_reason}")
        return 2

    print(format_operational_status(snapshot))

    if (
        snapshot.capture_service_health
        in {"STALE", "UNKNOWN"}
        or snapshot.stale_leases > 0
        or snapshot.captures_failed > 0
    ):
        return 1

    return 0
```

### Exit codes

Use:

```text
0
    = report generated and no urgent operator condition detected

1
    = report generated, but operator attention is required

2
    = status could not be read safely
```

Conditions requiring operator attention:

```text
capture-service health = STALE
capture-service health = UNKNOWN
stale leases > 0
captures failed > 0
```

Do not return a nonzero exit merely because Inbox contains notes. Inbox requires review, but it is not a service failure.

## Acceptance criteria

SB-109 is complete when:

* [ ] The existing `status` command still runs as `python -m secondbrain status`.
* [ ] The status command does not require Discord or Gemini credentials.
* [ ] The status command opens SQLite read-only.
* [ ] The status command does not create a missing database.
* [ ] The status command does not run migrations.
* [ ] The status command reports an explicit timezone.
* [ ] Captures received today are calculated from `received_at`.
* [ ] Captures filed today are calculated from `CAPTURE_FILED` events.
* [ ] Current Inbox backlog is reported.
* [ ] Current failed backlog is reported.
* [ ] Captures waiting for retry are reported.
* [ ] Stale active leases are reported.
* [ ] Last reconciled Discord message ID is reported.
* [ ] Last successful reconciliation timestamp is reported.
* [ ] Capture-service writes a periodic heartbeat.
* [ ] Capture-service health is derived from heartbeat freshness.
* [ ] An older service instance cannot overwrite a newer instance’s heartbeat or shutdown state.
* [ ] Status output remains metadata-only.
* [ ] Status returns a nonzero exit code for stale service health, stale leases, failed backlog, or unavailable SQLite.
* [ ] Existing MVP and Milestone 2 tests still pass.
* [ ] Retry backlog health can be diagnosed without opening SQLite manually.
