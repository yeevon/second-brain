# SB-106 — Add periodic Discord reconciliation

**Branch:** `feature/periodic-reconciliation`

## Objective

Add a bounded periodic Discord-history scan so a rare dropped Gateway event is recovered without requiring an application restart.

The MVP already performs startup reconciliation. This task extends the existing reconciliation path into a recurring safety net.

The system must preserve the invariant:

```text
Gateway event received normally
    OR
Gateway event missed and found during reconciliation
    ↓
one durable SQLite capture row
    ↓
at most one classification job
    ↓
at most one Markdown note
```

A missed Gateway event must not disappear permanently. A message seen through both paths must not create duplicate rows, receipts, queue work, or notes.

---

## Existing baseline

The current implementation already has:

* A stored Discord reconciliation high-water mark:

```text
last_reconciled_discord_message_id
```

* A bounded startup history scan.
* Oldest-first processing.
* The normal allowlist filters during reconciliation.
* High-water advancement after each durably handled or safely ignored message.
* Startup recovery of unfinished `RECEIVED` and stale `CLASSIFYING` rows.
* A unique SQLite constraint on `discord_message_id`.
* Duplicate handling that returns the existing capture instead of creating a new row.
* A capture handler that can insert without immediately enqueueing during startup reconciliation.

Preserve those behaviors.

---

## Critical marker rule

The stored high-water mark is a **history-scan cursor**, not a “latest Gateway event observed” value.

Only an ordered Discord-history scan may advance:

```text
last_reconciled_discord_message_id
```

Do not advance it directly from the live Gateway handler.

### Why

Assume Discord history contains:

```text
1001  dropped Gateway event
1002  delivered Gateway event
```

If the live handler stores message `1002` and advances the marker to `1002`, the next reconciliation scan requests:

```text
history after 1002
```

Message `1001` is skipped permanently.

Instead:

```text
live Gateway receives 1002
    ↓
store 1002 normally
    ↓
do not move reconciliation marker

periodic history scan starts after prior marker
    ↓
scan 1001
    ↓
store and enqueue missing capture
    ↓
scan 1002
    ↓
detect existing capture and skip duplicate side effects
    ↓
advance marker through 1002
```

The unique `discord_message_id` constraint remains the final idempotency boundary.

---

## Design decision

Reuse the existing reconciliation function for both:

```text
startup reconciliation
periodic reconciliation
```

Do not create a second Discord-history implementation.

Refactor the function so the caller supplies:

```text
scan mode
scan limit
capture handler
```

Use one periodic loop with one `asyncio.Lock`.

The loop must not overlap itself. If one reconciliation pass is still running when the next interval arrives, do not start another pass.

* Discord polling as the primary ingestion path.
* Unbounded history scans.
* Multiple overlapping reconciliation loops.
* A second database table solely for metrics.

* Aggressive retry loops that hammer the Discord API.

## Implementation plan

## 1. Add periodic reconciliation configuration

Add:

```dotenv
PERIODIC_RECONCILE_INTERVAL_SECONDS=60
PERIODIC_RECONCILE_LIMIT=100
```

Keep the existing startup limit separate:

```dotenv
STARTUP_RECONCILE_LIMIT=1000
```

The settings control different behavior:

```text
STARTUP_RECONCILE_LIMIT
    = maximum messages scanned during application startup

PERIODIC_RECONCILE_LIMIT
    = maximum messages scanned during one recurring safety-net pass

PERIODIC_RECONCILE_INTERVAL_SECONDS
    = delay between recurring passes
```

Add typed values to `Settings`.

Validate:

```text
PERIODIC_RECONCILE_INTERVAL_SECONDS >= 1
PERIODIC_RECONCILE_LIMIT >= 1
```

A sixty-second periodic interval is appropriate for a personal brain-dump channel. This is a recovery safety net, not the primary capture path.

## 2. Stop advancing the reconciliation marker from live Gateway events

The live handler currently supports:

```text
advance_reconcile_marker
```

Remove marker advancement from the live Gateway path.

The normal Discord handler should:

```text
persist accepted or rejected row
send receipt when applicable
enqueue new accepted captures
return
```

It must not call:

```text
ledger.set_system_state(
    LAST_RECONCILED_MESSAGE_ID,
    ...
)
```

The reconciliation function remains responsible for moving the marker after processing messages in Discord-history order.

Remove the `advance_reconcile_marker` option if it no longer has a valid caller. Keeping an unused option creates ambiguity around marker ownership.

---

## 3. Generalize the reconciliation result

Extend the result object.

Recommended shape:

```python
@dataclass(frozen=True)
class ReconcileResult:
    mode: str
    seen: int
    handled: int
    recovered: int
    duplicates: int
    ignored: int
    limit_exceeded: bool
    warning: str | None
    high_water_message_id: str | None
```

Definitions:

```text
seen
    = messages returned by the bounded history scan and processed

handled
    = allowlisted messages passed into the capture handler

recovered
    = allowlisted messages that created a new durable capture row

duplicates
    = allowlisted messages already represented by a SQLite capture row

ignored
    = bot, webhook, wrong-user, wrong-channel, or otherwise filtered messages

limit_exceeded
    = Discord returned more messages than the configured scan limit

high_water_message_id
    = final successfully reconciled Discord message ID
```

Do not count an existing live-captured message as recovered.

---

## 4. Return a capture disposition from the shared handler

The existing handler already knows whether an insert created a new row:

```text
InsertResult.created
```

Return that result to the reconciliation caller.

Recommended shape:

```python
@dataclass(frozen=True)
class CaptureDisposition:
    capture_id: str
    created: bool
    status: str
    queued: bool
```

The handler should return a disposition for:

```text
new accepted capture
duplicate accepted capture
new sensitive rejection
duplicate sensitive rejection
```

This allows reconciliation to count:

```text
recovered
duplicates
```

without querying SQLite a second time.

The live Gateway path may ignore the returned value.

## 5. Refactor the history scanner into one generic bounded function

Update:

```python
reconcile_discord_history(...)
```

Recommended signature:

```python
async def reconcile_discord_history(
    *,
    client,
    settings,
    ledger,
    handle_capture,
    mode: str,
    scan_limit: int,
) -> ReconcileResult:
    ...
```

Use:

```text
startup
periodic
```

as valid modes.

Continue fetching:

```text
scan_limit + 1
```

messages so overflow is visible.

Continue processing:

```text
oldest_first=True
```

For each returned message:

```text
filtered out safely
    ↓
count ignored
    ↓
advance marker

allowlisted message
    ↓
await shared capture handler
    ↓
count recovered or duplicate
    ↓
advance marker
```

Do not advance the marker before the message reaches a durable disposition.

If the handler raises before the SQLite commit, allow the exception to propagate. The marker for that message must remain unchanged so a later pass retries it.

## 6. Make overflow warnings mode-aware

Replace startup-specific text such as:

```text
startup reconcile limit reached
```

with mode-aware text.

Recommended structured warning:

```text
periodic reconciliation reached scan limit of 100 messages;
backlog remains and the next scan will continue
```

For startup:

```text
startup reconciliation reached scan limit of 1000 messages;
backlog remains and periodic reconciliation will continue recovery
```

The function should:

```text
process first scan_limit messages
advance high-water mark through that processed batch
return limit_exceeded = true
return warning text
```

The next periodic scan resumes after the stored marker.

Do not discard the remaining backlog.

## 9. Add safe visible Discord warnings

Add a warning helper in `reconcile.py` or `receipts.py`.

Recommended visible messages:

### Periodic reconciliation failure

```text
⚠️ Second Brain reconciliation failed.
Some Discord captures may be delayed.
The next scheduled scan will retry automatically.
```

### Periodic scan limit reached

```text
⚠️ Second Brain reconciliation reached its scan limit.
A backlog remains. Recovery will continue during the next scheduled scan.
```

Send warnings to the configured Discord capture channel.

The warning itself is bot-authored and will be ignored by the normal filter during later history scans.

Rules:

```text
warning send succeeds
    ↓
continue normal loop

warning send fails
    ↓
log warning delivery failure by error type
    ↓
continue normal loop
```

Never let a warning-delivery failure terminate periodic reconciliation.

Do not include exception details or message content in Discord warnings.

## 10. Add the periodic reconciliation loop

Add:

```python
async def run_periodic_reconciliation(
    *,
    client,
    settings,
    ledger,
    handle_capture,
) -> None:
    ...
```

Use one lock:

```python
scan_lock = asyncio.Lock()
```

Recommended loop:

```text
sleep configured interval
    ↓
acquire scan lock
    ↓
run one bounded periodic history scan
    ↓
persist metrics
    ↓
emit structured logs
    ↓
send safe warning if needed
    ↓
release lock
    ↓
repeat
```

Catch exceptions around each scan:

```text
scan fails
    ↓
increment failure metric
    ↓
log safe error type
    ↓
send safe Discord warning
    ↓
sleep until next interval
    ↓
retry
```

Do not exit the loop after one failed scan.

Do not launch a second scan while the first is still active.

A simple sequential loop already prevents self-overlap. Keep the lock as an explicit boundary so future manual-trigger or health-check hooks cannot accidentally overlap the scheduled scanner.

## 11. Wire startup and periodic handlers separately

Use three shared-handler configurations:

```text
live Gateway handler
    enqueue new captures: yes

startup reconciliation handler
    enqueue new captures: no

periodic reconciliation handler
    enqueue new captures: yes
```

### Live handler

```text
persist
receipt
enqueue
do not advance reconciliation marker
```

### Startup handler

```text
persist recovered rows
receipt
do not enqueue immediately
```

After startup reconciliation completes:

```text
read all unfinished RECEIVED and stale CLASSIFYING rows
    ↓
start classifier worker
    ↓
enqueue unfinished rows once
```

### Periodic handler

```text
persist newly recovered row
receipt
enqueue immediately
```

Duplicates must not resend receipts or enqueue duplicate jobs.

## 12. Start the periodic task exactly once

In the Discord `on_ready` startup flow:

```text
run startup reconciliation
    ↓
start classifier worker once
    ↓
enqueue startup recovery rows
    ↓
start periodic reconciliation task once
```

Track task state:

```python
periodic_reconcile_task: asyncio.Task | None = None
```

Discord may invoke `on_ready` again after reconnecting. Do not create duplicate periodic loops.

Recommended guard:

```python
if periodic_reconcile_task is None or periodic_reconcile_task.done():
    periodic_reconcile_task = asyncio.create_task(
        run_periodic_reconciliation(...)
    )
```

Do not restart a healthy running task merely because Discord reconnects.

* [ ] Startup reconciliation still works.
* [ ] Periodic reconciliation runs on a configurable interval.
