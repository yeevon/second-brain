# SB-118 — Implement targeted fix: corrections

**Branch:** `feature/fix-corrections`

Support correction via reply-to-receipt or explicit capture ID only:

```
reply to a receipt:
fix: this belongs under Learning, not Projects.

or:
fix SB-20260607-0042: this belongs under Learning.
```

Reject an unthreaded standalone message:

```
fix: move this to Learning
```

Do not guess the most-recent capture. The architecture explicitly requires unambiguous correction targeting.

---

## Service boundaries

- `capture-service` resolves the correction target from the reply thread or explicit capture ID. n8n does not read SQLite directly.
- `writer-service` performs the note update or `git mv` once capture-service has resolved and authorized the target.
- `capture-service` records `old_note_path`, `new_note_path`, and `git_commit_hash` on the correction event.
- Correction history is append-only. No record is modified in place.

## Acceptance criteria

- A reply-to-receipt correction moves the note and updates the receipt.
- An explicit `fix SB-YYYYMMDD-NNNN:` correction works without an open thread.
- An unthreaded bare `fix:` message is rejected with a clear error.
- A second correction after a previous move targets the current note path and never creates a duplicate.
- `capture_id` is retained on the moved note.
- `old_note_path`, `new_note_path`, and `git_commit_hash` are stored on the correction record.
- Correction history is append-only — no prior correction record is mutated.

**Done when:** all acceptance criteria above pass and a second correction after a move resolves correctly.
