# SB-117 — Implement clarification handling

**Branch:** `feature/clarification-flow`

For low-confidence classification, file the derived note into `00_inbox/` and send a clarification question through the original receipt or a thread. The raw accepted capture must remain immutable throughout.

---

## State contract

- After Inbox filing, the capture transitions to `NEEDS_CLARIFICATION` processing status.
- `capture-service` stores the derived Inbox note path and the Git commit hash at filing time.
- The clarification question is sent through the capture-service receipt/edit API — not written directly by n8n or writer-service.
- Unanswered clarifications remain visible in `status` output until resolved.
- Timeout does not delete or reclassify the note silently. The note stays in `00_inbox/` and the status entry persists.

## API

- `POST /internal/clarifications/:capture_id` — records the clarification question sent and transitions status.

## Acceptance criteria

- An ambiguous note remains safely available in `00_inbox/` without the classifier force-fitting it into a folder.
- The raw accepted capture is immutable: no field on the original capture record is mutated.
- Derived Inbox note path and Git commit hash are stored on the capture record.
- Clarification question appears on the Discord receipt via the receipt/edit API.
- Unresolved clarifications appear in `sb status` output.
- Timeout does not delete or reclassify; note and status entry remain until a user reply resolves them.

**Done when:** all acceptance criteria above pass and the NEEDS_CLARIFICATION state is visible in status.
